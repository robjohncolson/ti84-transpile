## Context
This is a TI-84 Plus CE ROM transpilation project. The transpiler lifts eZ80 machine code to JavaScript. Session 131 found 22 ROM instructions that write to the FP (floating-point) category byte at RAM address 0xD0060E — ALL 22 are NOT currently in transpiled blocks (uncovered code). The FP dispatch system uses this category byte to route operations like gcd, min, max, etc.

### Step C: Build a probe to test gcd(12,8)
Create `TI-84_Plus_CE/probe-phase132-fp-category-seeds.mjs`. The probe should:

1. Load the transpiled ROM and runtime (same pattern as existing probes — see `probe-phase99d-home-verify.mjs` for the import pattern)
2. Set up a fresh runtime with `timerInterrupt: false`
3. Run MEM_INIT at 0x09DEE0 to initialize allocator pointers
4. Tokenize "gcd(12,8)" into the token buffer at 0xD00800:
   - `0xBB, 0x18` = gcd( (two-byte token)
   - `0x31, 0x32` = "12" (digit tokens)
   - `0x2B` = comma
   - `0x38` = "8" (digit token)
   - `0x11` = close paren
   - `0x3F` = end token (newline/enter)
5. Set parser pointers: begPC=0xD00800, curPC=0xD00800, endPC=0xD00808 (past last token)
   - begPC is at RAM 0xD02317
   - curPC is at RAM 0xD0231A
   - endPC is at RAM 0xD0231D
6. Set OPBase/pTemp/progPtr = 0xD3FFFF, FPSbase/FPS = 0xD1A881 (standard MEM_INIT values)
7. Clear OP1 (9 bytes at 0xD005F8)
8. Push FAKE_RET (0x7FFFFF) sentinel onto stack
9. Set errSP to point at a PushErrorHandler frame
10. Execute ParseInp at 0x099914
11. Read OP1 and decode as BCD float
12. Report: steps taken, OP1 value (expected: 4.0 for gcd(12,8)), any error codes

Also test: Check if the FP category byte at 0xD0060E gets written during execution by reading it before and after.

Reference for BCD decode: OP1 format is [type, exponent, d1d2, d3d4, d5d6, d7d8, d9d10, d11d12, d13d14]. Type 0x00=positive real, exponent 0x80=10^0, 0x81=10^1, etc.
