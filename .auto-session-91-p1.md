# Auto-Session 91 P1: Trace WHEN cxCurApp gets zeroed during CoorMon

## Context
We're transpiling a TI-84 Plus CE ROM into JavaScript. Session 90 found that seeding cxCurApp (address 0xD007E0) with 0x40 before running CoorMon (0x08C331) results in the entire cx context range (0xD007CA-0xD007E1) being zeroed during CoorMon execution. The key finding: CoorMon runs 65,858 steps, GetCSC fires at step 1643, then cxCurApp ends up 0x00. We need to find the EXACT step when cxCurApp transitions from 0x40 to 0x00.

## Task
Create a new probe file `TI-84_Plus_CE/probe-phase25af-cxcurapp-watchpoint.mjs` that:

1. Uses the same cold boot + MEM_INIT + CoorMon infrastructure as the existing `TI-84_Plus_CE/probe-phase25ae-coormon-homeapp.mjs` (read that file for the full pattern)
2. Seeds cxCurApp at 0xD007E0 with 0x40 after MEM_INIT, same as session 90
3. Seeds keyboard matrix with ENTER key (same as session 90's probe)
4. Runs CoorMon at 0x08C331 with a WRITE watchpoint on address 0xD007E0:
   - Before each step, record `mem[0xD007E0]`
   - After each step, check if `mem[0xD007E0]` changed
   - When it changes, log: step number, PC at that step, old value, new value
   - Also log: whether this is BEFORE or AFTER step 1643 (the GetCSC step)
5. Additionally, track writes to the full cx range (0xD007CA-0xD007E1) — log each address that gets written, the step, and the PC
6. Budget: 70,000 steps (CoorMon typically HALTs at ~65,858)
7. Write a report to `TI-84_Plus_CE/phase25af-cxcurapp-watchpoint-report.md`

## Key implementation detail for the watchpoint
The executor's `onStep` callback fires each step. Use it to snapshot mem[0xD007E0] before/after. The `cpu.pc` (or `cpu._pc`) gives the current PC. Look at how `probe-phase25ae-coormon-homeapp.mjs` hooks the executor — it uses `installCxReadTracer` for reads. For writes, you need a similar approach but hooking `write8`/`write16`/`write24` on the memory object, OR simply polling mem[0xD007E0] in an onStep callback.

The simplest approach: use the executor's per-step callback. The `runFrom` method accepts options including `onStep`. Check the existing probes for the pattern. If `onStep` is not available, use the alternate approach: wrap the memory write functions to intercept writes to 0xD007CA-0xD007E1.

Actually, the simplest reliable approach: **snapshot the full cx range (24 bytes) before CoorMon, then after each N steps (use milestone checking every 100 steps), compare**. But even better: just check mem[0xD007E0] at every step — it's just a byte read, minimal overhead for 70K steps.
