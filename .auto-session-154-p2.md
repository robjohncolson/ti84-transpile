# Auto-Session 154 — P2: FPDiv JT Slot Investigation (0x07F7F2 → 0x07FAC2)

## Context

This is a TI-84 Plus CE ROM transpilation project. The ROM is at `TI-84_Plus_CE/ROM.rom` (4 MB eZ80). The transpiled JS is generated by `scripts/transpile-ti84-rom.mjs`. The runtime is in `TI-84_Plus_CE/cpu-runtime.js`.

Session 153 discovered that FPDiv at JT slot 0x020284 (impl at 0x07F7F2) is broken — it returns OP1=[0C 80 00...] for ALL inputs in only 33 steps. The entry at 0x07F7F2 just does `CALL 0x07FAC2`. The 0x0C in OP1[0] looks like a category/type metadata write, not a division result.

The gcd handler does NOT use this JT slot — it uses internal FP routines. But FPDiv is a critical system function that should work.

## Task — TWO PARTS

### Part A: Disassemble 0x07FAC2 and its call chain

Use the eZ80 decoder:
```js

// decodeInstruction(rom, offset, adlMode=true) returns { mnemonic, length, ... }
```

1. Disassemble 0x07FAC2 — at least 50 bytes (or until a RET/JP). This is the subroutine called by FPDiv.
2. For each CALL target found, check if it's a known address:
   - 0x07FA74 = Const 1.0 loader
   - 0x07F831 = Type validator
   - 0x082957 = FpPush
   - 0x0828FC = Real FP Pop
   - 0x07CC36 = FP subtraction core
3. Identify: does 0x07FAC2 set up a category/type byte (writing 0x0C somewhere) before calling the actual division? Is 0x0C a division category code?
4. Check if any CALL targets in the chain are missing blocks (not transpiled). Use the transpiler's block list by checking `ROM.transpiled.js.gz` won't work (too large), so instead check by searching for the address in the seed list or by running a quick test.
