# Auto-Session 113 — P2: Investigate RclVarSym Dereference Bug at Non-Default OPBase

## Task
Build a diagnostic probe that disassembles and traces the RclVarSym routine (entry 0x09AC77) to understand WHY it returns raw VAT entry bytes instead of following the data pointer when OPBase is set to 0xD40040 (Approach B), while working correctly at the default OPBase ~0xD3FFFF (Approach A).

This is a documentation/investigation task. The probe should produce a detailed report, NOT fix anything.

## Background
Session 112 proved that RclVarSym fails under Approach B:
- **Approach A** (OPBase ~0xD3FFF6 after CreateReal): RclVarSym returns OP1=[00 80 50 00 00 00 00 00 00] = 5.0 in 43 steps. PASS.
- **Approach B** (OPBase = 0xD40040): RclVarSym returns OP1=[00 00 72 d1 a8 81 00 00 00] — these are literally the VAT entry metadata bytes (type=0x00, name=0x72, data pointer=0xD1A881 LE). It copies the VAT entry into OP1 instead of dereferencing the data pointer.

The 43-step execution is the same length in both cases, suggesting the code path is identical but the pointer arithmetic produces different addresses.

## How to trace instructions
The executor's `runFrom` method supports `onBlock(pc, meta, step)` and `onMissingBlock(pc, meta, step)` callbacks. For fine-grained per-instruction tracing, use a step limit of 1 in a loop — call `runFrom` with `maxSteps: 1` repeatedly for 43 iterations, reading registers and memory after each step. Alternatively, check if the executor has a `step()` or single-step mode.

Actually, the simplest approach: use `runFrom` with `maxSteps: 50` and capture the `onBlock`/`onMissingBlock` callbacks to log PC at each block entry. Between blocks, dump registers. This won't give per-instruction detail but will show the block-level trace.

For per-instruction detail, read the ROM bytes at each block's entry PC and manually decode the first instruction (just the mnemonic and operands). The eZ80 instruction set is well-documented.
