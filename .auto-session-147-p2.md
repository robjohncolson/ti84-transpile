## Problem

GraphPars renders Y1=X but ALL 320 pixels land on row 239 (bottom row) instead of spanning the diagonal. Root cause (session 146): IPoint reads pixel Y from curGY at RAM 0xD022D1 (2 bytes, little-endian). curGY is never written by the graph rendering loop — it stays 0x0000, and 0 maps to row 225 (near bottom).

The VRAM formula inside IPoint is correct: `base + 320*2*(239-Y) + X*2` (via MLT HL at 0x07B58A).

IPoint reads curGY via CALL 0x07B793. The graph pipeline computes Y in OP1 (floating point, graph coordinates) but nobody converts to pixel Y and writes to curGY.

Also relevant: curGX is likely at 0xD022CF (2 bytes before curGY). IPoint may also read X from curGX — check by looking at what 0x07B793 returns for X.

Key RAM addresses:
- curGY: 0xD022D1 (2 bytes LE, pixel Y coordinate)
- curGX: 0xD022CF (2 bytes LE, pixel X coordinate — hypothesis)
- IY+43 bit 2: 0xD000AB — set for 24-bit bounds check
- IY+74 bit 2: 0xD000CA — clear for plotSScreen writes
- pixWideP: 0xD014FE (screen width, seed 320)
- pixWide_m_2: 0xD01501 (screen height-2, seed 238)
- Pen color: 0xD026AE
- plotSScreen: 0xD09466 (76,800 bytes)
- LCD VRAM 16bpp: 0xD40000

## Task

Build a probe `TI-84_Plus_CE/probe-phase147-graph-curgy.mjs` that:

### Part A: Verify curGX/curGY hypothesis
Call IPoint with manually seeded curGY values (0, 60, 120, 180, 239) and check whether the VRAM write address changes. Also seed curGX with different values and check if X positioning changes.

Use the optimal IPoint flags from session 144/145:
- IY+43 SET bit 2 (24-bit bounds)
- IY+74 CLEAR bit 2 (plotSScreen writes)
- pixWideP=320, pixWide_m_2=238
- drawMode A=1 (normal draw)
- Pen color at 0xD026AE = 0x10

### Part B: Full GraphPars Y1=X with curGY seeding
Run the full GraphPars loop for Y1=X:
1. Cold boot + MEM_INIT
2. CreateEqu for Y1=X
3. Loop X from Xmin to Xmax (default -10 to 10, 320 steps)
4. For each X: run GraphPars body at 0x099874 with tX intercept at 0x07D1B4 to get Y from OP1
5. Convert graph Y to pixel Y: `pixelY = Math.round((Ymax - Y) / (Ymax - Ymin) * 238)` where Ymin=-10, Ymax=10
6. Clamp pixelY to [0, 238]
7. Write pixelY to curGY at 0xD022D1 (16-bit LE)
8. Also compute pixelX and write to curGX at 0xD022CF (16-bit LE)
9. Call IPoint

After all 320 iterations, scan plotSScreen and LCD VRAM for non-zero bytes. Report how many unique rows have pixels.

### Part C: Also try writing curGX
The X coordinate might also come from curGX rather than from register state. Test: for a single point (X=0, Y=0 → center of screen), seed curGX=160, curGY=119, call IPoint. Check if the pixel appears at the center.
