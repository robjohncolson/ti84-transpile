# Auto-Session 154 — P1: GCD Handler Disassembly + Manual OP2=8.0 Restore Test

## Context

This is a TI-84 Plus CE ROM transpilation project. The ROM is at `TI-84_Plus_CE/ROM.rom` (4 MB eZ80). The transpiled JS is generated by `scripts/transpile-ti84-rom.mjs`. The runtime is in `TI-84_Plus_CE/cpu-runtime.js`.

We are investigating why `gcd(12,8)` returns the wrong result. The root cause was confirmed in session 153:

- The gcd handler at 0x068D3D pushes OP2 (=8.0) to the FP stack (FPS) at step 15 via FpPush (0x082957→0x082961 shared tail)
- At step 18, the const 1.0 loader at 0x07FA74 intentionally overwrites OP2 with 1.0
- The algorithm then computes INT(12/1)=12 instead of INT(12/8)=1, producing mod=0 instead of 4
- The original 8.0 pushed to FPS at step 15 is NEVER popped back into OP2 before division
- FPS tracking across 1442 steps shows only 2 pops (at steps 1424 and 1442), both too late

2. **0x068D61 → 0x068B78** (transition to const loader). This may be a JP or CALL. Trace the exact instruction sequence.

Key addresses:

- 0x07FA74 = Const 1.0 loader (writes [00 80 10 00 00 00 00 00 00] to OP2)
- 0x07F831 = Type validator
