# Auto-Session 165 P1: Test gcd with divisible pairs

## Context

You are working on a TI-84 Plus CE ROM transpilation project at `C:/Users/rober/Downloads/Projects/school/ti84-transpile`. The project lifts eZ80 machine code into JavaScript.

We've been investigating a bug in the gcd (greatest common divisor) function. For gcd(12,8), the OS produces E_Domain error instead of the correct answer (4). The algorithm runs a Euclidean-style loop using FPAdd/FPSub (no FPDiv). The root cause: in the inner body, a compound function at 0x07C747 copies OP1→OP2, normalizes OP1 to zero, then FPAdd restores OP1 from OP2. This makes OP1==OP2 at InvSub, producing mod=0 instead of mod=4.

**Setup pattern** (from existing probes):
```javascript
// Initialize executor
const { executor, mem } = createExecutor(romBytes, BLOCKS, MEM_SIZE, createPeripheralBus({ timerInterrupt: false }));

// Write OP1
const op1Bytes = [0x00, 0x81, 0x12, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]; // 12
for (let i = 0; i < 9; i++) mem[OP1_ADDR + i] = op1Bytes[i];

// Write OP2
const op2Bytes = [0x00, 0x80, 0x80, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]; // 8
for (let i = 0; i < 9; i++) mem[OP2_ADDR + i] = op2Bytes[i];

// Set category
mem[0xd0060e] = 0x28;

// Set errNo to 0
mem[ERR_NO_ADDR] = 0;

// Set error SP (needed for E_Domain handler)
const errSP = STACK_RESET_TOP - 6;
mem[ERR_SP_ADDR] = errSP & 0xFF;
mem[ERR_SP_ADDR + 1] = (errSP >> 8) & 0xFF;
mem[ERR_SP_ADDR + 2] = (errSP >> 16) & 0xFF;

// Initialize FPS
mem[FPS_ADDR] = 0x00;
mem[FPS_ADDR + 1] = 0xAA;
mem[FPS_ADDR + 2] = 0xD1;

// Initialize SP and return address
executor.sp = STACK_RESET_TOP;
const RETURN_SENTINEL = 0x7FFFFE;
mem[executor.sp - 1] = (RETURN_SENTINEL >> 16) & 0xFF;
mem[executor.sp - 2] = (RETURN_SENTINEL >> 8) & 0xFF;
mem[executor.sp - 3] = RETURN_SENTINEL & 0xFF;
executor.sp -= 3;

// Set PC
executor.pc = GCD_ENTRY;
```

Then run with `executor.step()` in a loop until `executor.pc === RETURN_SENTINEL` or step count > 3000.
