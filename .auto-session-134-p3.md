# Auto-Session 134 P3: Trace GraphPars Crash at 0x58C35B + Test Constant Equation

## Context
This is a TI-84 Plus CE ROM transpilation project. Session 133 successfully created a Y1=X equation via CreateEqu (EquObj type=0x03, OP1=[0x03, 0x10, 0x00...]). GraphPars (0x09986C) finds the equation but crashes at step 96, jumping to PC=0x58C35B — a bogus address.

The hypothesis is that 0x58 = tX (the token for the X variable), and GraphPars is misinterpreting equation token data as an executable address. This could happen if:
- A JP (HL) or CALL (HL) loads HL from equation data instead of code
- A missing block causes a fallthrough that reads data bytes as code

## Task
1. Build a probe at `TI-84_Plus_CE/probe-phase134-graphpars-trace.mjs` that:
   - Uses the same boot/MEM_INIT pattern as `probe-phase133-graph-yeq-create.mjs`
   - Creates a Y1 equation using CreateEqu (same as session 133)
   - Seeds graph window (Xmin=-10, Xmax=10, Ymin=-10, Ymax=10 in BCD at their standard RAM slots)
   - Seeds graph mode: 0xD01474 = 0x10 (function mode)
   - Seeds pixWideP (0xD014FE=320), pixWide_m_2 (0xD01501=240), pen color (0xD026AE=0xFF00)
   - Calls GraphPars at 0x09986C
   - Logs EVERY PC for the last 20 steps before reaching 0x58C35B (or any address > 0x3FFFFF which is outside ROM)
   - At each of those last 20 steps, also log the CPU registers (HL, DE, BC, A, SP, IX)
   - Specifically check: is there a `JP (HL)` or `CALL (HL)` where HL=0x58C35B? Or is it a RET with a bad return address on the stack?

2. Run a SECOND test with a CONSTANT equation Y1=1 instead of Y1=X:
   - Token for "1" is 0x31 (tl = digit 1)
   - Equation data: [0x01, 0x00, 0x31] (1-byte length, token 0x31)
   - This avoids the tX (0x58) token entirely
   - If GraphPars still crashes, the problem is NOT the tX byte — it's something else
   - If it works, the problem IS the variable lookup for X

3. Report:
   - The exact sequence of PCs leading to the crash
   - Which instruction caused the jump to 0x58C35B (or similar bogus address)
   - Whether Y1=1 (constant) also crashes or succeeds
   - What the CPU state was at the crash point (all registers)
