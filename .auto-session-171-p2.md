# Session 171 P2: Static Fidelity Check — Transpiled Blocks vs ROM Bytes

## Task
Build `TI-84_Plus_CE/probe-phase171-block-fidelity.mjs` that verifies every transpiled block visited during gcd(12,8) matches the ROM bytes — checking jump targets, immediate values, and call destinations.

## Context
We've exhausted all behavioral hypotheses for why gcd(12,8) produces E_Domain instead of 4. Every individual component (normalization, FPDiv, FPAdd, gcd_helper, etc.) has been verified correct in isolation. The bug must be a subtle transpilation error in one of the ~230 blocks — a wrong immediate value, wrong jump target, or wrong register assignment that only manifests in the full gcd execution context.

## Deliverables
Create `TI-84_Plus_CE/probe-phase171-block-fidelity.mjs` that:

1. Runs gcd(12,8) to collect all unique block PCs visited (use onBlock callback, ~230 unique PCs, ~1442 steps)
2. For each unique block PC, uses `ez80-decoder.js` to decode instructions from ROM bytes
3. For each decoded instruction, checks:
   - **CALL targets**: decode the 3-byte address from ROM, verify the transpiled block references the correct target block key (format: `XXXXXX:adl`)
   - **JP targets**: same check for unconditional and conditional jumps
   - **JR targets**: verify the relative offset produces the correct absolute target
   - **Immediate values**: for LD instructions with immediate operands, verify the value matches ROM bytes
   - **Block existence**: verify the target block exists in BLOCKS for every CALL/JP/JR target
4. Reports any mismatches with: block PC, instruction offset, expected vs actual values

## GCD Setup Pattern
```js
const peripherals = createPeripheralBus({ timerInterrupt: false });
const { cpu, mem, BLOCKS: exBlocks } = createExecutor(romBytes, BLOCKS, peripherals);

const STACK_TOP = 0xd1a87e;
const SENTINEL = 0x7ffffe;
cpu.sp = STACK_TOP;
mem[SENTINEL] = 0x76; mem[SENTINEL+1] = 0x76; mem[SENTINEL+2] = 0x76;
cpu.sp -= 3;
mem[cpu.sp] = SENTINEL & 0xff;
mem[cpu.sp+1] = (SENTINEL >> 8) & 0xff;
mem[cpu.sp+2] = (SENTINEL >> 16) & 0xff;

const FPS_ADDR = 0xd0258d;
const FPSBASE_ADDR = 0xd0258a;
const FPSBASE = 0xd1a9f7;
mem[FPS_ADDR] = FPSBASE & 0xff;
mem[FPS_ADDR+1] = (FPSBASE >> 8) & 0xff;
mem[FPS_ADDR+2] = (FPSBASE >> 16) & 0xff;
mem[FPSBASE_ADDR] = FPSBASE & 0xff;
mem[FPSBASE_ADDR+1] = (FPSBASE >> 8) & 0xff;
mem[FPSBASE_ADDR+2] = (FPSBASE >> 16) & 0xff;

const OP1_ADDR = 0xd005f8;
const op1 = [0x00, 0x81, 0x12, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00];
for (let i = 0; i < 9; i++) mem[OP1_ADDR + i] = op1[i];

const OP2_ADDR = 0xd00603;
const op2 = [0x00, 0x80, 0x80, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00];
for (let i = 0; i < 9; i++) mem[OP2_ADDR + i] = op2[i];

mem[0xd008df] = 0x00;
cpu.ix = 0xd1a860;
cpu.pc = 0x068d3d;
```

## How to Check Transpiled Block Content
The transpiled blocks are JS functions. You can convert them to strings to inspect:
```js
const blockKey = pc.toString(16).padStart(6, '0') + ':adl';
const blockFn = BLOCKS[blockKey];
if (!blockFn) { console.log(`MISSING BLOCK: ${blockKey}`); }
const blockSrc = blockFn.toString();
// Search blockSrc for hex addresses, immediate values, etc.
```

For each decoded instruction from ROM:
- If it's `CALL 0xABCDEF`, check that `blockSrc` contains reference to `0xabcdef` or the block key `0abcdef:adl`
- If it's `LD A,0x42`, check that `blockSrc` contains `0x42` or `66` (decimal)
- If it's `JP 0xABCDEF`, check the target address appears in the block source
