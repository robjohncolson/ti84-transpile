# Session 166 P3: Trace BIT 0,B test + PUSH AF/POP AF flag usage in gcd inner body

## Context
You are working on a TI-84 Plus CE ROM transpilation project at `C:/Users/rober/Downloads/Projects/school/ti84-transpile`. The project lifts eZ80 ROM machine code into JavaScript.

We are investigating why `gcd(12,8)` fails with E_Domain. Session 165 P2 showed the gcd inner body at 0x068D82 starts with:
- `CB 40` = BIT 0,B (tests bit 0 of register B, sets Z flag based on result)
- `F5` = PUSH AF (saves A and flags to stack)

Then 5 CALLs follow: OP1→OP4, compound(0x07C747), OP1→OP3, OP4→OP2, InvSub(0x07C74F).

**The question**: What is register B at the BIT 0,B test? And when does POP AF happen? The saved flags might control a branch later in the algorithm that determines what to do with the InvSub result. If B=0 at the test, Z=1, which might cause a different execution path than the real calculator expects.

## Your Task
Build a probe `TI-84_Plus_CE/probe-phase166-bit0b-flags.mjs` that runs the FULL gcd(12,8) handler and:

1. Sets up gcd context: OP1=[00 81 12 00 00 00 00 00 00] (=12.0), OP2=[00 80 80 00 00 00 00 00 00] (=8.0)
2. Calls the gcd handler entry at 0x068D3D (or 0x068D41 — check which one the gcd JT slot points to)
3. Tracks register B and the flags register F at EVERY block
4. Specifically watches for:
   - **0x068D82**: The BIT 0,B instruction — what is B? What is Z after?
   - **Every PUSH AF** (opcode F5): Log when flags are pushed and what values
   - **Every POP AF** (opcode F1): Log when flags are popped and what values
   - **Any conditional branch using flags from the POP**: JR Z, JR NZ, JP Z, JP NZ, CALL Z, CALL NZ, RET Z, RET NZ after a POP AF
5. Also track: the BIT test at the loop-top 0x068D75 (CP 0x82 at line 0x068D77, which sets B=0 or B=1)
