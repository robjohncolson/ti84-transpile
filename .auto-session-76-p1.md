# Task: Disassemble 0x061DB2 — TI-OS error handler convergence point

## Context
Every entry in the TI-OS error dispatch table at 0x061D1A-0x061D7E does `ld a, <error_code>; jr 0x061db2`. All error paths converge here. We need to understand what 0x061DB2 does — does it call DispErrorScreen (0x062160)? Does it longjmp via stack manipulation? Does it set flags and RET?

The address 0x061DBA is known to eventually reach 0x58c35b (a "missing_block" termination in our transpiler). Session 75 traced: `0x061db2 → 0x03e1b4 → 0x03e1ca → 0x061dba → 0x58c35b`.

## Deliverables
Create TWO files:
1. `TI-84_Plus_CE/probe-phase25q-error-handler.mjs` — static disassembly probe
2. `TI-84_Plus_CE/phase25q-error-handler-report.md` — generated report

## What the probe must do

Use the existing static disassembly pattern from `TI-84_Plus_CE/probe-phase25p-vat-disasm.mjs` as template. Import `decodeInstruction` from `./ez80-decoder.js`, read `ROM.rom`, decode instruction ranges, and write a report.

Disassemble these ranges:
- `0x061DB2` — 128 bytes (the main convergence point)
- `0x061D80` — 50 bytes (gap between end of dispatch table and 0x061DB2)
- `0x03E1B4` — 64 bytes (common crash subroutine, already partially known)
- `0x03E1CA` — 64 bytes (second crash subroutine)
- `0x062160` — 64 bytes (DispErrorScreen entry — check if 0x061DB2 calls this)

For each range:
1. Decode every instruction using `decodeInstruction(romBytes, pc, 'adl')`
2. Print hex bytes + mnemonic
3. Track all CALL/JP/JR targets
4. Track all RAM reads/writes (addresses 0xD00000-0xD1FFFF) — both absolute and IY-relative (IY=0xD00080)
5. Identify reads-before-writes (RAM addresses that must be pre-seeded)

The report must answer:
- Does 0x061DB2 CALL or JP to 0x062160 (DispErrorScreen)?
- Does 0x061DB2 manipulate SP (longjmp pattern)?
- Does 0x061DB2 read any RAM that would need seeding?
- What is the full control flow from 0x061DB2 to 0x58c35b?
