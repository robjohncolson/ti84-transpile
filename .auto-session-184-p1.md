# Auto-Session 184 P1: Investigate MBASE and SPS Stack Page Contents

## Context

This is a TI-84 Plus CE ROM transpilation project. The eZ80 CPU has been transpiled to JavaScript. We recently fixed z80-mode stack addressing to use `{MBASE, SPS[15:0]}` instead of raw SPL. With MBASE=0xD0 (our default) and SP=0xD1A87E, z80-mode operations access 0xD0A87E.

**Problem**: When ParseInp fires E_Undefined (errNo=0x8D), the error handler at 0x03E1B4 calls 0x03E187 which uses RSMIX to enter z80-mode, does POP AF + RET. The z80-mode RET at 0x03E1B3 pops from {MBASE=0xD0, SPS} = 0xD0xxxx, which contains zeros (boot vector) → ISR → HALT. The error handler never returns to JError to execute `LD SP,(errSP)`.

**Question**: What MBASE value does the real OS actually use? If MBASE=0xD1 (matching SPL page 0xD1xxxx), then z80-mode stack operations would access the SAME memory as ADL-mode operations, and the RET would pop a valid return address.

## Task

Build a Node.js probe at `TI-84_Plus_CE/probe-phase184-mbase-investigation.mjs` that:

1. **Search ROM for all `LD MB,A` instructions** (opcode: `ED 6D`). The eZ80 instruction `LD MB,A` sets MBASE from register A. Find ALL occurrences in the ROM and report what value A would have at each site (trace backwards if possible, or just report the address).

2. **Check what the OS boot sequence sets MBASE to**: We know `LD MB,A` exists at 0x0013C9 (from session 177 notes). Trace backwards from 0x0013C9 to find what value A has when this executes. Look at the preceding instructions (LD A, imm or whatever loads A).

3. **Dump memory at the 0xD0A870-0xD0A890 region** after a standard home-screen boot (use the probe-phase99d setup: run with IX=0xD1A860, boot through stages 1-4 with timer IRQ disabled). This shows what the OS leaves at the SPS stack page.

4. **Test MBASE=0xD1**: Run the same ParseInp error dispatch trace from session 183 but with `cpu.mbase = 0xD1` instead of 0xD0. Does the z80-mode RET then pop a valid return address from 0xD1xxxx (same page as SPL)?
