# Auto-Session 107 — P2: Fix Error Frame Placement for Trampoline Entry

## Goal
Build a probe that places the error frame BELOW the deepest SP reached during the trampoline CALL chain, then re-runs the trampoline to test if errors are now caught correctly by ERR_CATCH_ADDR instead of crashing into the common tail.

## Background
Session 106 discovered that entering ParseInp via 0x0586E3 (CALL 0x099910) clobbers the error frame:
1. CALL 0x099910 at 0x0586E3 writes return address 0x0586E6 at 0xD1A872-0xD1A874 (overwrites ERR_CATCH_ADDR bytes)
2. CALL 0x07FF81 at 0x099910 writes return address 0x099914 at 0xD1A86F-0xD1A871 (overwrites first 3 frame bytes)

The deepest SP reached during the CALL chain is 0xD1A86C (after 0x04C940 internal call inside 0x07FF81). So the error frame must be placed at or below 0xD1A866 (= 0xD1A86C - 6).

Currently, the probe uses errFrameBase = SP - 6 = 0xD1A86F, which is exactly in the clobber zone.

### Scenarios

#### Scenario A: Baseline control (direct ParseInp, standard error frame)
1. createEnv() → coldBoot() → MEM_INIT
2. prepareCallState(cpu, mem) + seedScenario(mem, cpu)
3. Standard error frame at SP-6 (as in existing probes)
4. Run ParseInp at 0x099914, budget=2000
5. Report: steps, OP1, errNo, whether ERR_CATCH_ADDR was hit

#### Scenario B: Trampoline with ORIGINAL (clobbered) error frame
1. Fresh createEnv() → coldBoot() → MEM_INIT
2. prepareCallState + seedScenario
3. cpu.sp = 0xD1A875 (STACK_RESET_TOP - 9)
4. Error frame at SP-6 = 0xD1A86F (the CLOBBERED position)
5. write24(mem, errFrameBase+3, ERR_CATCH_ADDR), write24(mem, ERR_SP_ADDR, errFrameBase)
6. Verify error frame before run: dump 6 bytes at errFrameBase
7. Run from 0x0586E3, budget=50000
8. Track: did execution hit ERR_CATCH_ADDR? How many steps? What's at errFrameBase after run? OP1? errNo?
9. Expected: ERR_CATCH_ADDR NOT hit, error frame clobbered, allocator loop

#### Scenario C: Trampoline with FIXED (low) error frame
1. Fresh createEnv() → coldBoot() → MEM_INIT
2. prepareCallState + seedScenario
3. cpu.sp = 0xD1A875
4. Return address at cpu.sp: FAKE_RET
5. **Error frame at 0xD1A860** (well below 0xD1A86C, safe from all CALLs in the chain)
   - errFrameBase = 0xD1A860
   - write24(mem, errFrameBase, 0x000000) — prev errSP (or 0)
   - write24(mem, errFrameBase + 3, ERR_CATCH_ADDR) — handler address
   - write24(mem, ERR_SP_ADDR, errFrameBase) — tell OS where the frame is
6. Verify error frame before run: dump 6 bytes at errFrameBase
7. Run from 0x0586E3, budget=50000
8. Track in onBlock callback:
   - Did 0x099914 (ParseInp entry) get hit? At what step?
   - Did ERR_CATCH_ADDR (0x7FFFFA) get hit? At what step?
   - Did 0x061D3A (error throw) get hit?
   - Did 0x061DB2 (JError) get hit?
   - Count steps in allocator region (0x082200-0x082900)
9. After run: dump error frame bytes, OP1, errNo, SP
10. Expected: even if ParseInp still takes the wrong branch (skips PushErrorHandler), the error should be caught by ERR_CATCH_ADDR instead of crashing into common tail code at 0x0586E7

#### Scenario D: Even lower error frame at 0xD1A850
Same as Scenario C but with errFrameBase = 0xD1A850, for extra safety margin. This tests if there are deeper CALL chains we haven't discovered.

### Critical notes
- Use `onBlock(pc)` to track PC hits. The callback fires BEFORE the block executes.
- Use `onMissingBlock(pc)` to catch ERR_CATCH_ADDR and FAKE_RET (these are sentinel addresses with no transpiled blocks — the probe catches them via the callback).
- For tracking, use an object like `{ parseInpHit: false, errCatchHit: false, errorThrown: false, jErrorHit: false, allocatorSteps: 0 }` and update in onBlock.
- `throw new Error('__ERR_CATCH__')` when ERR_CATCH_ADDR is hit in onMissingBlock, and `throw new Error('__FAKE_RET__')` when FAKE_RET is hit.
- Even with the fixed error frame, ParseInp may still take the wrong branch (that's priority 1's job to fix). The point of THIS probe is to verify that errors are caught by ERR_CATCH_ADDR rather than crashing into random code.
- Do NOT modify any existing files. Only create the new probe file.
- The executor cpu property is at `executor.cpu`. Registers: `cpu.a`, `cpu.f`, `cpu.sp`, `cpu._bc`, `cpu._de`, `cpu._hl`, `cpu._ix`, `cpu._iy`.
