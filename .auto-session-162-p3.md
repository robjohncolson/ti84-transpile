# Auto-Session 162 P3 — Trace FPDiv entry state: gcd context vs direct call

## Context
You are working on a TI-84 Plus CE ROM transpilation project. The transpiler lifts eZ80 ROM to JavaScript. We are debugging the gcd(12,8) function which should return 4 but instead produces -100 and E_Domain error.

**CRITICAL PUZZLE**: Session 155 found that FPDiv(1200,1200) = 0 when called during gcd, but = 1.0 when called directly at 0x07CAB9. Session 161 proved normalization is NOT the root cause (bypassing it gives identical results). **Where does the FPDiv computation actually happen in the gcd flow?**

Session 161's dense trace showed that after the OP1toOP2 compound function (0x07C747) returns, OP1 is restored to 1200. The algorithm then copies OP1→OP3, OP1→OP5, and enters 0x068D95. But session 155 said FPDiv(1200/1200)=0 in gcd context — where does this division actually take place?

**KEY ADDRESSES**:
- 0x07CAB9 = FPDiv implementation
- 0x0201F4 = FPDiv JT slot (JP 0x07CAB9, seeded in session 157)
- 0x07CA48 = normalization (enters via OP1toOP2 at 0x07C747, but NOT the root cause)
- 0x068D95 = mystery block after OP1→OP5 backup in gcd body
- 0x080188 = JmpThru (hit 4 times during gcd per session 155)

## Task
Build a probe that traces the FULL gcd(12,8) execution with specific focus on:

1. **Every time PC enters 0x07CAB9 (FPDiv impl)**: log step, OP1, OP2, A, SP, and all registers
2. **Every time PC enters 0x0201F4 (FPDiv JT slot)**: same logging
3. **Every time PC enters 0x080188 (JmpThru)**: log what it jumps to
4. **Every time PC enters 0x068D95-0x068DEA range**: log step, PC, OP1, OP2

Also run a **COMPARISON**: direct FPDiv(1200,1200) at 0x07CAB9, logging the initial state (all registers, OP1, OP2, FPS, key memory). Then compare that state snapshot with what's present when FPDiv is entered during gcd.
