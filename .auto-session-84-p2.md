# Task: Disassemble ParseInp entry (0x099914..0x099940) to verify curPC usage

## Context
We are reverse-engineering the TI-84 Plus CE OS ROM. ParseInp at 0x099914 is the expression parser. Session 83 demonstrated that pointing begPC/curPC/endPC (0xD02317/0xD0231A/0xD0231D) to a token buffer allows ParseInp to read tokens. We need to confirm: does ParseInp always read tokens through curPC, or does it use another pointer under certain conditions?

## What to do

### Step 1: Create a disassembly probe
Create `TI-84_Plus_CE/probe-phase25y-parseinp-entry-disasm.mjs` that:

1. Loads the ROM file directly (just `fs.readFileSync` on `ROM.rom` — no need for the full runtime for static disassembly).

2. Reads and decodes the first ~40 bytes of ParseInp starting at 0x099914. For each instruction, print:
   - Address
   - Raw hex bytes
   - Decoded instruction mnemonic
   
   Use the eZ80 ADL-mode instruction set. Key opcodes to handle:
   - Standard Z80 opcodes (LD, CALL, JP, JR, RET, PUSH, POP, etc.)
   - ADL 3-byte addresses (24-bit operands)
   - ED-prefixed extended instructions
   - CB-prefixed bit operations
   - DD/FD prefix for IX/IY
   
   For the first 40 bytes, a simple linear sweep decoder is sufficient. You can use the decoder in `TI-84_Plus_CE/ez80-decoder.js` if it has a suitable API, or write a minimal inline decoder for the common instructions.

3. After decoding, specifically check:
   - Does any instruction reference 0xD0231A (curPC)?
   - Does any instruction reference 0xD02317 (begPC)?
   - Does any instruction reference 0xD02593 (OPS)?
   - Are there any indirect loads through HL/DE/BC that could be reading from a pointer slot?

4. Also decode the first 20 bytes at 0x099920 and 0x099930 (overlap is fine — we want to see the full entry sequence).

### Step 2: Cross-reference with known patterns
Check if the first few instructions match the pattern from session 73's observation:
> "First 32 ROM bytes show `af 32 be 22 d0 ...` (= `xor a; ld (0xD022BE),a; ...`)"

This would confirm ParseInp starts with `XOR A; LD (0xD022BE),A` — clearing a flag before parsing.

### Step 3: Run the probe
Execute: `node TI-84_Plus_CE/probe-phase25y-parseinp-entry-disasm.mjs`

### Step 4: Run golden regression
Execute: `node TI-84_Plus_CE/probe-phase99d-home-verify.mjs`
Verify 26/26 PASS.

### Step 5: Write report
Create `TI-84_Plus_CE/phase25y-parseinp-entry-disasm-report.md` with:
- Full disassembly listing
- Which pointer(s) ParseInp reads tokens from
- Whether curPC is accessed in the first 40 bytes or deeper in the routine
- Any flags/state cleared at entry
