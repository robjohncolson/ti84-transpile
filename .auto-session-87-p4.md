# Task: Fix full error-frame probe's hlPayload to 0x099929

## Context

TI-84 Plus CE ROM transpilation project. The full PushErrorHandler error-frame probe (from session 84/85) now runs with PopErrorHandler transpiled, but gets stuck because `hlPayload=0x000000`. PopErrorHandler's RET uses hlPayload as a return address → execution jumps to 0x000000 (boot entry) → reinitializes everything → counting loop at 0x006202.

Session 86's store trace showed ParseInp catches errors internally at step 36, PC=0x099929. This is likely the real continuation address (the "setjmp" handler address that PushErrorHandler saves).

## Your task

Create a probe `TI-84_Plus_CE/probe-phase25ab-errframe-hlpayload.mjs` that:

1. Cold boots the ROM
2. Calls MEM_INIT at 0x09DEE0 (~17 steps)
3. Sets up a proper 18-byte PushErrorHandler error frame on the stack. The frame format (from session 79's disassembly of PushErrorHandler at 0x061DEF) is 6 × 3-byte entries pushed in order:
   - hlPayload (caller's handler address) = **0x099929** (ParseInp's internal error catch point)
   - previous errSP = some valid value (e.g., 0x000000)
   - FPS - FPSbase delta = 0 (since both will be at 0xD1A881 after MEM_INIT)
   - OPS - OPBase delta = will need to compute from post-MEM_INIT values
   - PopErrorHandler address = 0x061DD1
   - Normal-return cleanup stub = 0x061E27
   
   Then set errSP to point at this frame on the stack.

4. Seeds token buffer at 0xD00800 with [0x32, 0x70, 0x33, 0x3F] ("2+3")
5. Sets begPC/curPC to 0xD00800, endPC to 0xD00804
6. Calls ParseInp at 0x099914. Budget: 1,500,000 steps.
7. Monitors:
   - Does execution reach 0x099929 (the error catch point)?
   - What PC does ParseInp terminate at?
   - errNo value
   - OP1 contents
   - Step count
   - Whether PopErrorHandler (0x061DD1) is visited

Write a markdown report to `TI-84_Plus_CE/phase25ab-errframe-hlpayload-report.md`.

## Error frame layout on stack (18 bytes, 6 × 3-byte entries)

The frame is pushed by PushErrorHandler. When we build it manually, we write it to memory and point errSP at it. The stack grows downward. The entries at ascending addresses are:

```
[SP+0]  = normal-return cleanup stub (0x061E27)
[SP+3]  = PopErrorHandler (0x061DD1)  
[SP+6]  = OPS - OPBase delta
[SP+9]  = FPS - FPSbase delta
[SP+12] = previous errSP
[SP+15] = hlPayload (error handler address = 0x099929)
```

errSP should point to SP+15 (or wherever longjmp's `pop af; ret` will land to reach 0x099929).

Actually, the longjmp at 0x061DB2 does: `ld sp, (errSP); pop af; ret`. So errSP should point to 2 bytes before the hlPayload entry. When `pop af` pops 3 bytes (in ADL mode), then `ret` pops the next 3 bytes as the return address = hlPayload. So errSP = address of a 3-byte AF value, followed by hlPayload at errSP+3.

Check `probe-phase25t-parseinp-errhandler.mjs` for the exact layout used previously.
