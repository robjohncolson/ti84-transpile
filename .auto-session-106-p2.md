# Task: Investigate WHY 3-byte SP shift triggers compaction in ParseInp

## Context

You are working on a TI-84 Plus CE ROM transpilation project at `C:/Users/rober/Downloads/Projects/school/ti84-transpile`. The transpiler lifts eZ80 machine code into runnable JavaScript.

**The mystery**: Direct ParseInp at 0x099914 with SP=0xD1A872 succeeds (OP1=5.0, errNo=0x8D, 918 steps). But with SP=0xD1A86F (just 3 bytes lower), it triggers the allocator loop and fails. A 3-byte difference shouldn't matter for expression parsing. We need to understand WHY.

Hypothesis: ParseInp or its callees compare SP against a RAM pointer (e.g., FPS, OPBase) for stack/heap collision detection. When SP=0xD1A86F, the comparison triggers a "memory full" condition → compaction → pointer zeroing.

   - pTemp at 0xD0259A
   
   The hypothesis is that somewhere in ParseInp's call chain, there's a comparison like `if (SP < FPS) → trigger compaction`. Finding which pointer SP is compared against would explain the threshold.
