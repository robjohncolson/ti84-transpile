# Task: Phase 25AG — Disassemble CoorMon dispatch path (0x08C331 → ... → 0x001881)

## Goal
Statically disassemble the CoorMon dispatch path to understand: (a) where CoorMon checks cxMain/cxCurApp to decide which app handler to dispatch to, (b) what branch leads to the 0x001881 RAM CLEAR, and (c) what branch leads to ParseInp (the desired home-screen path).

## Background
Session 91 traced CoorMon execution with cxCurApp=0x40 manually seeded:
- GetCSC (0x03FA09) hit at step 1642 — key consumed
- ~2369 steps of dispatch logic after GetCSC
- RAM CLEAR at PC=0x001881 at step 4011

The 0x001881 RAM CLEAR zeroes ALL OS RAM (0xD00000-0xD13FD7): `LD HL,0xD00000; LD DE,0xD00001; LD BC,0x013FD7; LD (HL),0; LDIR`.

CoorMon (0x08C331) is the OS main event loop. It calls GetCSC to get a keypress, then dispatches it to the current app's handler. With cxCurApp=0x00, there's no valid handler → key is consumed silently. With cxCurApp=0x40 but no valid cxMain pointer, it falls through to RAM CLEAR.

The cxMain field at 0xD007CA is a 3-byte pointer to the app's main handler. For home screen, 0x08C8CB sets cxMain=0x058241. CoorMon presumably reads cxMain and CALLs/JPs to it for key dispatch.

## What to build
Create `TI-84_Plus_CE/probe-phase25ag-coormon-dispatch-disasm.mjs` that:

1. **Static ROM disassembly** of the CoorMon entry region: 0x08C331 to ~0x08C450 (about 300 bytes). Use the eZ80 decoder at `ez80-decoder.js` to decode each instruction. Show the full instruction listing with addresses and mnemonics.

2. **Trace the dispatch flow** by identifying:
   - Where GetCSC (0x03FA09) is called
   - Where cxMain (0xD007CA) or cxCurApp (0xD007E0) are read (look for `LD HL,(0xD007CA)` or `LD A,(0xD007E0)` patterns)
   - Where a JP (HL) or CALL (HL) dispatches to the app handler
   - Conditional branches that could lead to RAM CLEAR vs. ParseInp

3. **Also disassemble** the cxMain target for home screen: 0x058241 to ~0x058300 (about 192 bytes). This is where CoorMon dispatches TO when cxMain=0x058241. It should eventually route to ParseInp for keypress processing.

4. **Also check** what's at 0x001881 ± 50 bytes — is it called/jumped to from CoorMon directly or is it reached by falling through bad/zero pointers?

5. **Cross-reference**: in the ROM bytes, search for bytes that reference 0x001881 (look for `81 18 00` in LE order near CALL/JP instructions in the 0x08C300-0x08C500 range).
