# Auto-Session 99 P2: Investigate workspace pointer overlap causing cx-zeroing

## Context
This is a TI-84 Plus CE ROM transpilation project. The ROM is at `TI-84_Plus_CE/ROM.rom` (4 MB eZ80). We have a runtime at `TI-84_Plus_CE/cpu-runtime.js` and peripherals at `TI-84_Plus_CE/peripherals.js`.

Session 97 traced the compaction call chain that zeros the cx block:
```
0x058423 → 0x05E7F7 → 0x07FF7B → 0x08384F → 0x05E3A2 → 0x05E836 → 0x0831A4 (LDDR zeros cx)
```

The cx block lives at 0xD007CA-0xD007E1 (24 bytes). It gets zeroed because workspace pointers at 0xD0066F/0xD00672/0xD00678 (iMathPtr1-3) define a region that overlaps the cx block, and the LDDR block move sweeps through it.

Key question: Is this compaction part of the ENTER key's POST-evaluation cleanup? The call chain starts at 0x058423 (called from the home-screen handler at 0x058618). If so, the cx-zeroing happens AFTER ParseInp and isn't a blocker.

## Task
Create a probe that:

### Part A: Dump workspace pointers BEFORE compaction
1. Set up the standard runtime: cold boot, kernel init, post-init, MEM_INIT
2. Seed full cx context (cxMain=0x058241, cxCurApp=0x40, all handler pointers per session 96/97)
3. Seed keyboard (ENTER key: keyMatrix[1]=0xFE, kbdKey=0x05, kbdGetKy=0x05)
4. Seed error frame, tokenized "2+3" at userMem
5. Run CoorMon (0x08C331) with step budget 25,000 (enough to reach compaction at ~step 18000 but stop before)
6. At each step, check if PC is in range 0x05E3A0-0x05E840 (compaction entry area)
7. When compaction entry is detected, DUMP these workspace pointers:
   - iMathPtr1 at 0xD0066F (3 bytes)
   - iMathPtr2 at 0xD00672 (3 bytes)
   - iMathPtr3 at 0xD00675 (3 bytes)
   - iMathPtr4 at 0xD00678 (3 bytes)
   - iMathPtr5 at 0xD0067B (3 bytes)
   - asm_data_ptr1 at 0xD0067E (3 bytes)
   - asm_data_ptr2 at 0xD00681 (3 bytes)
   - editSym at 0xD0244E (3 bytes)
   - begPC at 0xD02317 (3 bytes)
   - curPC at 0xD0231A (3 bytes)
   - endPC at 0xD0231D (3 bytes)
   - OPBase at 0xD02590 (3 bytes)
   - pTemp at 0xD0259A (3 bytes)
   - progPtr at 0xD0259D (3 bytes)
8. Also dump the cx block (0xD007CA-0xD007E1) at the same moment to confirm it's still intact

### Part B: Check if 0x058423 is on the ENTER path
1. Statically disassemble 0x058618-0x058630 from the ROM to see if 0x058423 is called from the ENTER handler's setup sequence
2. Cross-reference: session 98 showed the ENTER path calls 0x05840B at step 23. Disassemble 0x05840B to see if it calls 0x058423.
3. If 0x058423 IS on the ENTER path, determine: does it fire BEFORE or AFTER ParseInp/0x0921CB?
