# Task: FP gcd — verify FPSub independently + trace intermediate loop values

## Context
This is a TI-84 Plus CE ROM transpilation project. The transpiler lifts eZ80 ROM code into runnable JavaScript. We're debugging why gcd(12,8) returns the wrong value.

## Problem
The gcd Euclidean algorithm completes 1442 steps but returns OP1=[80 82 10 00 00 00 00 00 00] (sign-corrupted 100.0) instead of [00 80 40 00 00 00 00 00 00] (4.0). Session 151 proved sign-bit masking is insufficient — the FP arithmetic itself produces the wrong numerical result.

## What to build
Create `TI-84_Plus_CE/probe-phase152-gcd-fpsub-verify.mjs` with TWO parts:

### Part A: Independent FPSub verification
Call FPSub (JT slot at ROM 0x020128, implementation at 0x07C771) directly with OP1=12.0 and OP2=8.0. Verify result = 4.0.

BCD encoding for TI-84 reals (9 bytes):
- Byte 0: type/sign (0x00=positive real, 0x80=negative real)
- Byte 1: exponent (biased by 0x80, so 0x81=10^1, 0x80=10^0)
- Bytes 2-8: BCD mantissa digits

So: 12.0 = [00, 81, 12, 00, 00, 00, 00, 00, 00]
    8.0 = [00, 80, 80, 00, 00, 00, 00, 00, 00]
    4.0 = [00, 80, 40, 00, 00, 00, 00, 00, 00]

FPSub computes OP1 = OP1 - OP2. So seed OP1=12.0 at 0xD005F8, OP2=8.0 at 0xD00603 (note: OP registers overlap — session 140 confirmed OP2=0xD00603, though some sessions say 11 bytes; use 9-byte spacing which session 144 confirmed the OS uses for FPS entries).

Entry: call address 0x07C771 (FPSub implementation). Set up with standard probe pattern (cold boot, MEM_INIT, then call).

Read the result from OP1 (0xD005F8, 9 bytes). Expected: [00 80 40 00 00 00 00 00 00] = 4.0.

Also test: OP1=8.0, OP2=4.0, FPSub → should get 4.0.
And: OP1=4.0, OP2=4.0, FPSub → should get 0.0 = [00 80 00 00 00 00 00 00 00] (exponent 0x80 for zero? or [00 00 00...] for zero — check result).

### Part B: Trace intermediate FP values during gcd(12,8)
Call gcd handler at 0x068D3D (same as sessions 148-151) with OP1 and OP2 pre-seeded on the FP stack.

The FP stack (FPS) base is at the address stored in 0xD0258D. FPS entries are 9 bytes. Push OP1=12.0 then OP2=8.0 onto FPS before calling 0x068D3D.

Use onBlock callbacks to log OP1 (0xD005F8, 9 bytes) and OP2 (0xD00603, 9 bytes) at EVERY hit of:
- 0x07F831 (type validator) — should hit at steps ~21/353/740/1064
- 0x07C771 (FPSub entry) or 0x07C77F (FPAdd entry) — log which FP operation is called
- 0x07FA74 (constant loader — loads 1.0 into OP2)
- 0x07CC36 (FP subtraction core — called from 0x07C783)

For each hit, log: step number, PC, full OP1[0..8] hex, full OP2[0..8] hex, decoded decimal values.

Expected Euclidean algorithm trace:
- gcd(12,8): 12 mod 8 = 4 → gcd(8,4): 8 mod 4 = 0 → return 4

## Standard probe pattern
```javascript
import { createExecutor } from './cpu-runtime.js';
import { createPeripheralBus } from './peripherals.js';
const fs = await import('fs');

// Load transpiled ROM
const ROM_PATH = new URL('./ROM.transpiled.js', import.meta.url).pathname.replace(/^\/([A-Z]:)/, '$1');
const mod = await import(`file://${ROM_PATH}`);
const { blocks, seedAddresses } = mod;

const mem = new Uint8Array(16 * 1024 * 1024);
// Load ROM binary
const romPath = new URL('./ROM.rom', import.meta.url).pathname.replace(/^\/([A-Z]:)/, '$1');
const romBuf = fs.readFileSync(romPath);
mem.set(new Uint8Array(romBuf.buffer, romBuf.byteOffset, romBuf.byteLength), 0);

const periph = createPeripheralBus({ timerInterrupt: false });
const FAKE_RET = 0x7FFFFF;
mem[FAKE_RET] = 0xC9; // RET at sentinel

const ex = createExecutor({ mem, peripherals: periph, blocks, seedAddresses });
// ... cold boot, MEM_INIT, then your test calls
```

For cold boot: `ex.callBlock(0x000000, { maxSteps: 500 })`.
For MEM_INIT: Call 0x09DEE0 after cold boot.
