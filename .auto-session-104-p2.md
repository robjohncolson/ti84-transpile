## Context
We are transpiling the TI-84 Plus CE OS ROM (eZ80) into JavaScript. The ENTER key handler converges at the "common tail" starting at 0x058693. Before reaching ParseInp (CALL 0x099910 at 0x0586E3), the common tail makes several calls:
- CALL 0x058C76 (flag helper, very short)
- CALL 0x082961 at ~0x0586A3
- CALL 0x09215E at ~0x0586B3
- CALL 0x082902 (between 0x082961 and 0x09215E region)
- CALL 0x0A1FD1
- CALL 0x0A27DD at ~0x0586D0

Session 103 showed that calling 0x058693 directly gets stuck after 0x082961 calls FindSym which enters an LCD/display loop (0x0BD19F→0x0A2A45→0x0A2A49). ParseInp is never reached.

Also important: at 0x0586CC there's a `JR NZ 0x0586F3` that could SKIP ParseInp if bit 4 of (IY+52) is set.

## Task
Create a probe file `TI-84_Plus_CE/probe-phase25as-common-tail-disasm.mjs` that:

1. Uses the standard probe boilerplate (import cpu-runtime.js, peripherals.js; load ROM.rom and ROM.transpiled.js)
2. Performs STATIC disassembly (reading ROM bytes directly, not running execution) of these functions:
   - **0x082961** — disassemble ~200 bytes, decode eZ80 instructions, identify what it does, what subroutines it calls, what RAM addresses it reads/writes
   - **0x09215E** — disassemble ~150 bytes, same analysis
   - **0x058693 to 0x058700** — the entire common tail, showing all CALLs and the JR NZ at 0x0586CC

3. For the disassembly, use the ROM bytes directly. The eZ80 instruction set is similar to Z80 but with 24-bit addressing. Key instruction encodings:
   - `CD xx yy zz` = CALL addr (3-byte LE address)
   - `C3 xx yy zz` = JP addr
   - `18 dd` = JR dd (signed 8-bit relative)
   - `20 dd` = JR NZ, dd
   - `28 dd` = JR Z, dd
   - `C9` = RET
   - `C0` = RET NZ, `C8` = RET Z, `D0` = RET NC, `D8` = RET C
   - `CB xx` = bit operations (BIT, SET, RES)
   - `FD CB dd xx` = IY-indexed bit ops (BIT n,(IY+d), SET n,(IY+d), RES n,(IY+d))
   - `ED 5B xx yy zz` = LD DE,(addr) [24-bit]
   - `ED 4B xx yy zz` = LD BC,(addr) [24-bit]
   - `ED 6B xx yy zz` = LD HL,(addr) [24-bit]
   - `3A xx yy zz` = LD A,(addr) [24-bit]
   - `32 xx yy zz` = LD (addr),A [24-bit]
   - `21 xx yy zz` = LD HL,imm24
   - `11 xx yy zz` = LD DE,imm24
   - `01 xx yy zz` = LD BC,imm24
   - Prefix `0x40` = SIS (short instruction suffix) — next instruction uses 16-bit operands
   - Prefix `0x52` = SIL
   - Prefix `0x49` = LIS
   - Prefix `0x5B` = LIL (default mode, 24-bit)
   
   For a full decoder, you can use `TI-84_Plus_CE/ez80-decoder.js` if available, or just do manual byte decoding for the key instructions (CALL, JP, JR, RET, LD, BIT/SET/RES on IY).

4. Also run a RUNTIME trace of the common tail at 0x058693:
   - Set up: MEM_INIT, seed allocator pointers (OPBase/pTemp/progPtr = 0xD3FFFF), seed IY=0xD00080, IX=0xD1A860, error frame, tokenized "2+3" at userMem
   - Run 0x058693 for 10,000 steps
   - Log all CALLs/JPs taken (by monitoring PC changes > 16 bytes as "call/jump")
   - Check whether 0x0586CC's JR NZ is taken (check bit 4 of (IY+52) = mem[0xD000B4] before the branch)
   - Log whether ParseInp (0x099914) or its trampoline (0x099910) is ever reached

5. Generate report `TI-84_Plus_CE/phase25as-common-tail-disasm-report.md` with:
   - Full disassembly listings for 0x082961, 0x09215E, and 0x058693-0x058700
   - Annotations identifying: what each function does, what OS globals it reads, why it might get stuck
   - Whether 0x0586CC JR NZ skips ParseInp
   - Recommendations: can we bypass 0x082961 and 0x09215E by calling closer to ParseInp (e.g., 0x0586CE or 0x0586DF)?
