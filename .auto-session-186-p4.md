# Auto-Session 186 — P4: BufInsert "2+3" end-to-end probe (edit buffer validation)

## Context
The TI-84 Plus CE ROM transpilation project has BufInsert (0x05E2A0) working — session 181 confirmed tokens placed correctly and cursor advances. But ParseInp (0x099914) fails with E_Undefined (errNo=0x8D). Session 185 showed the error handler's z80-mode RET pops garbage due to stack page mismatch.

The goal is to bypass the ParseInp error by using the **direct-eval path** that browser-shell.html already uses. Browser-shell evaluates expressions via a JS-side token parser (EVAL_TOKEN_MAP) that calls OS FP routines directly, bypassing ParseInp entirely.

## Task
Build a Node.js probe script `TI-84_Plus_CE/probe-phase186-bufinsert-eval.mjs` that:

1. Boot the ROM with timer interrupts disabled
2. Set up minimal OS state (memInit, IX=0xD1A860)
3. Use BufInsert (0x05E2A0) to insert tokens for "2+3":
   - Token 0x32 (digit 2)
   - Token 0x70 (plus)
   - Token 0x33 (digit 3)
4. Verify the edit buffer contains the correct tokens at the expected addresses
5. Instead of calling ParseInp (which fails), directly evaluate using the OS FP routines:
   - Call `_RclAns` (JT 0x4AD = `JP 0x08238A`, entry at 0x020B48) or `_CreateReal` to push 2.0 onto the FP stack
   - Call FP add routine to compute 2+3
   - Or simpler: just call the individual FP primitives:
     a. Load OP1 = 2.0 by writing to 0xD005F8: `[0x00, 0x80, 0x20, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]`
     b. Load OP2 = 3.0 by writing to 0xD00601: `[0x00, 0x80, 0x30, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]`
     c. Call FPAdd (JT entry 0 at 0x020000 → 0x0009DC)
     d. Read OP1 for result — should be 5.0: `[0x00, 0x80, 0x50, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]`
6. Output structured JSON with:
   - Edit buffer contents after BufInsert
   - OP1/OP2 before FPAdd
   - OP1 after FPAdd (should be 5.0)
   - Pass/fail verdict

## Key addresses
- BufInsert: 0x05E2A0
- Edit buffer: 0xD00A00 (or wherever the cursor points)
- OP1: 0xD005F8 (9 bytes)
- OP2: 0xD00601 (9 bytes)
- FPAdd: JT entry 0 → 0x020000 → target (read the JP instruction at 0x020000)
- FPSub: JT entry 1 → 0x020004
- FPMult: JT entry 2 → 0x020008
- FPDiv: JT entry 3 → 0x02000C
- memInit helper: see existing probes for the pattern

## TI-OS Real Number Format
9 bytes: [type, exponent, mantissa×7]
- type: 0x00 = positive real
- exponent: 0x80 + power_of_10 (0x80 = 10^0 = units)
- mantissa: BCD digits, 2 per byte, 14 digits total
- Example: 5.0 = [0x00, 0x80, 0x50, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]
