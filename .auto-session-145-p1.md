# Auto-Session 145 — P1: FP gcd FpPush/FpPop 9-byte confirmation + OP2 corruption trace

## Context
This is a TI-84 Plus CE ROM transpilation project. The transpiler lifts eZ80 ROM code into JavaScript. We're investigating why `gcd(12,8)` hits E_Domain (error 0x84). Session 144 confirmed FPS entries are 9 bytes (not 11). After FpPop, OP2 reads as 1.0 instead of the expected 8.0. The corruption happens between FpPop exit and the type validator at step 21.

## Task
Build a probe `TI-84_Plus_CE/probe-phase145-fppush-disasm.mjs` that:

### Part A — Disassemble FpPush
1. Use the standard probe pattern (see `probe-phase144-gcd-11byte.mjs` for reference).
2. Cold boot + MEM_INIT as usual: `coldBoot(ex, mem); runUntilHalt(ex, 300000); callFunction(ex, mem, 0x09DEE0, 50000);`
3. Disassemble the FpPush routine starting at ROM address 0x082961. Read raw bytes from the ROM file at `TI-84_Plus_CE/ROM.rom` and decode manually:
   - Start at file offset 0x082961
   - Read ~64 bytes
   - Look for patterns: LD HL,(FPS_ptr), LDIR/LDI chain (copy 9 or 11 bytes), store back FPS_ptr
   - Document: how many bytes does FpPush copy? What is the source (OP1 at 0xD005F8)? What direction does FPS move?
4. Similarly disassemble FpPop at 0x082957 (just 10 bytes before FpPush — it's likely just a few instructions). Look for how many bytes it copies and in which direction.

### Part B — Trace OP2 corruption step-by-step  
1. Set up gcd with properly seeded FPS (9-byte entries):
   - After MEM_INIT, seed allocator (FPSbase, FPS, OPBase, pTemp, etc. to standard values)
   - Push 12.0 BCD at FPSbase, then 8.0 BCD at FPSbase+9, set FPS = FPSbase + 18
   - Seed OP2 (0xD00603) with 8.0 BCD: `[0x00, 0x80, 0x80, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]`
2. Call gcd handler at 0x068D3D
3. Use `onBlock` callback to snapshot OP2 (9 bytes at 0xD00603) at EVERY block for the first 30 steps
4. Report exactly which PC causes OP2 to change from 8.0 to 1.0 (or whatever it changes to)
5. Also snapshot FPS pointer (3 bytes at 0xD0258D) to track push/pop operations

### Part C — Check FpPop copy destination
1. Before calling gcd, zero out both OP1 (0xD005F8, 11 bytes) and OP2 (0xD00603, 11 bytes)
2. Seed FPS with one 9-byte entry (8.0 BCD)
3. Call FpPop (0x082957) directly with a FAKE_RET sentinel
4. After return, check: did FpPop write to OP1 or OP2? (Check both for 8.0 pattern)

## Standard probe boilerplate
```js
import { readFileSync } from 'fs';
import { gunzipSync } from 'zlib';

const ROM_PATH = 'TI-84_Plus_CE/ROM.rom';
const TRANSPILED_PATH = 'TI-84_Plus_CE/ROM.transpiled.js';

// Load transpiled JS (gunzip if needed)
let src;
try { src = readFileSync(TRANSPILED_PATH, 'utf8'); }
catch { src = gunzipSync(readFileSync(TRANSPILED_PATH + '.gz')).toString('utf8'); }

const { createExecutor } = await import('./TI-84_Plus_CE/cpu-runtime.js');
const { createPeripheralBus } = await import('./TI-84_Plus_CE/peripherals.js');

const mem = new Uint8Array(16 * 1024 * 1024);
const rom = readFileSync(ROM_PATH);
mem.set(rom, 0);

const peripherals = createPeripheralBus({ timerInterrupt: false });
const ex = createExecutor(mem, peripherals);
// Load transpiled blocks
new Function('executor', src)(ex);
```

Then cold boot: `ex.coldBoot(); ex.runUntil(() => false, 300000);`
Then MEM_INIT: `ex.callFunction(0x09DEE0, 50000);`
