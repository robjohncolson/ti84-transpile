# Task: Edit Buffer — Integrate Corrected BufInsert into Browser-Shell + Test ParseInp from Buffer

## Context
This is a TI-84 Plus CE ROM transpilation project. Session 131 discovered a critical calling convention fix:
- BufInsert (0x05E2A0) takes the token in **DE** register (E=token byte, D=two-byte flag), NOT BC
- First instruction is PUSH DE, then CALL room-check, then POP BC — the POP overwrites BC with saved DE
- With correct convention (E=token, D=0): cursor advances +1 per token, buffer stores raw token bytes "32 70 33" = "2+3", 7 steps per insertion
- Gap-buffer pointers: editTop=0xD02437, editCursor=0xD0243A, editTail=0xD0243D, editBtm=0xD02440
- Manual pointer init works: editTop=editCursor=bufStart, editTail=editBtm=bufEnd
- CreateNumEditBuf (0x096E09) does NOT initialize pointers

## What you need to do

### Step A: Build a probe testing ParseInp from edit buffer
Create `TI-84_Plus_CE/probe-phase132-bufinsert-parseinp.mjs`. The probe should:

1. Load the transpiled ROM and runtime (see existing probes for import pattern)
2. Set up fresh runtime with `timerInterrupt: false`
3. Run MEM_INIT at 0x09DEE0

4. **Test 1: BufInsert correctness verification**
   - Allocate a 256-byte edit buffer at 0xD00A00
   - Initialize gap-buffer pointers:
     - editTop (0xD02437) = 0xD00A00 (24-bit LE)
     - editCursor (0xD0243A) = 0xD00A00
     - editTail (0xD0243D) = 0xD00B00 (buffer end)
     - editBtm (0xD02440) = 0xD00B00
   - For each token in "2+3\n" (token bytes: 0x32, 0x70, 0x33, 0x3F):
     - Set DE = token (E=token byte, D=0x00)
     - Push FAKE_RET (0x7FFFFF) sentinel
     - Execute BufInsert at 0x05E2A0
     - Read editCursor to verify it advanced by 1
   - Dump buffer contents at 0xD00A00-0xD00A10 to verify raw bytes [0x32, 0x70, 0x33, 0x3F]

5. **Test 2: ParseInp reading from edit buffer**
   - After BufInsert fills the buffer with "2+3\n" tokens:
   - Set parser pointers to read from the edit buffer:
     - begPC (0xD02317) = 0xD00A00 (buffer start, where tokens are)
     - curPC (0xD0231A) = 0xD00A00
     - endPC (0xD0231D) = 0xD00A04 (4 tokens)
   - Set allocator pointers: OPBase/pTemp/progPtr = 0xD3FFFF, FPSbase/FPS = 0xD1A881
   - Clear OP1 (9 bytes at 0xD005F8)
   - Push FAKE_RET sentinel, set up error frame (errSP)
   - Execute ParseInp at 0x099914 with 50,000 step budget
   - Read OP1 and decode as BCD — should be 5.0

6. **Test 3: Compare with direct-token-at-RAM approach**
   - Run the same expression "2+3" using the established direct approach:
     - Write tokens [0x32, 0x70, 0x33, 0x3F] at 0xD00800
     - Set begPC/curPC/endPC pointing at 0xD00800
     - Execute ParseInp
   - Compare: step counts, OP1 results, any differences in behavior

7. **Test 4: CreateNumEditBuf investigation**
   - Call CreateNumEditBuf at 0x096E09 with various HL values (e.g., HL=0xD00A00)
   - Dump editTop/editCursor/editTail/editBtm before and after
   - Also dump 0x096E0C (add-pair) and 0x096E19 referenced addresses
   - Try: seed HL=buffer address before calling, see if it initializes pointers
