# Auto-Session 168 P1: Instruction-Level Single-Step Through Normalize 0x07CA48 During GCD

## Context

This is a TI-84 Plus CE ROM transpilation project. The ROM (eZ80 machine code) has been lifted into JavaScript. We're investigating why `gcd(12,8)` produces E_Domain instead of 4.

**Root cause (15 sessions of investigation)**: The gcd inner body at 0x068D82 calls compound function 0x07C747 which does: copy OP1→OP2, then normalize OP1 via 0x07CA48, then negate OP1, then FPAdd(OP1,OP2). The normalize function zeros OP1's mantissa (shifts all BCD digits out), so FPAdd(0, OP2) = OP2 = original OP1. The compound function is an identity — it preserves OP1 but destroys OP2 by copying OP1 into it. This makes InvSub(OP1, OP1) = 0 instead of computing a mod b.

**Key question**: Is the normalize function at 0x07CA48 executing correctly? Specifically, does our transpiled RLD (Rotate Left Digit) instruction produce correct results at every step? The normalize function uses a Shl14 subroutine at 0x07FB33 which executes a chain of 7 RLD instructions to shift the BCD mantissa left by one digit.

## Task

Build a probe `TI-84_Plus_CE/probe-phase168-norm-singlestep.mjs` that:

1. Sets up gcd(12,8) using the exact same setup as `probe-phase167-stack-balance.mjs` (read that file for reference)
2. Runs execution until PC enters 0x07CA48 (the normalize function entry, first call from compound at step ~62)
3. Then single-steps EVERY instruction through the normalize function (not just blocks — every PC change), logging:
   - PC (current instruction address)
   - The opcode bytes at that PC (read from ROM, 1-4 bytes)
   - Register A value (this receives the shifted-out BCD digit during RLD)
   - Register HL value (RLD operates on (HL))
   - The byte at memory[HL] before and after (for RLD instructions)
   - The exponent byte at 0xD005F9 (OP1 exponent)
   - All 7 mantissa bytes at 0xD005FA-0xD00600

4. Continue single-stepping until the function returns (PC leaves the 0x07CA48-0x07CAB8 range AND leaves the Shl14 subroutine range 0x07FB33-0x07FB65)

5. For each RLD instruction encountered, VERIFY the expected BCD rotation:
   - RLD rotates: A[3:0] → (HL)[3:0], (HL)[7:4] → A[3:0], (HL)[3:0] → (HL)[7:4]
   - Log EXPECTED vs ACTUAL for A and (HL)
   - Flag any mismatch as **BUG FOUND**
