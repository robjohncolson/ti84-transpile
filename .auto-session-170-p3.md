# Task: Conditional Jump Audit During gcd(12,8)

## Context
We're investigating a systemic gcd bug in the TI-84 Plus CE ROM transpiler. Every gcd(a,b) call produces E_Domain instead of the correct result. All individual components have been verified correct in isolation. The bug must be in how components interact — possibly a conditional jump taking the wrong branch due to incorrect flag state from a prior instruction.

## What to build
Create `TI-84_Plus_CE/probe-phase170-cond-jump-audit.mjs` — a probe that logs every conditional branch during gcd(12,8), showing the condition, the flag state, whether taken, and the resulting PC.

## Reference pattern
Copy the FULL boilerplate from `TI-84_Plus_CE/probe-phase169-gcd-helper-disasm.mjs`:
- Same imports (fs, path, url, cpu-runtime, peripherals)
- Same constants (MEM_SIZE=0x1000000, BOOT_ENTRY=0x000000, KERNEL_INIT_ENTRY=0x08c331, POST_INIT_ENTRY=0x0802b2, STACK_RESET_TOP=0xd1a87e, OP1_ADDR=0xd005f8, OP2_ADDR=0xd00603, GCD_ENTRY=0x068D3D, ERR_NO_ADDR=0xd008df, FPS_ADDR=0xd0258d, etc.)
- Same `createRuntime()`, `coldBoot()`, `prepareCallState()` functions
- Same hex/BCD helpers

## Probe logic

1. Cold boot the system
2. Set up gcd(12,8): OP1=12 (bytes: [0x00, 0x81, 0x12, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]), OP2=8 (bytes: [0x00, 0x80, 0x80, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]), fpCategory mem[0xD0060E]=0x28

3. Read the ROM bytes at known conditional jump locations in the gcd code to identify all conditional branch opcodes. The key eZ80 conditional branch opcodes are:
   - JR Z (0x28), JR NZ (0x20), JR C (0x38), JR NC (0x30)
   - JP Z (0xCA), JP NZ (0xC2), JP C (0xDA), JP NC (0xD2)
   - CALL Z (0xCC), CALL NZ (0xC4), CALL C (0xDC), CALL NC (0xD4)
   - RET Z (0xC8), RET NZ (0xC0), RET C (0xD8), RET NC (0xD0)
   - Also: JP PO (0xE2), JP PE (0xEA), CALL PO (0xE4), CALL PE (0xEC), RET PO (0xE0), RET PE (0xE8)
   - Also: JP P (0xF2), JP M (0xFA), CALL P (0xF4), CALL M (0xFC), RET P (0xF0), RET M (0xF8)

4. Run gcd step-by-step. At each step, BEFORE executing:
   - Read the ROM byte at current PC to check if it's a conditional branch opcode
   - If yes, log BEFORE state: step, PC, opcode name, F register (hex + decoded S/Z/H/P/N/C bits), the condition being tested, the relevant flag value

5. AFTER each step:
   - If the previous step was a conditional branch, log the resulting PC
   - Determine if the branch was TAKEN (PC changed to target) or NOT TAKEN (PC advanced past the branch instruction)

6. For each conditional branch encountered, print a formatted line:
   ```
   Step NNN: PC=0xXXXXXX  JR Z +NN  F=0xXX (S=_ Z=_ H=_ P=_ N=_ C=_)  → TAKEN to 0xXXXXXX / NOT TAKEN → 0xXXXXXX
   ```

7. After execution, print summary:
   - Total conditional branches encountered
   - Branches TAKEN vs NOT TAKEN counts
   - Group by PC address — show how many times each conditional branch location was hit and its taken/not-taken pattern
   - Highlight any branch that was taken with DIFFERENT flag states at different visits (potential bug indicator)
   - Error code, final OP1/OP2

## Known conditional jumps in gcd code
- 0x068D65: JR Z → 0x068D5D (exit if type check fails)
- 0x068D67: JR C → 0x068D5D (exit if carry)
- 0x068D73: JR NC → 0x068D5D (exit if exp ≥ 0x84)
- 0x068D79: JR C → 0x068D7E (skip INC B if exp < 0x82)
- 0x068DA1: JR NZ → 0x068D5D (loop back if frac ≠ 0)
- 0x068DDF: CALL Z → 0x07FCF8 (OP1ExOP6 if Z set)

But there will be many MORE conditional branches inside the called subroutines (normalization loops, FPAdd, etc.).

## Deliverable
A single file `TI-84_Plus_CE/probe-phase170-cond-jump-audit.mjs` that runs with `node TI-84_Plus_CE/probe-phase170-cond-jump-audit.mjs` and prints the complete conditional jump audit.
