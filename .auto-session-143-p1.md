# Session 143 P1: FP gcd — Fix FPS Seeding Order

## Context

The TI-84 ROM transpilation project lifts eZ80 ROM to JS. In `TI-84_Plus_CE/probe-phase142-gcd-trace.mjs`, there's a bug: `seedFPSOperands()` is called BEFORE `seedAllocator()`, but `seedAllocator()` overwrites FPS/FPSbase to USERMEM (0xD1A881), clobbering the operands that were just seeded at `FPS_CLEAN_AREA` (0xD1A900).

## The Bug (lines 239-240 of probe-phase142-gcd-trace.mjs)

```js
seedFPSOperands(mem);   // writes 12.0 and 8.0 at 0xD1A900, sets FPS ptr = 0xD1A918
seedAllocator(mem);     // CLOBBERS FPS/FPSbase back to 0xD1A881 ← BUG
```

FpPop then reads from empty stack → OP2 = garbage (1.0) instead of 8.0 → gcd(12,1) → 0 → E_Domain.

## Your Task

Create a NEW probe file `TI-84_Plus_CE/probe-phase143-gcd-fixed.mjs` based on `probe-phase142-gcd-trace.mjs`. The ONLY changes should be:

1. **Swap the seeding order**: call `seedAllocator(mem)` FIRST, then `seedFPSOperands(mem)` SECOND. This ensures FPS/FPSbase end up pointing to the clean area with both operands.

2. **Also add a Test B**: After the first test (direct gcd handler at 0x068D3D), add a second test that calls the dispatch entry at 0x06859B with the same corrected seeding order. The dispatch entry expects A=0x28 (gcd category code), then CALL 0x0689DE → SUB 0x20 → index jump table → JP 0x068D3D.

3. **Report clearly**: For each test, print:
   - Whether OP1 = 4.0 (BCD: `[00 80 40 00 00 00 00 00 00]`) = gcd(12,8)
   - errNo value
   - Steps taken
   - FpPop hit count
   - Whether the test returned normally or via error catch

## File Structure

Copy the existing probe's boilerplate (imports, constants, createRuntime, coldBoot, prepareCallState, seedErrFrame, seedAllocator, seedFPSOperands, runMemInit). The probe should:
1. Cold boot + MEM_INIT
2. Test A: direct 0x068D3D with corrected seeding order
