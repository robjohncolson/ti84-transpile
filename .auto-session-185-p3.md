# Session 185 P3: OS Tokenizer Discovery — ConvKeyToTok at 0x05C52C

## Context

This is a TI-84 Plus CE ROM transpilation project. The browser-shell currently uses a hand-rolled `EVAL_TOKEN_MAP` (in browser-shell.html, line ~1543) to convert typed characters to TI-OS token bytes. The OS has a real tokenizer called `ConvKeyToTok` at 0x05C52C (referenced in phase133/136 reports at 0x087643: `call 0x05c52c`).

Priority #2 from the continuation prompt: "Find the tokenizer entry point, validate it against direct-eval's hand map for the digits/operators, then route browser-shell input through it."

## Task

Build a probe `probe-phase185-tokenizer-discovery.mjs` that:

### Part 1: Disassemble ConvKeyToTok (0x05C52C)
- Read ROM bytes from 0x05C52C forward (at least 60 instructions or until RET)
- Use `ez80-decoder.js` to decode each instruction
- Document the calling convention: what register holds the key code input? What register/memory holds the token output?
- Look for lookup tables, range checks, or dispatch patterns

### Part 2: OS Jump Table verification
- Check if 0x05C52C appears in the 980-entry OS jump table
- The jump table starts at ROM address 0x000000 (or wherever the PRELIFTED_BLOCKS seeds originate)
- Grep the transpiler seeds or ROM bytes for references to 0x05C52C

### Part 3: Test ConvKeyToTok with known key codes
After cold boot → kernel init → MEM_INIT:
- Call ConvKeyToTok (0x05C52C) with each of these key codes and record the output token:

| Key | Scan code | Expected token |
|-----|-----------|----------------|
| k0 | 0x8E | 0x30 |
| k1 | 0x8F | 0x31 |
| k2 | 0x92 | 0x32 (session 131 convention: token = keyCode - 0x8E + 0x30) |
| k9 | 0x97 | 0x39 |

| kEnter | 0x86 (guess) | 0x05 (tEnter) |

For each test:
1. Set up CPU registers (A = key code, or whatever the calling convention requires)
2. Push a fake return address (0x7FFFFE)
3. Call runFrom(0x05C52C, 'adl', { maxSteps: 200 })
4. Read the output register/memory location
5. Record: key code → token byte, steps taken, errNo

### Part 4: Scan code → key code mapping
The TI-84 keyboard has scan codes (from _GetCSC / hardware scan) and key codes (from the OS key handler). Document:
- Where does the scan code → key code conversion happen?
- Check `keyboard-matrix.md` for the scan code layout
- Check if 0x0159C0 (foreground scanner) outputs scan codes or key codes

### Part 5: Validate against hand-rolled map
Compare the OS tokenizer results against the hand-rolled `EVAL_TOKEN_MAP` in browser-shell.html:
```javascript
const EVAL_TOKEN_MAP = {
  '0': 0x30, '1': 0x31, '2': 0x32, ..., '9': 0x39,
  '+': 0x70, '-': 0x71, '*': 0x82, '/': 0x83,
  '.': 0x3A, '(': 0x10, ')': 0x11,
  ...

```
Report any discrepancies.
