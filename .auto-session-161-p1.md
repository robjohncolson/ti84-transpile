# Task: Intercept gcd to use 0x07C755 instead of 0x07C747

## Context

The TI-84 CE ROM gcd handler at 0x068D3D calls OP1toOP2 at 0x07C747. This is a COMPOUND function:
1. CALL 0x07F8FA (copy OP1 → OP2)
2. CALL 0x07CA48 (normalize — zeros mantissa for ALL inputs with exp >= 0x80)
3. CALL 0x07CA06 (InvOP1S — negate OP1 sign)
4. JR 0x07C77F (FPAdd)

The normalization at step 2 destroys OP1 mantissa for integer inputs (exp >= 0x80), zeroing it. This causes gcd(12,8) to produce E_Domain error instead of 4.

There is an ALTERNATE entry at 0x07C755 that does: copy OP1→OP2 then JR to FPAdd (NO normalize, NO negate). Session 160 P3 confirmed this.

The gcd handler calls OP1toOP2 from 0x068D61 via `CALL 0x07C747`. We want to test: if gcd calls 0x07C755 instead, does gcd(12,8) produce the correct result (4)?

5. Set up gcd call: OP1=12.0 ([0x00, 0x81, 0x12, ...zeros]), OP2=8.0 ([0x00, 0x80, 0x80, ...zeros])

## Intercept technique

In the `onBlock` callback:
```javascript
onBlock(pc, block) {
  if (pc === 0x07C747) {
    const altBlock = BLOCKS['07c755:adl'];
    if (altBlock) {
      interceptCount++;
      console.log(`  [INTERCEPT] step ${stepCount}: redirecting 0x07C747 → 0x07C755`);
      return altBlock;  // return the alternate block to execute instead
    }
  }
  // normal execution
}
```

If block `07c755:adl` doesn't exist, try `07c755:z80` or check if the block key uses a different format. The probes in this repo use `${hex6}:adl` format (lowercase hex, no 0x prefix).
