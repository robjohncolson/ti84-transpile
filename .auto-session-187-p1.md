# Auto-Session 187 P1 — PC Intercept at 0x03E1B1 to Fix JError z80-mode RET

## Context

The TI-84 Plus CE ROM transpilation project has a known bug: when ParseInp encounters an error (e.g., E_Undefined), the JError handler chain fails because of a broken z80-mode RET at address 0x03E1B1.

The chain is: JError (0x061DB2) → CALL wrapper (0x03E1B4) → CALL helper (0x03E187 via RSMIX) → z80-mode port cleanup → z80-mode RET at 0x03E1B1. The z80-mode RET pops a 2-byte value (garbage) instead of properly returning to ADL mode with a 3-byte address. The correct return address is 0x03E1CA (the instruction after `CALL 0x03E187` in the wrapper).

Session 186 fully traced the SP values through this chain. The solution: hard-code a PC intercept at 0x03E1B1 that sets PC=0x03E1CA and switches to ADL mode.

## Task

Build a probe `TI-84_Plus_CE/probe-phase187-jerror-intercept.mjs` that:

1. Sets up CPU + memory the same way as `probe-phase186-bufinsert-eval.mjs` (read that file for the pattern — it already seeds edit buffer state, calls BufInsert, etc.)
2. Inserts tokens "2+3" into the edit buffer using BufInsert (0x05E2A0) — same as session 186 did
3. **Before calling ParseInp**, sets up a PushErrorHandler frame. PushErrorHandler is at JT entry 0x0201C4 → look it up in the ROM.transpiled.js PRELIFTED_BLOCKS. Call it with HL = some recovery address (e.g., a block that just returns). Also set errSP (at 0xD1A869 per session 186) to point to the error frame.
4. Calls ParseInp at 0x099914 (direct entry)
5. **Intercepts the z80-mode RET block**: In the executor loop, when `nextPC` would be the result of executing block `0x03E1B1` in z80 mode (or when the block key is `'3e1b1:z80'` or similar), override the return: set `cpu.pc = 0x03E1CA`, set `cpu.madl = 1` (ADL mode), and continue execution
6. Logs what happens after the intercept — does JError's LD SP,(errSP) at 0x061DCA fire now? Does the error handler route to 0x099929?
7. Reports: did ParseInp complete with error recovery? What's errNo? What's in OP1? Did we avoid the crash?

## Key addresses and conventions

- Edit buffer: 0xD00A00
- Cursor pointer: 0xD00A87 (2 bytes, little-endian)  
- BufInsert: 0x05E2A0 (JT entry at 0x020190)
- ParseInp: 0x099914 (direct entry)
- JError: 0x061DB2
- Error wrapper: 0x03E1B4
- Error helper: 0x03E187 (RSMIX to z80)
- z80-mode RET block: 0x03E1B1 (THIS IS THE INTERCEPT TARGET)
- Correct return after intercept: PC=0x03E1CA, ADL mode
- errSP: address 0xD1A869
- errNo: check 0xD0008C (IY+0 offset, with IY=0xD00080, so IY+0x0C = 0xD0008C)
- PushErrorHandler: JT entry at 0x0201C4
- Error recovery target: 0x099929 (the hlPayload from session 183's trace)
- Stack: 0xD1A87E (top)
- IX = 0xD1A860 (per-stage convention for home screen)
- IY = 0xD00080 (OS flags base)
- Timer interrupt: DISABLED (use `createPeripheralBus({ timerInterrupt: false })`)
- MBASE = 0xD0 (default in cpu-runtime.js since session 177)
