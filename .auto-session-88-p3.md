# Task: ParseInp expression-only path — detailed PC trace around StoAns with/without Ans pre-created

## Context
TI-84 Plus CE ROM transpilation project. eZ80 machine code lifted to JavaScript. Runtime at `TI-84_Plus_CE/cpu-runtime.js`, peripherals at `TI-84_Plus_CE/peripherals.js`, FP decode at `TI-84_Plus_CE/fp-real.mjs`.

## Background
Session 86's store trace (`probe-phase25aa-parseinp-store-trace.mjs`) showed:
- errNo=0x8D is set at step 27 (PC=0x03E1B4) by ChkFindSym failing on OP1=zeros
- ParseInp catches this error internally and continues parsing for 881 more steps
- StoAns starts at step 869 (PC=0x099A9B, sets OP1[0]=0xFF for Ans type)
- ChkFindSym for Ans at ~step 878 fails (Ans doesn't exist in VAT)
- StoAns is handled gracefully even when Ans doesn't exist — no second errNo write
- Total: 918 steps, errNo=0x8D, OP1=5.0

Session 87's P1 showed: creating Ans before ParseInp → 902 steps (16 fewer), errNo=0x8D still persists.

## Question
If Ans exists in VAT, does the StoAns sequence at steps ~869-917 take a DIFFERENT code path? Does it actually STORE 5.0 into the Ans variable? The 16-step difference (918 vs 902) suggests something changes, but we don't know exactly what.

## What to build
Create probe: `TI-84_Plus_CE/probe-phase25ac-stoans-trace.mjs`

Run TWO scenarios, each recording a FULL PC trace for EVERY block visited:

**Scenario A (control)**: MEM_INIT → ParseInp("2+3") with OP1=zeros, no variables
- Expected: 918 steps, the baseline trace

**Scenario B**: MEM_INIT → CreateReal(Ans) → clear OP1 to zeros → ParseInp("2+3")
- Ans OP1 for CreateReal: `[0x00, 0x72, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]`

For each scenario:
1. Record EVERY PC visited with step number using the onBlock callback: `onBlock(pc, mode, meta, step)`
2. After the run, find the StoAns entry point — look for step where OP1[0] transitions to 0xFF (around step 869 in scenario A). Use a per-step OP1[0] tracker in onBlock.
3. Extract the sub-trace from step (StoAns_start - 20) to end — these are the last ~70 steps
4. Compare the two sub-traces side by side in the report

Also report for each scenario:
- Total steps, errNo, OP1 bytes + decoded
- Pointer snapshots before/after ParseInp
- The full list of UNIQUE PCs visited (deduplicated, sorted)
- Whether the Ans variable's data slot (from CreateReal's DE) has any non-zero data after ParseInp (in scenario B only)

**Tracking OP1[0] per step**: In the onBlock callback, read `mem[0xD005F8]` each step. Record the step number and new value whenever it changes. This gives us the exact step where StoAns begins (OP1[0] → 0xFF) and where it finishes (OP1[0] → 0x00).

**Key addresses:**
- MEM_INIT_ENTRY = 0x09DEE0
- CREATEREAL_ENTRY = 0x08238A  
- PARSEINP_ENTRY = 0x099914
- OP1_ADDR = 0xD005F8
- ERR_NO_ADDR = 0xD008DF, ERR_SP_ADDR = 0xD008E0
- TOKEN_BUFFER_ADDR = 0xD00800
- BEGPC/CURPC/ENDPC = 0xD02317/0xD0231A/0xD0231D
- TEMPMEM/FPSBASE/FPS = 0xD02587/0xD0258A/0xD0258D
- OPBASE/OPS = 0xD02590/0xD02593
- PTEMP/PROGPTR = 0xD0259A/0xD0259D
- NEWDATA_PTR = 0xD025A0
- FAKE_RET = 0x7FFFFE, ERR_CATCH_ADDR = 0x7FFFFA

**Boot sequence** (from `probe-phase25ab-parseinp-recall.mjs`):
- Cold boot: runFrom(0x000000, 'z80', maxSteps 20000, maxLoopIterations 32)
- Reset: halted=false, iff1/iff2=0, sp=0xD1A87E-3, fill 0xFF at sp..sp+3
- Kernel init: runFrom(0x08C331, 'adl', maxSteps 100000, maxLoopIterations 10000)
- Set: mbase=0xD0, iy=0xD00080, hl=0, halted=false, iff1/iff2=0, sp reset, fill 0xFF
- Post init: runFrom(0x0802B2, 'adl', maxSteps 100, maxLoopIterations 32)

**Before each OS call**: halted=false, iff1/iff2=0, madl=1, mbase=0xD0, iy=0xD00080, f=0x40, ix=0xD1A860, sp=STACK_RESET_TOP-12, fill 0xFF at sp for 12. Push return addr, setup minimal error frame.

**Minimal error frame**: push FAKE_RET on stack, write 6-byte frame below: [00 00 00 | ERR_CATCH_ADDR(0x7FFFFA)], set errSP(0xD008E0) to frame base, clear errNo.

**Timer IRQ**: MUST disable: `createPeripheralBus({ timerInterrupt: false })`
**Token bytes for "2+3"**: `[0x32, 0x70, 0x33, 0x3F]`
**mem.fill**: uses (val, start, END) — third arg is END not length
**STACK_RESET_TOP** = 0xD1A87E

Write report to `TI-84_Plus_CE/phase25ac-stoans-trace-report.md`.
