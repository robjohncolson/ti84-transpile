# Auto-Session 99 P3: Run ENTER dispatch with 500K steps + seed 0xD01D0B=0x00

## Context
This is a TI-84 Plus CE ROM transpilation project. Session 98 confirmed the ENTER dispatch chain reaches 0x099211 (DisarmScroll) at step 54, then calls 0x0921CB at 0x05862B. The probe ran out of budget at 200K steps in the allocator loop (0x0827xx region, 251 unique PCs).

0x0921CB starts with `CALL 0x08384B` (ChkFindSym area), then checks `LD A,(0xD01D0B); OR A; RET Z`. If 0xD01D0B is zero, 0x0921CB returns immediately and the ENTER handler continues at 0x05862F.

## Task
Create a probe that runs TWO scenarios:

### Scenario A: 500K step budget (no extra seeding)
1. Set up identical to session 98 P1 probe (`probe-phase25am-enter-via-0585E9.mjs`):
   - Cold boot, kernel init, post-init, MEM_INIT
   - Full cx seed (cxMain=0x0585E9, cxCurApp=0x40 initially then 0x00 for second pass)
   - A=5 (kEnter) in register
   - Tokenized "2+3" at userMem (0xD1A881): bytes [0x32, 0x70, 0x33, 0x3F]
   - begPC/curPC = 0xD1A881, endPC = 0xD1A885
   - Error frame seeded (errSP pointing to a 6-byte catch frame with FAKE_RET)
   - OPS at 0xD3FFFF (MEM_INIT default), FPSbase/FPS at 0xD1A881
2. Run 0x0585E9 directly (not CoorMon) with max_steps=500000
3. Track: did ParseInp (0x099914) or 0x0973C8 get reached?
4. Track: did the allocator loop finish? What PC did execution reach?
5. Log errNo, OP1, and all allocator pointers after completion

2. ADDITIONALLY: write 0x00 to RAM address 0xD01D0B before calling 0x0585E9
3. Run with max_steps=500000
4. Track same targets as Scenario A
5. Key question: does 0x0921CB return quickly? Does the ENTER handler reach ParseInp?
