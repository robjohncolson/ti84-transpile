## Context

You are working on a TI-84 Plus CE ROM transpilation project at `C:/Users/rober/Downloads/Projects/school/ti84-transpile`. The project lifts eZ80 machine code into JavaScript.

We've been investigating a gcd bug. The inner body call sequence starting at 0x068D82 is:
1. CALL 0x07F8A2 (OP1→OP4, saves OP1)
2. CALL 0x07C747 (compound: OP1→OP2 copy + normalize + InvOP1S + FPAdd)
3. CALL 0x07F95E (OP1→OP3)
4. CALL 0x07F8B6 (OP4→OP2, restores saved OP1 into OP2)
5. CALL 0x07C74F (InvSub = negate OP1 then add, i.e., OP2 - OP1)

The compound call at step 2 (0x07C747) copies OP1 into OP2, then normalizes OP1 to zero, then FPAdd(0, OP2) = OP2, and copies OP2→OP1. So OP1 is restored to its original value. But OP2 now equals OP1. Then step 4 overwrites OP2 with OP4 (which is the ORIGINAL OP1 from step 1). So at InvSub, both OP1 and OP2 are the same value (OP1), producing mod=0.

The question: **Is the compound function at 0x07C747 SUPPOSED to be called in this context?** Maybe the ROM bytes at 0x068D82 actually encode a DIFFERENT call target, and our transpiler decoded it wrong.
