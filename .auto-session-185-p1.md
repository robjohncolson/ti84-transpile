# Session 185 P1: MADL PC-Based Runtime Intercept for z80-mode RET After RSMIX

## Context

The TI-84 Plus CE uses an eZ80 CPU. The transpiler lifts ROM blocks to JavaScript. When a block contains RSMIX (switch from ADL to z80 mode), the subsequent z80-mode RET pops wrong data because the eZ80 MADL suffix byte mechanism is not implemented. On real hardware, CALL pushes [MADL_byte][return_addr] and RET pops MADL first to determine the caller's mode + address width.

**Session 173** tried a global MADL fix (push extra byte on ALL CALLs) — it broke golden regression because it corrupted manually-pushed return addresses in probes/browser-shell.

**Session 183** fixed z80-mode stack addressing: z80-mode push/pop now uses `{MBASE, SPS[15:0]}` instead of SPL addressing. This is correct but the MADL byte is still missing.

**Session 184** confirmed MADL is the only correct fix. z80-mode RET at 0x03E1B1-0x03E1B3 pops garbage (0x0000 from uninitialized 0xD0xxxx page) because the MADL suffix byte was never pushed.

## The Fix: PC-Based Runtime Intercept

Instead of modifying all 151 RSMIX CALL/RET pairs, add a **targeted intercept** in `popReturn()` in `cpu-runtime.js`. When the CPU is in z80 mode (`!this.madl`) and the `_currentBlockPc` matches one of the known problematic z80-mode RET addresses, override the behavior to pop the correct return value from the ADL stack page.

### Implementation

In `cpu-runtime.js`, modify `popReturn()` (currently at line 749-751):

```javascript
popReturn() {
  if (!this.madl) {
    // MADL intercept: z80-mode RET after RSMIX needs to switch back to ADL
    // and pop 3 bytes from the ADL stack (SPL-addressed), not 2 bytes from
    // the z80 {MBASE, SPS} page. This simulates the MADL suffix byte that
    // real eZ80 hardware pushes on CALL.
    //
    // The caller (runFrom loop) already tracks mode via resolveNextMode,
    // but the POP width/address is wrong. Fix: temporarily force ADL-style
    // pop (3 bytes from SPL) when a z80-mode block returns.
    const addr = this.sp; // Use SPL directly (24-bit), not {MBASE, SPS}
    const value = this.read24(addr);
    this.sp = (this.sp + 3) & 0xffffff;
    this.madl = 1; // Switch back to ADL mode (as MADL byte would indicate)
    return value;
  }
  return this.pop();
}
```

### Why This Is Safe

1. z80-mode blocks that do PUSH/POP pairs (register saves) use `push()` and `pop()` — those stay unchanged and correctly use `{MBASE, SPS}` addressing.
2. Only `popReturn()` (used by RET/RETI/RETN instructions) is affected — this is exactly where MADL byte would be consumed.
3. The `madl = 1` switch-back is what the MADL byte encodes: "the caller was in ADL mode."
4. The runFrom loop at line 1036 sets `cpu.madl = mode === 'adl' ? 1 : 0` at the TOP of each iteration, so the explicit `this.madl = 1` here won't persist incorrectly — the next block dispatch will set it correctly from the resolved mode.

**WAIT** — there's a subtlety. Not every z80-mode RET is returning to ADL mode. Some z80-mode subroutines call other z80-mode subroutines. The above blanket approach would break those. Let me think...

Actually, in this ROM, the ONLY way to enter z80 mode is via RSMIX from ADL mode. There are 151 RSMIX sites and 0 STMIX sites. So every z80-mode RET is returning to an ADL-mode caller. This means the blanket "z80 RET → ADL pop" approach should be safe for this specific ROM.

But we need to be careful: z80-mode CALLs within z80 code push 2-byte return addresses via `push()` (which correctly uses MBASE addressing). Those z80→z80 RETs also use `popReturn()`. For z80→z80 returns, we should NOT switch to ADL.

**Revised approach**: Track whether the most recent CALL was a mode-crossing CALL. This is too complex. Let's use a simpler heuristic:

**Simplest safe approach**: Add a configurable map of z80-mode PC addresses where RET should do ADL-style pop. Start with 0x03E1B1 (the known problematic site). The probe can test this specific intercept.
