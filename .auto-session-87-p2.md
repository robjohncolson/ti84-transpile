# Task: ParseInp with OP1=variable name + stored value — does it recall?

## Context

TI-84 Plus CE ROM transpilation project. Session 86 confirmed that when OP1 contains a valid variable name (e.g., "A") and that variable exists in the VAT, ParseInp takes only 253 steps and returns errNo=0x00 without computing the expression "2+3". It reads begPC/curPC from VAT data. But in that test, the variable's data was all zeros (CreateReal allocates 9 zero bytes).

## Your task

Create a probe `TI-84_Plus_CE/probe-phase25ab-parseinp-recall.mjs` that tests whether the "found" path recalls the stored value:

1. Cold boot the ROM
2. Call MEM_INIT at 0x09DEE0 (~17 steps)
3. Call CreateReal at 0x08238A with OP1=[0x00, 0x41, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00] (RealObj, name "A"). Budget: 50,000 steps.
4. After CreateReal returns, note the DE register value — it points to the 9-byte data area of the newly created variable. Write 42.0 there in TI BCD format: `[0x00, 0x81, 0x42, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]`.
5. Set OP1 to [0x00, 0x41, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00] again (variable name "A")
6. Seed token buffer at 0xD00800 with [0x32, 0x70, 0x33, 0x3F] ("2+3" with end token)
7. Set begPC/curPC (0xD02317/0xD0231A) to 0xD00800, endPC (0xD0231D) to 0xD00804
8. Call ParseInp at 0x099914. Budget: 1,500,000 steps.
9. Check:
   - Does OP1 contain 42.0 (recalled from stored variable)?
   - Or does OP1 stay as [00 41 ...] (unchanged variable name)?
   - Or does OP1 become 5.0 (computed 2+3)?
   - errNo value?
   - Step count (should be ~253 if taking the "found" path)?

Also read the 9 bytes at the DE address after ParseInp to check if the variable data was modified.

Write a markdown report to `TI-84_Plus_CE/phase25ab-parseinp-recall-report.md`.

## Template

Copy structure from `TI-84_Plus_CE/probe-phase25aa-empty-op1-with-var.mjs`:

- Same error-frame seeding pattern (errSP, FAKE_RET, ERR_CATCH_ADDR)
