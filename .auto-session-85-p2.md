# Task: Investigate ParseInp's internal PushErrorHandler mechanism

## Context
The TI-84 Plus CE ROM transpiler lifts eZ80 machine code to JavaScript. ParseInp (0x099914) computes 2+3=5.0 in 918 steps using the committed probe `TI-84_Plus_CE/probe-phase25x-meminit-then-parseinp.mjs`. This probe uses a MINIMAL 6-byte error catch frame (just FAKE_RET pushed on stack). Yet the parser internally recovers from ErrUndefined (0x8D) without crashing.

Session 84 found that JError/PushErrorHandler addresses 0x061DB2-0x061E27 are NOT in transpiled blocks. The executor must be interpreting these blocks from raw ROM bytes via its missing-block fallback interpreter.

## What to investigate

1. Read `TI-84_Plus_CE/cpu-runtime.js` and find the missing-block fallback/interpreter code. Look for what happens when the executor encounters a PC that has no transpiled block (no entry in `BLOCKS`). Document:
   - How does it handle missing blocks? Does it interpret raw ROM bytes instruction-by-instruction?
   - How many instructions can it execute in fallback mode before stopping?
   - Is there a step limit or does it run until it hits a transpiled block?
   - Does it support the full eZ80 instruction set or only a subset?

2. Read the ROM bytes at these addresses and statically decode them:
   - 0x061DB2 (JError/longjmp) — session 76 already decoded this as: `ld (0xD008DF),a; call 0x03E1B4; [clear IY bits]; ld sp,(0xD008E0); pop af; ret`
   - 0x061DD1 (PopErrorHandler/error-restore stub) — NOT yet decoded. Decode the bytes here.
   - 0x061DEF (PushErrorHandler) — session 79 decoded: pushes 18-byte ADL frame
   - 0x061E27 (normal-return cleanup stub) — NOT yet decoded. Decode the bytes here.

   The ROM file is at `TI-84_Plus_CE/ROM.rom`. Read bytes starting at offset 0x061DB2 through 0x061E40 and decode as eZ80 ADL-mode instructions.

3. Write a probe `TI-84_Plus_CE/probe-phase25z-executor-fallback.mjs` that:
   - Uses the same boot + MEM_INIT setup as `probe-phase25x-meminit-then-parseinp.mjs`
   - Runs ParseInp with step-by-step PC logging enabled
   - Counts how many steps are spent at PCs in the 0x061D00-0x061E40 range
   - Reports whether the executor's fallback interpreter or transpiled blocks handle those PCs

   Use the exact same imports and setup pattern as `probe-phase25x-meminit-then-parseinp.mjs`.

4. Write a report `TI-84_Plus_CE/phase25z-executor-fallback-report.md` with:
   - How the missing-block fallback works in cpu-runtime.js
   - Full decode of 0x061DD1 and 0x061E27
   - How many steps ParseInp spends in the error handler region
   - Assessment: is the 918-step success reliable or partly lucky?
