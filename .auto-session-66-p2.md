# Phase 25G-e — Find the actual dispatcher to 0x00B608

## Context

Phase 25G-b proved that 0x00B608 is NOT reached from the OS event loop (0x19BE). The event loop cycles 0x19BE→0x19EF→0x1A17→0x1A23→0x1A2D→0x1A32→0x19B6 HALT without dispatching to 0x00B608.

I've now traced the static call graph and found that 0x00B608 lives in a large FP (floating-point) math function region (0xADxx–0xB6xx). The call chain is:

```
0x00AE16 → 0x00AE24 → (call 0x0021c2) → 0x00AE2B → (jp z) → 0x00B55F → 0x00B566 → 0x00B56A → 0x00B586 → (call 0x015349) → 0x00B58C → (call 0x0025ca) → 0x00B5A8 → 0x00B5EF → 0x00B5F6 → (call 0x00276b) → 0x00B5FF → 0x00B608
```

Key observations:
- 0x00ADEF ends with `rst 0x28` (floating-point dispatch entry)
- Blocks use FP workspace pointers at 0xD1770A, 0xD17716, 0xD1771A, 0xD1772A
- The entire 0xADxx region has NO external callers — it's all seeds added for coverage
- 0x00AE0F/0x00ADB9 are entry blocks with no inbound jumps — they're reached via indirect dispatch

## Your task

Create a probe `TI-84_Plus_CE/probe-phase25g-e-dispatch.mjs` and a report `TI-84_Plus_CE/phase25g-e-report.md`.

The probe should:

1. **Search the transpiled JS** for all blocks that jump to addresses in 0x00AD00–0x00B700 via indirect mechanisms. Specifically:
   - Grep `ROM.transpiled.js` for blocks containing `return cpu.` patterns (indirect jumps via register)
   - Grep for `jp (hl)` / `jp (ix)` / `jp (iy)` disassembly comments near these addresses
   - Grep for any block that loads 0x00AD-0x00B6 range addresses into HL/IX/IY before an indirect jump

2. **Search for the rst 0x28 dispatch mechanism**. `rst 0x28` jumps to 0x000028. Find `block_000028` in the transpiled JS and trace what it does — it likely reads the return address from the stack, uses it as a pointer to an FP opcode byte, then dispatches to a handler. The 0xADxx region would be one such handler.

3. **Search for address 0x00ADB9 or nearby** in:
   - The OS jump table (0x020000+ region) — maybe it's an OS entry point
   - Any dispatch tables in ROM (look for consecutive 3-byte address entries)
   - RAM pointer 0xD176AB (which 0x00ADB9 reads — maybe the FP stack base)

4. **Try to reach 0x00B608 dynamically**: Set up the FP workspace pointers (0xD1770A, 0xD17716, 0xD1771A, 0xD1772A, 0xD176AB) with plausible values, then try executing from 0x00ADB9 directly. Log whether execution reaches 0x00B608.
