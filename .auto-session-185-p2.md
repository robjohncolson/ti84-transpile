# Session 185 P2: Token Pipeline Probe — What Populates 0xD0230E Before Key Classifier

## Context

This is a TI-84 Plus CE ROM transpilation project. The ROM is lifted to JavaScript blocks that execute in a `runFrom()` executor loop.

Session 184 decoded the key classifier at 0x07F7BD: it's only 3 instructions:
```
LD A,(0xD005F8)   ; read OP1[0] type byte
AND 0x3F
RET
```

Before the classifier is called (at 0x0584A3 in the home handler), a copy helper at 0x07F9FB copies 9 bytes from **0xD0230E** → OP1 (0xD005F8):
```
0x0584A3: LD HL,0xD0230E; CALL 0x07F9FB
```

**The question**: What populates 0xD0230E with the token data for a keypress? This is the key-to-token pipeline gap. The home handler at 0x058483+ processes keys, but we don't know what runs BEFORE 0x0584A3 to convert a raw key code into token data at 0xD0230E.

Session 181 found `ConvKeyToTok` at 0x05C52C (referenced in phase133/136 reports). Session 131 found key-to-token dispatch at 0x045C37 (cascading CP/JP pattern for digit keys 0x8E-0x97).

## Task

Build a probe `probe-phase185-token-pipeline.mjs` that:

### Part 1: Static disassembly of 0x058460-0x0584A3
Disassemble the bytes between the start of the key processing section (0x058460, approximately) and 0x0584A3 where the copy helper is called. This should reveal what writes to 0xD0230E.

### Part 2: ConvKeyToTok (0x05C52C) investigation
- Disassemble 0x05C52C forward (at least 30 instructions)
- Check if it writes to 0xD0230E
- Test: call ConvKeyToTok with A = 0x92 (k2 key code) and check what ends up in 0xD0230E and OP1

### Part 3: Trace execution from 0x058483 with pre-seeded token
- Cold boot → kernel init → MEM_INIT (standard pipeline)
- Pre-seed 0xD0230E with a digit token: `mem[0xD0230E] = 0x00; mem[0xD0230F] = 0x32; ...` (type=0x00 real, token=0x32 = digit "2")
- Set A = 0x92 (k2 scan code) in case the handler also checks the raw key code
- Start execution at 0x058483 with step limit ~2000
- Watch for: writes to 0xD0230E, calls to BufInsert (0x05E2A0), calls to ConvKeyToTok (0x05C52C)
- Skip the LCD stall at 0x09EFDE (session 182 found it stalls there for 409 of 500 steps — use a PC intercept to skip past it, or start after it)

### Part 4: Map key-to-token for digit keys
If ConvKeyToTok works, test all 10 digit keys (k0=0x8E through k9=0x97) and record what token each produces. Compare against EVAL_TOKEN_MAP expectations: k0→0x30, k1→0x31, ..., k9→0x39.
