# Auto-Session 107 â€” P1: Register State Divergence at ParseInp Entry

## Goal
Build a probe that captures ALL eZ80 CPU registers at the moment PC hits 0x099914 (ParseInp entry) in TWO scenarios, diffs them, and identifies which register/flag causes ParseInp to skip PushErrorHandler in the trampoline path.

## Background
Session 106 proved that the trampoline failure through 0x0586E3 â†’ 0x099910 â†’ 0x099914 is caused by register/flag state, NOT by SP, OP1, or scrapMem. In the **control** (direct ParseInp after `prepareCallState`), ParseInp calls PushErrorHandler before ChkFindSym, catches the "not found" error gracefully, and parses "2+3" â†’ OP1=5.0. In the **trampoline**, ParseInp SKIPS PushErrorHandler, goes straight to ChkFindSym, and the error crashes into a clobbered error frame. Same code at 0x099914, different branch in the first 6 blocks.

`prepareCallState` sets `cpu.f = 0x40` (Z flag set, all others clear). The trampoline runs through 0x07FF81 first which does `XOR A` and various register loads â€” it leaves F with different flags.

### Critical notes
- The executor's cpu object stores registers as: `cpu.a`, `cpu.f`, `cpu.b`, `cpu.c`, `cpu.d`, `cpu.e`, `cpu.h`, `cpu.l` for 8-bit access. For 24-bit: `cpu._bc`, `cpu._de`, `cpu._hl`, `cpu._ix`, `cpu._iy`. `cpu.sp` is 24-bit.
- `cpu.f` contains eZ80 flags: bit 7=S, bit 6=Z, bit 5=Y(unused), bit 4=H, bit 3=X(unused), bit 2=PV, bit 1=N, bit 0=C
- The `onBlock(pc)` callback fires BEFORE the block at `pc` executes. Register state at that point is what ParseInp will see.
- Use `throw new Error('__CAPTURE__')` as sentinel, catch it outside.
- Budget for scenarios A and B: 10000 steps (ParseInp entry is within first 10 steps for both).
- Budget for scenario C: 50000 steps. Budget for D: 2000 steps.
- Do NOT modify any existing files. Only create the new probe file.
- Entry point for trampoline is 0x0586E3 (CALL 0x099910 in the common tail). 0x099910 does CALL 0x07FF81 then falls through to 0x099914.
