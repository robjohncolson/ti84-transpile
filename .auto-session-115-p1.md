# Auto-Session 115 — P1: Variable Storage Beyond Ans

## Task
Implement letter variable storage in the browser-shell direct-eval system (`TI-84_Plus_CE/browser-shell.html`). After this change, users can type `5→A` (stores 5 in variable A), then `A*3` (recalls A and computes 15).

## Background
The direct-eval system lives in `browser-shell.html` starting around line 1519. Key functions:
- `tokenizeExpression(expr)` — converts user text to TI-OS token bytes
- `runDirectEval(expr)` — orchestrates MEM_INIT → ParseInp pipeline
- `evalEnsureRuntime()` — cold boots once, caches runtime state for reuse
- `evalRunCreateReal(ex, c, mem)` — calls ROM's CreateReal to allocate a VAT entry

The system already supports multi-expression chaining via Ans (session 114). After each eval, it:
1. Saves OP1 (result BCD)
2. Resets OPS/FPS to post-CreateReal values (ParseInp leaves these dirty)
3. Re-runs CreateReal(Ans) to rebuild the Ans VAT entry (OPS overflow corrupts VAT during ParseInp)
4. Writes saved BCD into the Ans data slot

**Critical constraint**: After every ParseInp call, OPS overflow corrupts the VAT. Any stored variables must be re-created before the NEXT ParseInp call, or FindSym won't find them.

## Architecture for Variable Storage

### 1. JS-side variable store
Add a module-level `Map` keyed by variable letter (A-Z) storing 9-byte BCD `Uint8Array` values:
```js
const evalVarStore = new Map(); // 'A' → Uint8Array(9) BCD bytes
```

### 2. Parse STO arrow in expression
Before tokenizing, detect the `→` pattern. TI-84 syntax: `expr→VARNAME`. The `→` can be typed as:
- Unicode → (U+2192)
- ASCII `->` (convenience alias)
- The word `sto` (case-insensitive, convenience alias)

If STO is detected:
- Split into LHS (expression) and RHS (variable name, single letter A-Z or `theta`/`Ans`)
- Evaluate LHS via the normal ParseInp pipeline
- Store result: save BCD in `evalVarStore`, also write into ROM memory via CreateReal

If NO STO detected, evaluate normally (existing behavior).

### 3. Re-create all stored variables before each ParseInp
In `runDirectEval`, after resetting OPS/FPS but BEFORE writing tokens and running ParseInp:
1. Create Ans (existing behavior)
2. For each variable in `evalVarStore`, call `evalRunCreateReal` with the appropriate OP1 bytes, then write the stored BCD into the data slot at `deAddr`

The OP1 bytes for a Real variable named "A" are: `[0x00, 0x41, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]` where 0x41 is the ASCII code for 'A'. For Ans: `[0x00, 0x72, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]` (already defined as `EVAL_ANS_OP1`).

After creating all variables, update the cached pointer values (postCreateFps, postCreateFpsBase, postCreateOpBase) from the final state.

### 4. Token mapping
Add `→` to EVAL_TOKEN_MAP: `'→': 0x3C`. But we won't actually send STO expressions to ParseInp — we'll parse STO in JS and only send the LHS expression to ParseInp.

### 5. Clear button
The existing "C" button clears `evalCachedMem` etc. Also clear `evalVarStore` when C is pressed.

### 6. Display feedback
- After store: show `= 5 → A  (stored)` in green
- After recall that uses a variable: show result normally (ParseInp handles recall via ChkFindSym)
- If store target is invalid (not A-Z): show error in red

## Implementation Plan

### Modify `evalEnsureRuntime()`:
No changes needed here — it already creates Ans on first boot. Variable creation happens in `runDirectEval` before each ParseInp.

### Modify `runDirectEval(expr)`:
1. At the top, check for STO pattern: `expr.indexOf('→')`, `expr.indexOf('->')`, or `expr.match(/^(.+?)(?:→|->|sto\s+)([A-Za-z])$/i)`
2. If STO found:
   a. Extract lhsExpr and varName
   b. Call self recursively (or inline) to evaluate lhsExpr → get {value, op1Hex, ...}
   c. Save the 9-byte BCD from OP1 into evalVarStore.set(varName.toUpperCase(), bcd)
   d. Also store in Ans (existing behavior)
   e. Return `{ value, display: formatResult(value) + ' → ' + varName.toUpperCase(), stored: true, ansStored: true }`
3. If no STO:
   a. Before ParseInp, re-create all variables from evalVarStore:
      - For each (name, bcd) in evalVarStore: set OP1 to [0x00, name.charCodeAt(0), 0x00...], call evalRunCreateReal, write bcd to deAddr
      - Update cached pointers after all creates
   b. Also re-create Ans with its stored value (existing behavior, but move it to happen alongside variable recreation)
   c. Run ParseInp normally
   d. Post-ParseInp: save result, re-create Ans (existing behavior)

### Modify clear button handler:
Add `evalVarStore.clear();` to the btnEvalClear click handler.
