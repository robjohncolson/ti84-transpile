# Task: Build SP-adjusted bypass probe for common-tail → ParseInp pipeline

## Context

You are working on a TI-84 Plus CE ROM transpilation project at `C:/Users/rober/Downloads/Projects/school/ti84-transpile`. The transpiler lifts eZ80 machine code into runnable JavaScript.

Session 105 proved that entering the ENTER handler's common tail at 0x0586CE bypasses all blockers and reaches ParseInp (0x099914) in ~136 steps. However, entering via CALL at 0x0586E3 (which calls trampoline 0x099910 → ParseInp) shifts SP by 3 bytes vs the validated direct-ParseInp baseline. This 3-byte SP shift causes ParseInp's internal routines to trigger the allocator loop (compaction zeros all pointers).

**The validated baseline**: Direct ParseInp at 0x099914 with SP=0xD1A872 returns OP1=5.0, errNo=0x8D in 918 steps.

**The problem**: Entering at 0x0586CE or 0x0586E3 adds extra CALL frames, so ParseInp starts at SP=0xD1A86F (3 bytes lower). This causes failure.

## Your Task

Create a new probe file `TI-84_Plus_CE/probe-phase25au-sp-adjusted-bypass.mjs` that tests entering at 0x0586CE with the SP raised by 6 bytes to compensate for the extra call depth.

### Approach

Copy the structure from the existing probe `TI-84_Plus_CE/probe-phase25at-bypass-corrected.mjs` (read it first). The key changes:

1. **Raise mainSP by 6 bytes**: In `prepareCallState`, set `cpu.sp = STACK_RESET_TOP - 6` (was `STACK_RESET_TOP - 12`). This gives 6 extra bytes of stack space so that after the common-tail's CALLs (0x0586E3→0x099910→0x07FF81), ParseInp starts at approximately SP=0xD1A872 — matching the validated baseline.

2. **Actually, let me be more precise**: The baseline has `cpu.sp = STACK_RESET_TOP - 12 = 0xD1A872`. The return address FAKE_RET is written at cpu.sp. So ParseInp runs at SP=0xD1A872. When entering at 0x0586CE, there are 2 CALLs before ParseInp starts (CALL 0x099910 at 0x0586E3, then CALL 0x07FF81 at 0x099910, then 0x07FF81 does a CALL+RET, then falls through to ParseInp at 0x099914). The net effect on SP at ParseInp entry is -3 bytes (one extra return address pushed by the CALL at 0x0586E3; the trampoline's internal CALL/RET is balanced). So set `cpu.sp = STACK_RESET_TOP - 9` (3 bytes higher than baseline's -12) for the bypass scenario, so ParseInp starts at SP ≈ 0xD1A872 after the CALL pushes.

3. **Test THREE scenarios**:
   - **C (control)**: Direct ParseInp at 0x099914 with SP = STACK_RESET_TOP - 12 (= 0xD1A872). Expected: OP1=5.0, errNo=0x8D, ~918 steps.
   - **B (SP-adjusted trampoline)**: Entry at 0x0586E3 with SP = STACK_RESET_TOP - 9 (= 0xD1A875). The CALL 0x099910 pushes 3 bytes → ParseInp starts at SP=0xD1A872. Expected: OP1=5.0 if SP theory is correct.
   - **A (SP-adjusted bypass)**: Entry at 0x0586CE with SP = STACK_RESET_TOP - 9 (= 0xD1A875). The common tail does CALLs before 0x0586E3. We need to trace whether there are any additional CALL frames between 0x0586CE and the CALL 0x099910 at 0x0586E3. Looking at the disassembly from session 104: between 0x0586CE and 0x0586E3 there are calls to 0x0A27DD and possibly 0x082902, but these should CALL+RET (balanced). So the net SP change from 0x0586CE to ParseInp entry should also be -3. Use SP = STACK_RESET_TOP - 9.

4. **Use the exact same validated-scratch seeding** as the existing probe:
   - Tokens `[0x32, 0x70, 0x33, 0x3f]` ("2+3" with end marker) at 0xD00800
   - begPC = curPC = 0xD00800
   - endPC = 0xD00803 (AT the final token byte, not past-end)
   - OP1 cleared to zeros
   - Error frame at sp-6 with ERR_CATCH_ADDR=0x7FFFFA
   - MEM_INIT pointers: tempMem/FPSbase/FPS at 0xD1A881, OPBase/pTemp/progPtr at 0xD3FFFF

5. **Budget**: 50,000 steps per scenario (same as existing probe).

6. **Output format**: Same as the existing probe — print scenario label, milestones, termination, steps, errNo, OP1 decoded, pointer snapshot, recent PCs, and a summary table comparing all scenarios.

7. **Success criteria**: If scenario B and/or A return OP1=5.0 with errNo=0x8D matching the control, print `PASS: SP adjustment fixes the bypass path`.
