# Session 171 P1: Opcode Census of GCD Blocks

## Task
Build `TI-84_Plus_CE/probe-phase171-opcode-census.mjs` that runs gcd(12,8) and catalogs every unique eZ80 instruction type used across all ~230 blocks visited.

## Context
We're building toward a reference eZ80 interpreter to find a transpilation bug in the gcd function. First we need to know exactly which instruction types the interpreter must support.

The gcd handler starts at 0x068D3D. We set up OP1=12.0, OP2=8.0 in the FP format (9-byte BCD: sign, exponent, 7 mantissa bytes), then execute. It visits ~230 unique PCs over ~1442 steps and produces E_Domain (errNo=0x84) — which is wrong.

## Deliverables
Create `TI-84_Plus_CE/probe-phase171-opcode-census.mjs` that:

1. Uses the standard probe boilerplate (see any `probe-phase170-*.mjs` for the pattern — imports `createExecutor` from `./cpu-runtime.js`, `createPeripheralBus` from `./peripherals.js`, loads `ROM.rom` and `ROM.transpiled.js`)
2. Sets up gcd(12,8) execution context (see below)
3. Runs with `cpu.onBlock` callback, logging each block's PC
4. For each unique block PC visited, uses the ez80-decoder to decode all instructions from ROM bytes at that PC
5. Catalogs: instruction mnemonic, operand types, total count of each instruction type
6. Prints a summary table: instruction type | count of blocks using it | total instruction count

## GCD Setup Pattern (copy exactly)
```js
const peripherals = createPeripheralBus({ timerInterrupt: false });
const { cpu, mem, BLOCKS: exBlocks } = createExecutor(romBytes, BLOCKS, peripherals);

// Stack
const STACK_TOP = 0xd1a87e;
const SENTINEL = 0x7ffffe;
cpu.sp = STACK_TOP;
mem[SENTINEL] = 0x76; mem[SENTINEL+1] = 0x76; mem[SENTINEL+2] = 0x76;
cpu.sp -= 3;
mem[cpu.sp] = SENTINEL & 0xff;
mem[cpu.sp+1] = (SENTINEL >> 8) & 0xff;
mem[cpu.sp+2] = (SENTINEL >> 16) & 0xff;

// FP stack init
const FPS_ADDR = 0xd0258d;
const FPSBASE_ADDR = 0xd0258a;
const FPSBASE = 0xd1a9f7;
mem[FPS_ADDR] = FPSBASE & 0xff;
mem[FPS_ADDR+1] = (FPSBASE >> 8) & 0xff;
mem[FPS_ADDR+2] = (FPSBASE >> 16) & 0xff;
mem[FPSBASE_ADDR] = FPSBASE & 0xff;
mem[FPSBASE_ADDR+1] = (FPSBASE >> 8) & 0xff;
mem[FPSBASE_ADDR+2] = (FPSBASE >> 16) & 0xff;

// OP1 = 12.0 (BCD: sign=0x00, exp=0x81, mantissa=12 00 00 00 00 00 00)
const OP1_ADDR = 0xd005f8;
const op1 = [0x00, 0x81, 0x12, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00];
for (let i = 0; i < 9; i++) mem[OP1_ADDR + i] = op1[i];

// OP2 = 8.0 (BCD: sign=0x00, exp=0x80, mantissa=80 00 00 00 00 00 00)
const OP2_ADDR = 0xd00603;
const op2 = [0x00, 0x80, 0x80, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00];
for (let i = 0; i < 9; i++) mem[OP2_ADDR + i] = op2[i];

// Clear error
mem[0xd008df] = 0x00;

// IX for home-screen context
cpu.ix = 0xd1a860;

// Entry
cpu.pc = 0x068d3d;
```

## Block Boundary Detection
A block ends at: RET, JP (unconditional), JR (unconditional), CALL (the CALL itself is the last instruction of the block — execution continues at the call target as the next block). Also HALT. The transpiler treats each linear run of instructions ending at a control transfer as one block.
