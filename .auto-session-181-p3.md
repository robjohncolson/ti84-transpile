# Auto-Session 181 P3 — Key Classification Subroutine Disassembly (0x058D49)

## Goal
Write a Node.js probe (`TI-84_Plus_CE/probe-phase181-key-classify.mjs`) that disassembles the key classification subroutine at 0x058D49, called by the home handler at step 11 of its execution (see phase129-edit-buffer-report.md). Also disassemble the key dispatch table structure used by the home handler to route key codes to their handlers (BufInsert for digits, special actions for ENTER/STAT/GRAPH/etc).

## Context
From the phase129 report, the home handler at 0x058241:
1. Entry — loads HL=0, stores to 0xD026AC
2. Checks various IY flags
3. Calls 0x058BA3 (key code lookup?)
4. Calls 0x058D49 (key classification) at step 11
5. Then calls 0x08BF22 (CoorMon event loop) which stalls

The gap: we don't know what 0x058D49 does with the key code. It may be the function that classifies digit keys vs. function keys vs. menu keys and routes them. Understanding this is critical for priority #1 (wiring the keyboard path).

Also: session 129 found the home handler runs a "key dispatch table scan loop" at 0x0825D1/0x0825D9/0x0825DB (27 times with different targets). This is likely iterating a key-code → handler-address table. Dump this table.

## Deliverables
Write `TI-84_Plus_CE/probe-phase181-key-classify.mjs` that:

### Part 1: Disassemble 0x058D49
- Decode instructions starting at 0x058D49 in ADL mode
- Continue for ~150 instructions or until RET at call depth 0
- Print address, hex bytes, mnemonic for each instruction
- Look for: comparisons against key code ranges (CP instructions), jump tables (JP (HL)), calls to BufInsert (0x05E2A0), references to kbdKey (0xD0058C)

### Part 2: Disassemble 0x058BA3
- Same treatment — this is the "key code lookup" called just before 0x058D49
- ~100 instructions

### Part 3: Dump key dispatch table
- From the phase129 report, the key dispatch table scan loop visits 0x0824FD through 0x0825xx
- Disassemble the loop at 0x0825D1 and its surrounding context (0x0825C0 to 0x0825F0)
- If there's a table pointer in a register (HL typically), dump 256 bytes from that address
- Look for patterns: pairs of (key_code, handler_address) or indexed arrays

### Part 4: Trace 0x058D49 with k2 (0x90)
After boot + MEM_INIT:
- Seed kbdKey (0xD0058C) = 0x90 (k2, the digit '2')
- Set CPU state: `madl=1, mbase=0xD0, iy=0xD00080, ix=0xD1A860, f=0x40`
- SP = 0xD1A87E - 12, push FAKE_RET = 0x7FFFFE
- Set A = 0x90 (since the handler may pass the key code in A)
- Run from 0x058D49, maxSteps 50000, maxLoopIterations 8192
- Log every unique PC visited (first 200)
- Check if BufInsert (0x05E2A0) is ever reached
- After return, check if editCursor (0xD0243A) changed
