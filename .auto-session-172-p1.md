# Task: Audit cpu-runtime.js FP-critical methods against eZ80 specification

## Context
We are transpiling the TI-84 Plus CE ROM (eZ80 CPU) into JavaScript. The transpiled gcd(12,8) function produces E_Domain error instead of the correct result (4). Session 171 proved via block-by-block comparison that 1421/1423 transpiled blocks match a mini reference interpreter — but that interpreter uses the SAME cpu-runtime.js methods. So if a cpu-runtime.js method is buggy, both sides agree on the wrong answer.

**The bug is almost certainly in a cpu-runtime.js method, not in the transpiled blocks.**

## Your task
Build a probe file `TI-84_Plus_CE/probe-phase172-cpu-method-audit.mjs` that tests every FP-critical cpu-runtime.js method against known eZ80 reference values. For each method, construct test cases with known inputs and expected outputs per the eZ80 CPU User Manual, then call the method and compare.

## Methods to audit (with line numbers in cpu-runtime.js)

1. **`rld()`** (line 582): BCD rotate left digit. Takes A register and (HL) byte. High nibble of (HL) goes to low nibble of A. Low nibble of A goes to low nibble of (HL). High nibble of (HL) becomes low nibble of (HL). Flags: S,Z from A result, H=0, PV=parity(A), N=0, C preserved.
   - Test: A=0x7A, (HL)=0x31 → A=0x73, (HL)=0x1A. F: S=0, Z=0, H=0, PV=parity(0x73), N=0.

2. **`rrd()`** (line 593): BCD rotate right digit. Low nibble of (HL) goes to low nibble of A. High nibble of (HL) stays. Low nibble of A goes to high nibble of (HL).
   - Test: A=0x84, (HL)=0x20 → A=0x80, (HL)=0x42. Flags from A=0x80.

3. **`add8(a, b)`** (line 206): 8-bit add. Result=a+b. S,Z from result. H from bit 4 carry. PV=overflow. N=0. C from bit 8.
   - Tests: (0x44, 0x11)→0x55, (0x80, 0x80)→0x00 with overflow+carry, (0x0F, 0x01)→0x10 with H.

4. **`subtract8(a, b)`** (line 216): 8-bit subtract. Result=a-b. S,Z from result. H from bit 4 borrow. PV=overflow. N=1. C if borrow.
   - Tests: (0x44, 0x11)→0x33, (0x00, 0x01)→0xFF with C, (0x80, 0x01)→0x7F with overflow.

5. **`decimalAdjustAccumulator(a)`** (line 256): DAA. Adjusts BCD. Uses N flag to know if last op was add or subtract. H flag for half-carry. C flag for full carry.
   - Tests after ADD: 0x9A → 0x00 (C set), 0x0A → 0x10. After SUB: with H set, 0xFA → 0xF4.
   - **IMPORTANT**: DAA is heavily used in the gcd path (17 times per opcode census). Test with realistic gcd-path scenarios.

6. **`subtractWithBorrowWord(a, b)`** (line 356): SBC HL,rr. 16-bit subtract with borrow. S,Z from 16-bit result. H from bit 12. PV=overflow. N=1. C if borrow.
   - eZ80 quirk: In ADL mode the result is masked to addressMask (24-bit), but flags are computed from the 16-bit result `r16 = result & 0xffff`. **Check if this is correct for ADL mode** — eZ80 SBC HL in ADL mode operates on 24-bit values and flags should be set from the 24-bit result, not 16-bit.

7. **`addWithCarryWord(a, b)`** (line 343): ADC HL,rr. Same ADL mode concern as above — flags computed from 16-bit `r16` but result returned as `addressMask`-masked. **In ADL mode, should use 24-bit flag computation.**

8. **`addWord(a, b)`** (line 335): ADD HL,rr. Only sets H,N,C. C check uses `addressMask`. **Verify H flag computation is correct for 24-bit ADD in ADL mode** — should check bit 20 carry, not bit 12.

9. **`testBit(value, bit)`** (line 484): BIT b,r. Z=!(value & (1<<bit)). PV mirrors Z. S set only if bit 7. H=1. N=0.
   - Test: BIT 0, 0x01 → Z=0, PV=0. BIT 7, 0x80 → Z=0, S=1. BIT 3, 0x00 → Z=1, PV=1, S=0.

10. **`rotateShift8(op, value)`** (line 423): All rotate/shift variants. SRL used in gcd path.
    - Test SRL: 0x83 → 0x41 with C=1. 0x02 → 0x01 with C=0.

11. **`_szFlags(result)`** (line 197): Sets S,Z,X,Y from result byte.
    - Verify X (bit 3) and Y (bit 5) are set from the result byte correctly.

12. **`parity(value)`** (line 13): Even parity — returns true if even number of 1-bits.
    - Tests: 0x00→true (0 bits), 0x01→false (1 bit), 0x03→true (2 bits), 0xFF→true (8 bits).

## CRITICAL: ADL mode flag computation
The eZ80 in ADL mode uses 24-bit registers. For 16-bit arithmetic instructions (SBC HL, ADC HL, ADD HL), the flags should be computed from the FULL register width, not just the low 16 bits. The current code computes some flags from `r16 = result & 0xffff` — this may be wrong. The eZ80 CPU User Manual says:
- In ADL mode, ADD/ADC/SBC HL operate on the full 24-bit register values
- Carry is from bit 23 (not bit 15)
- Half-carry is from bit 19 (not bit 11) for 24-bit ops... actually check this — it might still be bit 11

**Research the exact ADL-mode flag behavior for these instructions.**
