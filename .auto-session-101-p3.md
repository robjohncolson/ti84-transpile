# Auto-Session 101 — P3: Investigate 0x07FF81 trampoline pre-setup purpose

## Context

This is a TI-84 Plus CE ROM transpilation project. The ROM is at `TI-84_Plus_CE/ROM.rom` (4 MB, eZ80), the transpiler is `scripts/transpile-ti84-rom.mjs`, and the runtime is `TI-84_Plus_CE/cpu-runtime.js`.

**Background**: The ParseInp trampoline at 0x099910 calls 0x07FF81 before falling through to ParseInp at 0x099914. Session 100 confirmed the trampoline is functionally identical to direct ParseInp for expression evaluation (both produce OP1=5.0). But 0x07FF81 does interesting pre-setup:
- Sets OP1[0] = 0x05 (List type byte)
- Sets OP1[1..3] = 0x000023
- Stores 0x23 at RAM 0xD02AD7 via helper 0x04C940

The question is: WHY does the ENTER handler call ParseInp through this trampoline? What is the List-type pre-setup for?

## Your task

### Part A: Static disassembly of 0x07FF81

Create a probe at `TI-84_Plus_CE/probe-phase25ap-trampoline-preseed.mjs` that:
1. Reads and disassembles ROM bytes at 0x07FF81 through to the RET or fall-through to 0x099914
2. Identifies ALL RAM addresses written by 0x07FF81
3. Disassembles the helper at 0x04C940 (stores HL to 0xD02AD7)
4. Check what 0xD02AD7 is in ti84pceg.inc (if the file exists at `TI-84_Plus_CE/references/ti84pceg.inc`)

### Part B: Cross-reference 0xD02AD7

In the same probe:
1. Search the ROM for all references to 0xD02AD7 (reads and writes) — scan for bytes `D7 2A D0` in the ROM
2. Count how many locations read vs write this address
3. Identify the purpose: is it a variable index? A list pointer? An equation number?

### Part C: Cross-reference callers of 0x07FF81

Search the ROM for CALL 0x07FF81 (bytes `CD 81 FF 07`) and JP 0x07FF81 (bytes `C3 81 FF 07`):
1. List all callers
2. Note whether 0x099910 is the only caller or if others exist
3. If multiple callers, what context do they call from?

### Part D: Analysis

Write a report at `TI-84_Plus_CE/phase25ap-trampoline-preseed-report.md` answering:
- What is 0xD02AD7? (Name from ti84pceg.inc if available)
- Why does the ENTER path pre-seed OP1 with List type?
- Is this pre-setup relevant for non-expression inputs (list recall, program execution)?
- Does ParseInp always overwrite this pre-setup, or are there paths where it matters?
