# Task: Disassemble 0x08A98F helper called by MEM_INIT

## Context
We are reverse-engineering the TI-84 Plus CE OS ROM. The MEM_INIT routine at 0x09DEE0 initializes heap/VAT pointers and then calls `CALL 0x08A98F` before returning. We fully disassembled MEM_INIT in session 82 but NOT this helper. ParseInp (the expression parser at 0x099914) still fails with errNo=0x8E (ErrMemory) even when all MEM_INIT pointer values are correctly seeded. We suspect 0x08A98F does critical additional setup.

## What to do
1. Read `TI-84_Plus_CE/ROM.rom` (binary, 4MB) starting at offset 0x08A98F
2. Decode eZ80 instructions starting at 0x08A98F until you hit a RET (0xC9) or unconditional JP that leaves the function
3. For each instruction, note: address, hex bytes, mnemonic, operands
4. If there are CALL instructions, note the target but do NOT recursively disassemble — just record the address
5. Identify what RAM addresses are read/written (LD (nn), r or LD r, (nn) patterns)
6. Cross-reference any RAM addresses against known equates from `TI-84_Plus_CE/references/ti84pceg.inc` (search for the address in hex)

## Decoder reference
- The ROM is eZ80 (Z80 superset with 24-bit addressing). Use `TI-84_Plus_CE/ez80-decoder.js` as reference for opcode encoding.
- Common patterns: `21 xx yy zz` = LD HL, imm24. `22 xx yy` = LD (nn), HL (16-bit addr in Z80 mode). `CD xx yy zz` = CALL imm24. `C9` = RET. `ED 53 xx yy zz` = LD (nn), DE (24-bit addr in ADL mode).
- The eZ80 has SIS/LIS/SIL/LIL prefix bytes (0x40, 0x49, 0x52, 0x5B in specific contexts) — check the decoder if you see unexpected opcodes.
