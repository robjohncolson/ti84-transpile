# Auto-Session 112 P2: Investigate Approach B RclVarSym failure (higher OPBase)

## Goal
Build a diagnostic probe to determine WHY RclVarSym returns garbage `[00 00 72 d1 a8 81 00 00 00]` when OPBase is set to 0xD40040 (Approach B from session 111). Two hypotheses to test:

1. **errNo=0x8D leftover from ParseInp** causes RclVarSym to take a different code path
2. **progPtr/pTemp don't bracket the VAT entry** — RclVarSym's FindSym scans from progPtr downward and may not find the entry at ~0xD40037

## What to build

Create `TI-84_Plus_CE/probe-phase25ba-approach-b-debug.mjs` that runs TWO scenarios:

### Scenario 1: Approach B with errNo cleared before RclVarSym
- Same as session 111 Approach B: MEM_INIT → override OPBase/OPS=0xD40040, pTemp/progPtr=0xD40060 → CreateReal("Ans") → ParseInp("2+3") → **clear errNo to 0x00** → RclVarSym
- If OP1=5.0 after this, errNo was the root cause

### Scenario 2: Approach B with progPtr/pTemp dump before RclVarSym
- Same pipeline but before RclVarSym, dump:
  - The exact VAT entry location (read OPBase after CreateReal — the entry is between the new OPBase and the original OPBase)
  - progPtr, pTemp, OPBase values at the moment of RclVarSym call
  - The 32 bytes around the VAT entry area to see the entry structure
  - Whether progPtr > VAT entry address > pTemp (the expected bracket)
- Also try setting progPtr to a value ABOVE the VAT entry (e.g., 0xD40060 or higher) and pTemp to a value BELOW it

## Reference implementation

Use `probe-phase25az-rclvar-ops-fix.mjs` as the template. The key functions are:
- `createRuntime()` — creates fresh 16MB memory + executor with `timerInterrupt: false`
- `coldBoot()` — z80 boot + kernel init + post-init
- `runMemInit()` — calls MEM_INIT at 0x09DEE0
- `runCreateReal()` — calls CreateReal at 0x08238A with OP1=[00 72 00 00 00 00 00 00 00] (Ans), A=0x00, HL=0x000009
- `runParseInp()` — calls ParseInp at 0x099914 with tokens [0x32, 0x70, 0x33, 0x3F] ("2+3\n")
- `runRclVarSym()` — calls RclVarSym at 0x09AC77 with OP1=[00 72 00 00 00 00 00 00 00]

All use `seedErrFrame()` for error handling and the `runCall()` harness for execution.
