# Auto-Session 108-P2: Trampoline + fixed error frame + PushErrorHandler pre-call

## Context

You're working on a TI-84 Plus CE ROM transpiler. ParseInp (0x099914) succeeds when called directly with OP1=zeros (918 steps, OP1=5.0 parsing "2+3") but FAILS via the trampoline at 0x0586E3 because 0x07FF81 pre-seeds OP1 with a List type [0x05, 0x00, 0x23, ...], causing ChkFindSym inside 0x099B18 to fail (carry set → jp c 0x061D3A → JError).

Session 107 proved error frame at 0xD1A850 survives clobbering. The trampoline error is properly caught there, but ParseInp still returns OP1=0 because it throws to our OUTER catch instead of handling errors INTERNALLY via PushErrorHandler.

**In the real OS**, callers of ParseInp call PushErrorHandler BEFORE ParseInp. For example, at 0x097406: `call 0x061DEF` (PushErrorHandler with HL=0x09746E as catch address). When ChkFindSym fails inside ParseInp, JError longjmps to the PushErrorHandler frame, which restores SP and jumps to the catch address. The catch code then continues parsing.

## Task

Build a probe `TI-84_Plus_CE/probe-phase25aw-trampoline-with-peh.mjs` that tests 3 scenarios:

### Scenario A: Trampoline with outer catch only (baseline — reproduces OP1=0)
1. Cold boot + MEM_INIT
2. Seed tokens "2+3" at 0xD00800, seed begPC/curPC/endPC
3. Set up FAKE_RET on stack, error frame at 0xD1A850 with catch = ERR_CATCH_ADDR (0x7FFFFA)
4. Enter at trampoline 0x0586E3 (which calls 0x099910 → 0x07FF81 → ParseInp)
5. Expected: ERR_CATCH hit, OP1=0, errNo=0x8D

### Scenario B: Direct ParseInp with OP1 cleared to zeros (control — expect OP1=5.0)
1. Same cold boot + MEM_INIT + token seeding
2. Clear OP1 to all zeros
3. Set up FAKE_RET on stack, error frame at 0xD1A850 with catch = ERR_CATCH_ADDR
4. Enter at 0x099914 directly
5. Expected: FAKE_RET hit, OP1=5.0, 918 steps

### Scenario C: Trampoline + PushErrorHandler with catch at 0x099929
This is the KEY experiment. We manually set up what OS callers do:

1. Same cold boot + MEM_INIT + token seeding
2. Set up the OUTER error frame at 0xD1A850 with catch = ERR_CATCH_ADDR (safety net)
3. **ALSO** set up an INNER PushErrorHandler-style frame:
   - Push a 6-byte error frame (simplified) below the current SP
   - The catch address in this inner frame = 0x099929 (the POP BC instruction right after `call 0x099B18`)
   - Set errSP to point to this inner frame
   - The idea: when ChkFindSym fails → JError → longjmp restores SP from errSP → jumps to 0x099929
   - ParseInp then continues from POP BC as if 0x099B18 returned normally
4. Enter at trampoline 0x0586E3
5. **If this works**: OP1=5.0, ParseInp parsed "2+3" correctly despite ChkFindSym failure
6. **If OP1 still 0**: the catch at 0x099929 isn't sufficient — JError does more than just jump to catch address

### Scenario D: Direct ParseInp with OP1=[0x05, 0x00, 0x23, ...] (List pre-seed, no trampoline)
Confirms that OP1 content is the sole cause of divergence:

1. Same cold boot + MEM_INIT + token seeding
2. Set OP1 = [0x05, 0x00, 0x23, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00] (what 0x07FF81 writes)
3. Error frame at 0xD1A850 with catch = ERR_CATCH_ADDR
4. Enter at 0x099914 directly (NOT trampoline)
5. Expected: should FAIL just like trampoline (confirming OP1 content is the divergence, not the call path)

```javascript
function runMemInit(executor, cpu, mem) {
  cpu.halted = false; cpu.iff1 = 0; cpu.iff2 = 0;
  cpu.madl = 1; cpu.mbase = 0xD0; cpu._iy = 0xD00080;
  cpu.f = 0x40; cpu._ix = 0xD1A860;

  mem.fill(0xff, cpu.sp, cpu.sp + 12);
  cpu.sp -= 3;

  try {
    executor.runFrom(0x09DEE0, 'adl', {
      maxSteps: 100000, maxLoopIterations: 8192,
      onBlock(pc) { if ((pc & 0xFFFFFF) === 0x7FFFF6) throw new Error('__RET__'); },
      onMissingBlock(pc) { if ((pc & 0xFFFFFF) === 0x7FFFF6) throw new Error('__RET__'); },
    });
  } catch (e) { if (e.message !== '__RET__') throw e; }
}
```

### Error frame setup for Scenario C
For the inner PushErrorHandler frame, the key is:
- errSP (0xD008E0) points to a 6-byte frame
- Frame layout: [3 bytes SP-restore value] [3 bytes catch-address]
- The catch address = 0x099929

Actually, JError (0x061DB2) does: load SP from errSP frame, pop registers, jump to catch address. The exact frame format matters. From phase25ab report: JError flow is `0x061D3A → 0x061DB2 → 0x061DD1 → catch_address`. The 0x061DD1 path restores from the PushErrorHandler frame.

A simplified approach: just set errSP to point to a frame where bytes [3..5] contain the catch address (0x099929 in LE). This is what all existing probes do:
```javascript
write24(mem, errFrameBase + 3, 0x099929);
write24(mem, ERR_SP_ADDR, errFrameBase);
```

For Scenario C, layer two frames:
1. Outer frame at 0xD1A850: catch = ERR_CATCH_ADDR (0x7FFFFA) — safety net
2. Inner frame at (SP - 6): catch = 0x099929 — actual ParseInp error catch
3. errSP = inner frame address
