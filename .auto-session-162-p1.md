# Auto-Session 162 P1 — Dense trace gcd steps 180-450 (algorithm body between iterations)

## Context
You are working on a TI-84 Plus CE ROM transpilation project. The transpiler lifts eZ80 ROM to JavaScript. We are debugging the gcd(12,8) function which should return 4 but instead produces -100 and E_Domain error.

**KEY FINDINGS FROM PRIOR SESSIONS (161)**:
- The normalization at 0x07CA48 is NOT the root cause (bypassing it produces identical results)
- After normalization+FPAdd, OP1 is RESTORED to 1200 at step 182 via OP2→OP1 copy
- The gcd algorithm body at 0x068D82-0x068DEA is where the bug lives
- Session 155 found: FPDiv(1200,1200) = 0 in gcd context but = 1.0 when called directly
- The gcd handler scales 12→1200 (exponent bump 0x81→0x83) for integer extraction

## Task
Build a probe that does a **dense trace of every block executed during gcd(12,8) from step 180 to step 450**. This covers the algorithm body between the first OP1toOP2 call returning (OP1 restored to 1200) and what happens before the second iteration. At each step log:
- Step number
- PC (block address)  
- OP1 (9 bytes, decoded BCD value)
- OP2 (9 bytes, decoded BCD value)
- OP5 at 0xD00624 (9 bytes — the backup register)
- A register
- FPS pointer (3 bytes at 0xD0258D)
- SP register

**KEY QUESTIONS TO ANSWER**: 
1. Where exactly does FPDiv get called? What PC address transitions into FPDiv?
2. What are OP1 and OP2 at the exact moment FPDiv is entered?
3. After FPDiv returns, what is the result in OP1?
4. Is FPTrunc (integer part extraction) called? Where?
5. How does the Euclidean mod operation work? (mod = a - INT(a/b)*b)
