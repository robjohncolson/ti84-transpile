# Auto-Session 108-P1: Memory state divergence probe at 0x099B18

## Context

You're working on a TI-84 Plus CE ROM transpiler project. The codebase lifts eZ80 machine code to JavaScript. We're investigating why ParseInp (expression parser at 0x099914) FAILS when entered via the trampoline path (0x0586E3 → 0x099910 → 0x07FF81 → 0x099914) but SUCCEEDS when called directly.

### The key subroutine: 0x099B18

ParseInp calls 0x099B18 at address 0x099925. This subroutine:
1. Calls ChkFindSym (0x08383D) which looks up the variable named in OP1 (0xD005F8)
2. If carry set (not found): `jp c, 0x061D3A` → JError → longjmp to errSP
3. If found: sets begPC/curPC to the variable's data area

### The divergence

- **Control path** (OP1=zeros): ParseInp runs 918 steps, OP1=5.0 (parses "2+3"). ChkFindSym likely returns carry=0 for null OP1 (special handling or found something).
- **Trampoline path** (OP1=[0x05, 0x00, 0x23, ...]): 0x07FF81 pre-seeds OP1 with a List type. ChkFindSym then looks for this List variable in the VAT, fails (empty VAT after MEM_INIT), carry=1, jp c to error path, JError fires.

Session 107 proved this is NOT register-gated (F, BC, DE, IX, IY, SP identical or irrelevant). Must be memory-gated — most likely OP1 content affecting ChkFindSym's result.

## Task

Build a probe `TI-84_Plus_CE/probe-phase25aw-memory-divergence.mjs` that:

### Scenario A: Direct ParseInp with OP1=zeros (control)
1. Cold boot + MEM_INIT
2. Seed tokens "2+3" at SCRATCH_TOKEN_BASE (0xD00800), seed begPC/curPC/endPC
3. Set OP1 to all zeros
4. Breakpoint at 0x099B1C (the jp c instruction AFTER ChkFindSym returns)
5. Capture: carry flag (F register bit 0), A, HL, DE, BC registers, and OP1 bytes
6. If carry=0, let ParseInp continue to completion and report final OP1

### Scenario B: Direct ParseInp with OP1=[0x05, 0x00, 0x23, ...] (List pre-seed)
1. Same cold boot + MEM_INIT + token seeding
2. Set OP1 to [0x05, 0x00, 0x23, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00] (what 0x07FF81 sets)
3. Same breakpoint at 0x099B1C
4. Capture: carry flag, A, HL, DE, BC, OP1 bytes
5. Report whether carry is set (confirming the jp c divergence)

### Scenario C: Direct ParseInp with OP1=[0x05, ...] BUT with PushErrorHandler pre-setup
1. Same setup as B
2. Before calling ParseInp, manually set up a PushErrorHandler frame:
   - errSP points to a frame with catch address = 0x099929 (the POP BC after `call 0x099B18`)

3. Run ParseInp. If the error at 0x061D3A is caught and execution resumes at 0x099929, report the final OP1 and step count.

### Error frame structure (for PushErrorHandler simulation)

PushErrorHandler at 0x061DEF pushes an 18-byte frame on the stack:
- AF, HL (the catch address), DE, BC, IX, IY
Each is a 3-byte ADL push. The key is that HL = catch address (where execution resumes on error). For our purposes, a simplified version: write catch address into errSP frame at errFrameBase+3 (3-byte LE address).

For Scenario C, set up:
