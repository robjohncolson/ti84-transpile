# Auto-Session 153 P2: Direct FPDiv test — does FPDiv(12.0/8.0) return 1.5?

## Goal
Build a probe `TI-84_Plus_CE/probe-phase153-fpdiv-direct.mjs` that directly calls FPDiv with OP1=12.0 and OP2=8.0, and reports the result. This tells us whether the FP division hardware is correct independently of the gcd algorithm.

## Context
- The gcd algorithm computes `a mod b = a - INT(a/b) * b`. Session 152 found the first mod result is 0 instead of 4 for gcd(12,8).
- If FPDiv(12/8) = 1.5 correctly, then the bug is in INT() or the mod assembly, not in division itself.
- Also test: FPDiv(8/4) = 2.0, FPDiv(12/1) = 12.0 (to check if OP2=1.0 is the division argument)

## Technical details
- Use the same boilerplate as `probe-phase152-gcd-fpsub-verify.mjs`.
- Run MEM_INIT first (entry 0x09DEE0, returns to FAKE_RET=0x7FFFFE).
- FPDiv JT slot is at ROM address 0x020278. Read the JP target from ROM bytes at 0x020278: it should be `C3 xx yy zz` → the target is the 3-byte LE address after C3. That's the FPDiv implementation address.
  - Alternative: the JT table has FPDiv. From the jump table cross-reference (session 69), look for FPDiv. It should be near FPMult (0x07C785) and FPAdd (0x07C77F). Try 0x07C763 as a candidate (from priority 1 notes).
  - SAFEST: Read ROM bytes at jump table offset for `_FPDiv`. The jump table starts at 0x020104, each entry is 4 bytes (C3 + 3-byte address). FPDiv is os_FPDiv. From ti84pceg.inc, FPDiv = `$020284`. So read 3 bytes at ROM[0x020285..0x020287] for the LE target address.
  
- For each test case:
  1. Reset OP1/OP2
  2. Set OP1 = dividend BCD, OP2 = divisor BCD
  3. Push FAKE_RET onto stack
  4. Call FPDiv implementation address
  5. Run up to 500 steps
  6. Read OP1 result and decode BCD to float
  7. Print PASS/FAIL

- BCD encodings:
  - 12.0 = [0x00, 0x81, 0x12, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]
  - 8.0 = [0x00, 0x80, 0x80, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]
  - 4.0 = [0x00, 0x80, 0x40, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]
  - 1.0 = [0x00, 0x80, 0x10, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]
  - Expected 1.5 = [0x00, 0x80, 0x15, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]
  - Expected 2.0 = [0x00, 0x80, 0x20, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]
  
- BCD format: byte[0] = sign/type (0x00=positive real), byte[1] = exponent (0x80 = 10^0, 0x81 = 10^1), bytes[2-8] = BCD digits (2 per byte)
