# Phase 25G Batch 1 — Seed 0x00B608 + Retranspile + Event Loop Test

## Context

The TI-84 Plus CE OS event loop at 0x0019BE runs through a dispatch loop. After the ISR handles a keyboard IRQ and exits via the handler at 0x03D184, the next ISR cycle takes the no-keyboard path, which hits **missing block 0x00B608**. This blocks all event loop progress. The block must be seeded in the transpiler so the event loop can cycle.

Additionally, the scan code translation table at 0x09F79B (228 bytes) maps raw scan codes to key identifiers across 4 modifier states × ~56 keys. We want to dump and decode this table.

## Task 1: Seed 0x00B608 in the transpiler

**File**: `scripts/transpile-ti84-rom.mjs`

Find the seed array (it's a large array of hex addresses used as entry points for the transpiler's CFG discovery). Add `0x00B608` to the array. Place it near other Phase 25-era seeds if there's a commented section, or at the end of the array.

Then retranspile:
```bash
node scripts/transpile-ti84-rom.mjs
```

Verify the new block exists:
```bash
node -e "
const fs = require('fs');
const code = fs.readFileSync('TI-84_Plus_CE/ROM.transpiled.js', 'utf8');
console.log('Block 0x00B608 exists:', code.includes('case 0x00b608:') || code.includes('case 0xb608:'));
"
```

## Task 2: Event loop cycling probe

Create `TI-84_Plus_CE/probe-phase25g-eventloop.mjs` that:

1. Loads the transpiled ROM and creates CPU + peripherals with `timerInterrupt: false`
2. Seeds OS state the same way the golden regression does:
   - Sets IX = 0xD1A860
   - Sets IY = 0xD00080
   - Sets callback pointer: `mem.write24(0xD02AD7, 0x0019BE)`
   - Sets post-init flag: `mem.write8(0xD177BA, 1)`
   - Sets system flag bit 6 at (IY+27): `mem.write8(0xD0009B, mem.read8(0xD0009B) | 0x40)`
   - Sets SP to 0xD1A87E and seeds stack: `cpu.sp = 0xD1A87E - 12; mem.fill(0xFF, cpu.sp, cpu.sp + 12);` (note: correct `start + length` form!)
3. Runs from 0x0019BE for up to 50,000 cycles
4. Logs every block visited (the `onBlock` or step callback)
5. Reports whether 0x00B608 was reached and what happened after it
6. Reports the final PC and any missing blocks encountered

**Import pattern** — look at `probe-phase99d-home-verify.mjs` for the import style. It uses:
```javascript
import { createCPU } from './cpu-runtime.js';
import { createPeripheralBus } from './peripherals.js';
```

## Task 3: Scan code translation table dump

Create `TI-84_Plus_CE/probe-phase25g-scancode-table.mjs` that:

1. Loads the ROM binary directly: `fs.readFileSync('TI-84_Plus_CE/ROM.rom')`
2. Reads 228 bytes starting at offset 0x09F79B from the ROM
3. Interprets the table as 4 modifier groups × 57 entries (228 / 4 = 57):
   - Group 0: no modifier
   - Group 1: 2nd modifier
   - Group 2: Alpha modifier  
   - Group 3: 2nd+Alpha modifier
4. For each entry, maps the index to the physical key using the SDK-authoritative mapping from `TI-84_Plus_CE/keyboard-matrix.md`:
   - Scan code = (group_index << 4) | bit_index, where group_index is 0-6 and bit_index is 0-7
   - The scan code for key K is `(idx << 4) | bit` per the keyboard matrix
5. Prints a formatted table showing: scan_code_hex → [no_mod, 2nd, alpha, 2nd_alpha]
6. Saves the output to `TI-84_Plus_CE/phase25g-scancode-table-report.md`

**Reference**: The keyboard matrix file at `TI-84_Plus_CE/keyboard-matrix.md` has the physical key layout with scan codes.
