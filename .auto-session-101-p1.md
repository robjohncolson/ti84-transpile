# Auto-Session 101 — P1: Trace 0x0827xx allocator loop after empty-ENTER return

## Context

This is a TI-84 Plus CE ROM transpilation project. We lift eZ80 machine code to JavaScript and run it in a custom executor. The ROM is at `TI-84_Plus_CE/ROM.rom` (4 MB), the transpiler is `scripts/transpile-ti84-rom.mjs`, and the runtime is `TI-84_Plus_CE/cpu-runtime.js`.

**The problem**: When the ENTER key handler runs on a fresh boot (no expression typed), it takes the Z path → JP 0x058C65 (empty-ENTER handler) → RET at 0x058C82. But after this RET, the CALLER enters a 0x0827xx allocator loop that runs 272K+ iterations and exhausts the step budget. We need to understand:
1. What address does RET at 0x058C82 return TO? (What's on the return stack?)
2. What code runs in the 10-20 instructions after that RET before the allocator loop starts?
3. What condition does the allocator loop check, and what would let it terminate?

## What we know

- The ENTER handler entry is at 0x0585E9 (second-pass handler, dispatched by CoorMon)
- When A==0x05 (kEnter), it calls setup functions then reaches 0x058621 (PUSH AF) → 0x05862B (CALL 0x0921CB, history manager) → 0x05862F (POP AF) → 0x058630 (JR NZ, 0x05866A)
- After MEM_INIT, bit 5 of IY+0x44 is 0 → Z flag set → JR NZ NOT taken → Z path
- Z path: LD A,(0xD01D0B) → OR A → JP Z,0x058C65 (numLastEntries=0 on fresh boot)
- 0x058C65 is the "empty ENTER" handler — self-contained, 155 bytes, 10 RET instructions, ZERO jumps to 0x058693 (the common tail with ParseInp)
- After the empty-ENTER handler RETs at 0x058C82, the 0x0827xx allocator loop starts (272,482 hits in 500K steps, 251 unique PCs)
- The allocator loop includes PCs like 0x082754 and is in the InsertMem region

## Your task

Create a probe at `TI-84_Plus_CE/probe-phase25ap-allocator-after-enter.mjs` that:

1. Sets up the full environment: MEM_INIT, cx seed (cxMain=0x058241, cxCurApp=0x40 at 0xD007E0), error frame, tokenized "2+3" at userMem
2. Runs the ENTER handler at 0x0585E9 with A=5 (kEnter) 
3. Uses a step-by-step trace (record every PC for the first ~200 steps) to capture:
   - The exact PC sequence from entry through the empty-ENTER handler
   - The RET at 0x058C82 and the address it returns to
   - The next 20-30 instructions after that RET
   - Where the 0x0827xx allocator loop is entered from
4. Also dump SP and the top 6 bytes of the stack at key moments (before the CALL 0x0921CB, at the RET from empty-ENTER, and after entering the allocator loop)

**Output format**: The probe should print:
- The full PC trace for the first 200 steps
- "RET TARGET: 0xXXXXXX" when the empty-ENTER handler returns
- Stack dumps at key moments
- The first 10 unique PCs after the RET target (these are the instructions leading into the allocator loop)
