# Auto-Session 155, Priority 2: Investigate JmpThru at 0x080188

## Context

You are working on a TI-84 Plus CE ROM transpilation project. The gcd(12,8) handler's algorithm body (0x068D82+) calls `JmpThru` twice:
- At 0x068DC7: `CALL 0x080188` (JmpThru indirect call #1)
- At 0x068DDB: `CALL 0x080188` (JmpThru indirect call #2)

JmpThru (0x080188) is an indirect jump/call mechanism in the TI-OS. It reads a target address from the code stream or from a register and jumps there. If the target is not transpiled, the gcd algorithm will produce wrong results.

## Task

1. Read the ROM bytes at 0x080188-0x0801C0 (56 bytes) and decode them as eZ80 instructions
2. Check if block 0x080188 exists in PRELIFTED_BLOCKS
3. Determine what JmpThru does: does it read a 3-byte target address from (HL)? From the return address on stack? From a register pair?

### Part B: Trace JmpThru calls during gcd(12,8)
1. Set up gcd(12,8) exactly as in probe-phase154-gcd-op2-restore.mjs
2. Run the gcd handler and intercept execution at PC=0x080188
3. At each hit of 0x080188, log:
   - Step number
   - Register HL, DE, BC values
   - Stack top 6 bytes (the return address and one more)
   - OP1 and OP2 (first 4 bytes each)
   - Where JmpThru ACTUALLY jumps to (the PC at step+1)
4. Also log the PC at step+1 after each JmpThru hit to see where execution continues

### Part C: Verify targets are transpiled
For each unique target address that JmpThru jumps to:
1. Check if that address exists as a key in PRELIFTED_BLOCKS
2. If not, report it as a MISSING BLOCK — this could be the root cause of the gcd failure
