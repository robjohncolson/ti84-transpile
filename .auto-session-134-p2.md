# Auto-Session 134 P2: Disassemble FP Category Byte Clearing Code

## Context
This is a TI-84 Plus CE ROM transpilation project. During gcd(12,8) evaluation via ParseInp, the FP category byte at RAM address 0xD0060E follows this lifecycle:
1. Written to 0xFF at step 1425 (PC=0x07CAEA) — part of FP eval chain
2. **Cleared back to 0x00 at step 1688 (PC=0x07FA7F)** — BEFORE dispatch table reads it
3. Also: manual seed of 0x28 gets overwritten to 0x00 at step 666 (PC=0x0828C0)

The dispatch table at 0x0686EF reads 0xD0060E via `LD A,(0xD0060E)` at 0x0663AD. If the byte is 0x00, dispatch falls through to the wrong handler.

## Task
1. Build a probe at `TI-84_Plus_CE/probe-phase134-fp-clearing.mjs` that:
   - Uses the same boot pattern as `probe-phase133-fp-jrerror-trace.mjs` (MEM_INIT, create tokens, set up ParseInp)
   - Runs gcd(12,8): tokens [0xBB, 0x07, 0x31, 0x32, 0x2B, 0x38, 0x29, 0x3F]
   - At steps near 1688 (say steps 1680-1700), logs the FULL PC trail (last 10 PCs) and the current instruction
   - At steps near 666 (say steps 660-680), also logs the PC trail
   - For PC=0x07FA7F: read the ROM bytes at 0x07FA7F through 0x07FA90 (about 17 bytes) and disassemble them manually. What instruction writes to 0xD0060E? Is it an `XOR A; LD (0xD0060E),A` pattern? Or `LD (nn),HL` where HL=0?
   - For PC=0x0828C0: same — read ROM bytes at 0x0828C0 through 0x0828D0 and determine the instruction

2. Additionally, do a static ROM byte dump analysis:
   - Read 32 bytes from ROM at offset 0x07FA70 (so we see context around 0x07FA7F)
   - Read 32 bytes from ROM at offset 0x0828B0 (context around 0x0828C0)
   - Disassemble both regions to identify:
     - What subroutine are these clearing instructions part of?
     - Are they "cleanup" after FP operation, or initialization before?
     - What's the calling context (what CALLs lead here)?

3. Run the probe and report:
   - The exact instruction at 0x07FA7F that clears 0xD0060E
   - The exact instruction at 0x0828C0 that clears 0xD0060E
   - Whether these are cleanup (post-operation reset) or init (pre-operation zero)
   - Recommendation: should we skip/NOP these clears, or should the category byte be set AFTER them?

- ROM addresses are little-endian 3-byte in eZ80 ADL mode

## Success criteria

- Static disassembly of both regions provided
