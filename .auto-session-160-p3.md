# Session 160 P3: Examine OP1toOP2 post-0x07CA48 code path — what does it do with the zeroed result?

## Context
You are working on a TI-84 Plus CE ROM transpilation project. The gcd handler calls OP1toOP2 at 0x07C747, which copies OP1 into OP2 and then calls 0x07CA48 (normalization). After 0x07CA48 returns (with mantissa potentially zeroed), execution continues at 0x07C74F (the return address from the CALL). 

Session 159 showed 0x07CA48 has a RET at 0x07CA9F and does NOT fall through to FPDiv. So after normalization returns, OP1toOP2 continues with whatever 0x07CA48 left in OP1.

**KEY QUESTION**: What does the code at 0x07C74F (return from CALL 0x07CA48) DO with the zeroed OP1? Does it use the zeroed value (which would be a bug in the calling convention), or does it restore OP1 from somewhere?

## Task
Build a probe file `TI-84_Plus_CE/probe-phase160-op1toop2-flow.mjs` that:

1. **Part A: Disassemble OP1toOP2 function (0x07C747 through ~0x07C770)**
   - Read ROM bytes from 0x07C740 through 0x07C780 (64 bytes)
   - Decode eZ80 ADL-mode instructions
   - Map the full function flow: entry at 0x07C747, the CALL 0x07CA48 at 0x07C74B, and all instructions after the return at 0x07C74F
   - Does it RET, JP, or fall through to another function?
   - Does it read/restore OP1 from FPS or OP3/OP5 backup?

2. **Part B: Dynamic trace of OP1toOP2 during gcd**
   - Set up runtime with cold boot + MEM_INIT
   - Call gcd handler at 0x068D3D with OP1=12.0, OP2=8.0 (standard test case)
   - Set an onBlock callback that logs OP1 (first 4 bytes) at every block between 0x07C740-0x07C780
   - Also log OP1 at blocks 0x07CA48 (normalization entry) and 0x07CA9F area (normalization exit)
   - Run up to 2000 steps
   - Show the OP1 values before and after each 0x07CA48 call

3. **Part C: Check what OP1toOP2 does BEFORE calling 0x07CA48**
   - Session 157 said "OP1 was copied into OP2 by OP1toOP2 BEFORE calling 0x07CA48"
   - Verify: at 0x07CA48 entry, are OP1 and OP2 identical?
   - If so, 0x07CA48 normalizes OP1 (destroying it) while OP2 retains the original value
   - After 0x07CA48 returns, does OP1toOP2 use OP2 (preserved) or OP1 (zeroed)?

## Key addresses
- 0x07C747 = OP1toOP2 entry
- 0x07C74B = CALL 0x07CA48 (inside OP1toOP2)
- 0x07C74F = return address from CALL 0x07CA48
- 0x07CA48 = normalization entry
- 0x07CA9F = normalization RET
- OP1 = 0xD005F8 (9 bytes)
- OP2 = 0xD00603 (9 bytes)
- 0x068D3D = gcd handler entry
- 0x068D61 = gcd calls OP1toOP2 at 0x07C747

## eZ80 ADL-mode instruction reference
- CALL nn = 0xCD + 3-byte LE address
- JP nn = 0xC3 + 3-byte LE address  
- RET = 0xC9
- LD HL,nn = 0x21 + 3-byte LE
- LD DE,nn = 0x11 + 3-byte LE
- LD (nn),A = 0x32 + 3-byte LE address
- LD A,(nn) = 0x3A + 3-byte LE address
- LD (HL),n = 0x36 + 1-byte immediate
