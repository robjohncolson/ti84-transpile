# Auto-Session 151 P1: FP gcd — post-loop sign-byte cleanup

## Context
The TI-84 CE ROM transpiler lifts eZ80 machine code to JavaScript. The FP gcd function (gcd(12,8)) runs 1442 steps but returns E_Domain (0x84) instead of 4.0. The root cause is fully traced:

- The Euclidean loop's FP subtraction sets sign bits (bit 7) on OP1[0] and OP2[0] during computation
- During the loop body, a repair routine at 0x07FA86 clears sign bits back to 0x00
- But the FINAL two corruptions (step 1428: OP2[0]→0x80 at PC=0x07C783, step 1439: OP1[0]→0x80 at PC=0x068D59) happen AFTER the last repair and AFTER the last type validator check (step 1064)
- The type validator at 0x07F831 always sees clean bytes — intercepting there is a no-op (session 150 proved this)
- 0x80 = negative Real sign bit, NOT invalid type code. The fix is simply masking: `OP1[0] &= 0x3F; OP2[0] &= 0x3F`

## Task
Build a probe `TI-84_Plus_CE/probe-phase151-gcd-postloop-fix.mjs` that:

1. **Approach A — intercept after the last type validator check**: Use `onBlock` to intercept at any block executed AFTER step 1064 that is part of the return path. A good candidate: intercept at the block containing 0x082912 (FPS dec-9 helper, one of the terminal blocks). When 0x082912 fires AND we're past step 1400, mask `mem[0xD005F8] &= 0x3F` (OP1[0]) and `mem[0xD00603] &= 0x3F` (OP2[0]). Then let execution continue. Check if errNo stays 0x00 and OP1 contains 4.0 (`[00 80 40 00 00 00 00 00 00]`).

2. **Approach B — brute-force post-loop mask**: Run gcd(12,8) for exactly 1442 steps (or until it naturally completes/errors), then manually mask OP1[0] and OP2[0] and check if the type validator would now accept them. This is diagnostic only — doesn't fix the runtime path.

3. **Approach C — find the exact return block**: After the gcd handler at 0x068D3D finishes its work, it must RET or JP somewhere. Track what block executes at step 1442+ — that's where the sign masking should go. Use onBlock to log blocks at steps 1440-1445.
