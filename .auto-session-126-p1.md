# Auto-Session 126 — Priority 1: ENTER Keypress Through Full OS Event Loop Probe

## Task
Build a probe `TI-84_Plus_CE/probe-phase25be-enter-event-loop.mjs` that simulates a single ENTER keypress through the REAL OS event loop path and produces OP1=5.0.

## Background
Sessions 99-115 extensively traced the ENTER → ParseInp path. The validated approach is:
- **Direct ParseInp at 0x099914** with OP1=zeros, MEM_INIT seeding, tokenized expression at userMem works perfectly (918 steps, OP1=5.0)
- The REAL OS path is: keystroke → keyboard MMIO → ISR → kbdKey → CoorMon → home handler → ENTER dispatch → common-tail at 0x058693 → ParseInp at 0x0586E3
- Session 105 validated: entering at **0x0586CE** (past the 0x09215E LCD-loop blocker) reaches ParseInp in ~130 steps
- Session 108 found: the trampoline's extra CALL shifts errSP by 3 bytes — direct ParseInp at 0x099914 is the correct approach, NOT the trampoline
- Session 110 found: ParseInp leaves FPS/OPS dirty — must reset after each eval

## What to build
The probe should test TWO scenarios:

### Scenario A: Common-tail bypass at 0x0586CE → ParseInp
1. Cold boot + kernel init + MEM_INIT (use the standard pipeline from existing probes like `probe-phase25ax-multi-expr.mjs`)
2. Write tokenized "2+3" (bytes [0x32, 0x70, 0x33, 0x3F]) at userMem=0xD1A881
3. Set begPC=curPC=0xD1A881, endPC=0xD1A884 (at addresses 0xD02317, 0xD0231A, 0xD0231D)
4. Clear OP1 (9 bytes at 0xD005F8)
5. Seed allocator pointers: OPBase(0xD02590)=0xD3FFFF, OPS(0xD02593)=0xD3FFFF, pTemp(0xD0259A)=0xD3FFFF, progPtr(0xD0259D)=0xD3FFFF, FPSbase=FPS=0xD1A881 (at 0xD0258A and 0xD0258D)
6. Set up PushErrorHandler error frame (see existing probes for pattern)
7. Execute starting at 0x0586CE with 10,000 step budget
8. Check: did execution reach 0x099914 (ParseInp)? Did it reach FAKE_RET? What is OP1?

### Scenario B: Direct ParseInp baseline (control)
1. Same setup as Scenario A
2. Execute starting at 0x099914 directly
3. Confirm OP1=5.0, errNo=0x8D (baseline)
