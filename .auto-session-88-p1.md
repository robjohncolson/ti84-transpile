# Task: ParseInp variable-recall data offset investigation

## Context
This is a TI-84 Plus CE ROM transpilation project. We lift eZ80 machine code into JavaScript and run it in a browser. The transpiled ROM is at `TI-84_Plus_CE/ROM.transpiled.js` (load via dynamic import). The runtime is in `TI-84_Plus_CE/cpu-runtime.js`, peripherals in `TI-84_Plus_CE/peripherals.js`, FP decode in `TI-84_Plus_CE/fp-real.mjs`.

## Problem
Session 87's probe `probe-phase25ab-parseinp-recall.mjs` found that ParseInp with OP1=variable "A" containing stored 42.0 returns OP1=0.0 (710 steps, errNo=0x00). The variable IS found (errNo=0x00 confirms), but the recalled value is 0.0 instead of 42.0.

Two hypotheses:
- (a) The VAT data pointer DE=0xD1A881 is correct, but allocator activity during ParseInp (newDataPtr moved 0xD1A88A→0xD1A893) overwrites the 9 bytes at DE
- (b) The variable data is at a different offset from DE — maybe DE points to the VAT header, not the data

## What to build
Create a new probe file: `TI-84_Plus_CE/probe-phase25ac-recall-offset.mjs`

The probe should:
1. Cold boot + MEM_INIT (same as existing probes — see `probe-phase25ab-parseinp-recall.mjs` for the exact boot sequence pattern)
2. CreateReal("A") at 0x08238A — capture DE return value
3. Write 42.0 BCD (`[0x00, 0x81, 0x42, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]`) to DE
4. **ALSO** write 42.0 to DE-9, DE+9, and DE+2 (offset by the 2-byte VAT header) — so we can check all plausible data locations
5. Dump memory around DE: bytes at [DE-18 .. DE+27] (45 bytes total) — BEFORE ParseInp
6. Set OP1 = variable "A" name: `[0x00, 0x41, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]`
7. Setup tokens "2+3" at 0xD00800 with begPC/curPC/endPC
8. Run ParseInp at 0x099914 with budget 1,500,000 steps
9. After ParseInp, dump same 45-byte region again
10. Track newDataPtr, OPBase changes during execution (snapshot before/after)
11. Report which of the 42.0 copies (if any) survived, which were overwritten, and what OP1 contains

**Key addresses:**
- MEM_INIT_ENTRY = 0x09DEE0
- CREATEREAL_ENTRY = 0x08238A
- PARSEINP_ENTRY = 0x099914
- OP1_ADDR = 0xD005F8 (9 bytes)
- ERR_NO_ADDR = 0xD008DF
- ERR_SP_ADDR = 0xD008E0
- TOKEN_BUFFER_ADDR = 0xD00800
- BEGPC/CURPC/ENDPC = 0xD02317/0xD0231A/0xD0231D
- TEMPMEM/FPSBASE/FPS = 0xD02587/0xD0258A/0xD0258D
- OPBASE/OPS = 0xD02590/0xD02593
- PTEMP/PROGPTR = 0xD0259A/0xD0259D
- NEWDATA_PTR = 0xD025A0
- FAKE_RET = 0x7FFFFE, ERR_CATCH_ADDR = 0x7FFFFA, MEM_INIT_RET = 0x7FFFF6, CREATEREAL_RET = 0x7FFFF2

**Boot sequence** (copy from `probe-phase25ab-parseinp-recall.mjs`):
- Cold boot: runFrom(0x000000, 'z80', maxSteps 20000, maxLoopIterations 32)
- Reset CPU: halted=false, iff1/iff2=0, sp=0xD1A87E-3, fill 0xFF at sp for 3 bytes
- Kernel init: runFrom(0x08C331, 'adl', maxSteps 100000, maxLoopIterations 10000)
- Set: mbase=0xD0, iy=0xD00080, hl=0, halted=false, iff1/iff2=0, sp=0xD1A87E-3, fill 0xFF
- Post init: runFrom(0x0802B2, 'adl', maxSteps 100, maxLoopIterations 32)
- Before each OS call: halted=false, iff1/iff2=0, madl=1, mbase=0xD0, iy=0xD00080, f=0x40, ix=0xD1A860, sp=STACK_RESET_TOP-12, fill 0xFF at sp for 12 bytes

**Error frame setup** (minimal 6-byte frame on stack before ParseInp):
- Push FAKE_RET (0x7FFFFE) onto stack (3 bytes)
- Write 6-byte error frame below SP: 3 zero bytes + ERR_CATCH_ADDR (0x7FFFFA) at offset +3
- Set errSP at 0xD008E0 to point to the error frame base
- Clear errNo at 0xD008DF to 0x00

**Timer IRQ**: MUST be disabled: `createPeripheralBus({ timerInterrupt: false })`

**Token bytes for "2+3"**: `[0x32, 0x70, 0x33, 0x3F]` (where 0x32='2', 0x70='+', 0x33='3', 0x3F=newline/enter)

Also write a report to `TI-84_Plus_CE/phase25ac-recall-offset-report.md`.

## Reference
