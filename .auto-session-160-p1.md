# Session 160 P1: Disassemble ROM bytes at 0x07C74B to verify CALL target + characterize 0x07CA48

## Context
You are working on a TI-84 Plus CE ROM transpilation project. The ROM is at `TI-84_Plus_CE/ROM.rom` (4MB eZ80 binary). The transpiler lifts eZ80 machine code into JavaScript.

We are investigating why `gcd(12,8)` produces E_Domain instead of 4. The gcd handler at 0x068D3D calls OP1toOP2 at 0x07C747, which internally calls 0x07CA48 (a normalization function). 0x07CA48 shifts the mantissa left (via Shl14 at 0x07FB33) and decrements the exponent until exp < 0x80. For gcd inputs with exp >= 0x80, this DESTROYS the mantissa entirely (shifts all digits out), producing OP1=0.

**CRITICAL QUESTION**: Is 0x07CA48 a FP-to-integer conversion helper? What is it SUPPOSED to do? And does block 0x07C74B (inside OP1toOP2) actually CALL 0x07CA48, or could our transpilation have the wrong target?

## Task
Build a probe file `TI-84_Plus_CE/probe-phase160-07c74b-disasm.mjs` that:

1. **Part A: Disassemble ROM bytes at 0x07C74B** (the instruction inside OP1toOP2 that calls 0x07CA48).
   - Read ROM bytes from 0x07C740 through 0x07C760 (32 bytes).
   - Print each byte in hex.
   - Manually decode the eZ80 ADL-mode instructions. CALL nn in ADL mode = 0xCD followed by 3-byte little-endian address.
   - Verify: does the CALL at 0x07C74B target 0x07CA48?

2. **Part B: Disassemble the FULL 0x07CA48 function** from entry to its RET at 0x07CA9F.
   - Read ROM bytes 0x07CA48 through 0x07CAA0.
   - Decode each instruction with its address.
   - Annotate what each instruction does in the context of eZ80 FP math.
   - Look for patterns: does this look like FP-to-integer extraction? Denormalization? Something else?
   - Specifically check: after the normalization loop (shift left + decrement exp), what happens to the result? Is there integer extraction logic (storing the shifted-out digits)?

3. **Part C: Check the callers of 0x07CA48** in the ROM.
   - Scan ROM for `CD 48 CA 07` (CALL 0x07CA48) and `C3 48 CA 07` (JP 0x07CA48).
   - For each caller, read the surrounding context (8 bytes before and after) to understand the calling pattern.
   - How many callers are there? Are they all from FP math routines?

## Key files

Print all findings clearly with hex addresses and decoded instructions. End with a CONCLUSIONS section summarizing:
- Whether the CALL at 0x07C74B targets 0x07CA48 (confirmed/denied)
- What 0x07CA48 appears to implement
- Whether the function has any integer-extraction logic that would explain why mantissa destruction is intentional
