# Session 146 P2 — Graph Y-coordinate mapping: trace IPoint's VRAM address computation

## Context

This is a TI-84 Plus CE ROM transpilation project. We lift eZ80 machine code from a 4MB ROM into JavaScript and run it in a browser. The ROM file is at `TI-84_Plus_CE/ROM.rom`.

**Session 145 found that GraphPars Y1=X produces 320 successful IPoint calls but ALL pixels land on row 239 (bottom row).** The VRAM addresses written are 0xD65580–0xD655A6, which all correspond to the last row of the 320x240 16bpp framebuffer. For Y=X, pixels should span rows 0–238 diagonally.

IPoint (0x07B451) writes pixels via the NZ path when IY+43 bit 2 is SET and IY+74 bit 2 is SET. The pixel write is `LD (BC),A` at 0x07B682 where BC comes from address 0xD02A8A. The VRAM address computation populates 0xD02A8A before the write.

## Task

Build a new probe: `TI-84_Plus_CE/probe-phase146-graph-ycoord.mjs`

### Part A: Disassemble IPoint's VRAM address computation

Read ROM bytes from 0x07B451 to 0x07B700 (~688 bytes). Focus on finding where BC (the framebuffer address at 0xD02A8A) is computed. The formula should involve:
- A framebuffer base address (likely from an LCD control register or RAM variable)
- Y coordinate × stride (320 pixels × 2 bytes/pixel = 640 bytes/row for 16bpp)
- X coordinate × 2 (bytes per pixel)

Look for: multiplication by 640 or shift-add equivalent, base address loads, writes to 0xD02A8A.

### Part B: Trace VRAM address for multiple Y values

Using the existing probe pattern from `probe-phase145-graph-flags.mjs`:
1. Set up full graph pipeline: cold boot, MEM_INIT, CreateEqu (Y1 = tX, equation at userMem), seed allocator, seed graph RAM (pixWideP=320, pixWide_m_2=238, pen color=0x10, IY+43 bit 2 SET, IY+74 bit 2 SET)
2. Run 5 IPoint calls with DIFFERENT Y coordinates but same X:
   - (X=160, Y=0) — should be top row
   - (X=160, Y=60) — upper quarter
   - (X=160, Y=120) — middle
   - (X=160, Y=180) — lower quarter
   - (X=160, Y=239) — bottom row (where all pixels currently land)
3. For each call, use onBlock to capture the value at 0xD02A8A after IPoint returns
4. Print all 5 VRAM addresses and compute the stride (delta between consecutive Y values)
5. Also dump: what registers does IPoint receive? Where does it get X and Y from?

### Part C: Find the Y-coordinate input path

Session 145's GraphPars produces Y values in OP1 after tX intercept. But IPoint needs pixel coordinates, not graph coordinates. There must be a Y-to-pixel conversion: `pixelY = (Ymax - Y) / (Ymax - Ymin) * screenHeight`. 

Trace: what address does IPoint read Y from? Is it a RAM slot like curGY (0xD022D1-ish) or DE register? Use onBlock at IPoint entry (0x07B451) to capture all registers for each of the 5 test points.

Also check: is there a graph-to-pixel conversion routine between GraphPars output and IPoint call that we're skipping or that has wrong inputs?

### Part D: Test framebuffer base seeding

The 16bpp LCD framebuffer typically starts at 0xD40000 on the TI-84 CE. Row 0 address = base, row 239 = base + 239×640 = base + 0x25780.

If all pixels hit row 239 (addr ~0xD65580), that means base + 0x25580 = 0xD65580, so base = 0xD40000. That's correct! So the BASE is fine — the Y coordinate being passed to IPoint is always 239.

The bug is UPSTREAM — the Y pixel coordinate computation always produces 239. Check:
1. What RAM slots hold curGY / graph Y pixel coordinate?
2. What values are in those slots during the GraphPars loop?
3. Does the graph-to-pixel conversion (likely at a routine between 0x099874 and IPoint) even exist in the current pipeline?

## Key addresses

- IPoint entry = 0x07B451
- Framebuffer addr register = 0xD02A8A (3 bytes, read into BC)
- LCD VRAM base = 0xD40000 (16bpp, 320x240, 640 bytes/row)
- plotSScreen = 0xD09466 (8bpp monochrome)
- OP1 = 0xD005F8
- curGY = 0xD022D1 (2 bytes? check)
- Graph window: Xmin=0xD01E33, Xmax=0xD01E3C, Ymin=0xD01E4E, Ymax=0xD01E57
- pixWideP = 0xD014FE
- pixWide_m_2 = 0xD01501
- Pen color = 0xD026AE
- IY+43 = 0xD000AB (bounds check path selector)
- IY+74 = 0xD000CA (pixel write path selector)
