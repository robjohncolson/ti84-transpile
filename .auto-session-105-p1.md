# Task: Bypass common-tail blockers — enter at 0x0586CE or 0x0586E3

## Context
This is a TI-84 Plus CE ROM transpilation project. We lift eZ80 machine code to JavaScript and run it in Node.js. The OS ENTER key handler has a "common tail" at 0x058693 that eventually calls ParseInp (expression parser) at 0x0586E3 via trampoline 0x099910→0x099914. However, several pre-ParseInp calls in the common tail are blockers:

- 0x082961 (FPS push — NOT a blocker, completes in 12 steps)
- 0x09215E (calls 0x0800B8, then enters LCD refresh infinite loop via 0x0BD19F — THE BLOCKER)
- 0x082902 (unknown, may block)
- 0x0A1FD1 (unknown, may block)
- 0x0A27DD (unknown, may block)

Session 104 proved 0x09215E is the sole known blocker. The JR NZ at 0x0586CC does NOT skip ParseInp (bit 4 of IY+52 = 0 after MEM_INIT).

## What to build
Create a probe file: `TI-84_Plus_CE/probe-phase25at-bypass-common-tail.mjs`

The probe should test THREE scenarios:

### Scenario A: Enter at 0x0586CE (past 0x09215E and BIT 4 test)
- Full MEM_INIT via JT slot
- Seed allocator pointers: OPBase(0xD02590)=0xD3FFFF, OPS(0xD02593)=0xD3FFFF, pTemp(0xD0259A)=0xD3FFFF, progPtr(0xD0259D)=0xD3FFFF
- Seed FPSbase and FPS at 0xD0258A and 0xD0258D to 0xD1A881
- Seed tokenized "2+3" at userMem 0xD1A881: bytes [0x72, 0x70, 0x71, 0x3F] (tAdd=0x70, t2=0x72, t3=0x71, newline=0x3F)
- Seed begPC(0xD02317)=0xD1A881, curPC(0xD0231A)=0xD1A881, endPC(0xD0231D)=0xD1A885
- Push error frame: write FAKE_RET address to error handler stack, set errorOffset at 0xD02594 (actually use the PushErrorHandler pattern from prior probes)
- Set IY=0xD00080
- Set SP to a safe stack area (e.g., 0xD1A860)
- Push FAKE_RET sentinel (0xCECECE) onto stack as return address
- Run from PC=0x0586CE with 50K step budget
- Record: did ParseInp (0x099914) get hit? What's OP1 after? What's errNo? How many steps?

### Scenario B: Enter at 0x0586E3 (direct CALL 0x099910 = ParseInp trampoline)
- Same seeding as Scenario A
- Run from PC=0x0586E3 with 50K step budget
- This should behave identically to the known-good direct ParseInp call

### Scenario C: Enter at 0x099914 (direct ParseInp, control)
- Same seeding as Scenario A
- Run from PC=0x099914 with 50K step budget
- This is the known-good baseline — should return OP1=5.0, errNo=0x8D

## How to load ROM and create CPU (copy this pattern exactly)
```javascript
import { readFileSync } from 'fs';
import { createCPU } from './cpu-runtime.js';

const rom = readFileSync(new URL('./ROM.rom', import.meta.url));
const transpiled = await import('./ROM.transpiled.js');
const cpu = createCPU(rom, transpiled.default, { timerInterrupt: false });
```

Note: `ROM.transpiled.js` must exist (not just the .gz). If it doesn't exist, run `node scripts/transpile-ti84-rom.mjs` first.

## Important constraints
- Use `{ timerInterrupt: false }` in createPeripheralBus/createCPU options
- MEM_INIT JT slot is at ROM address 0x020164 — call it via `cpu.callFunction(0x020164)` or whatever the runtime API is. Check the reference probes for the exact API.
- FAKE_RET sentinel: use whatever the reference probes use (check probe-phase25ao-trampoline-0x099910.mjs)
- Read OP1 from 0xD005F8 (9 bytes), errNo from 0xD008AF
- DO NOT modify cpu-runtime.js, ez80-decoder.js, peripherals.js, or ROM.transpiled.js
- DO NOT modify the transpiler or any existing probe files

## Success criteria
- Scenario C (control) returns OP1=5.0, errNo=0x8D — this validates the seeding is correct
- Scenario B should match Scenario C (same result, maybe +3 steps for trampoline)
- Scenario A is the key test — if it returns OP1=5.0, the full ENTER→common-tail→ParseInp pipeline works when display calls are bypassed
- If Scenario A hits a new blocker (0x0A1FD1, 0x0A27DD, etc.), document which call blocks and at what step
