# Auto-Session 151 P2: IPoint pen color investigation — why A=0x00 at pixel write

## Context
The TI-84 CE ROM transpiler lifts eZ80 machine code to JavaScript. IPoint (pixel drawing, entry 0x07B451) IS writing VRAM bytes at correct addresses (session 150 confirmed with sentinel fill). But it writes 0x0000 instead of the seeded pen color (0x0010 at 0xD026AE).

The pixel store instruction is `LD (BC),A` at 0x07B682. A=0x00 when it executes. The pen color at 0xD026AE is 0x10. Something between pen color load and pixel store either never loads the pen color into A, or overwrites it.

## Key background
- IPoint entry: 0x07B451. Stores A (draw mode) to 0xD02AC8.
- Session 148 confirmed: C=pixelY (register), DE=pixelX (register). No RAM reads for coordinates.
- Session 128 documented: "Loads pen color from 0xD026AE"
- Session 129 claimed pen color 0x10 worked — but may have been checking plotSScreen (monochrome buffer) not LCD VRAM (16bpp at 0xD40000+)
- drawMode=1 = normal draw, drawMode=0 = erase (writes 0xFFFF)
- Pen color at 0xD026AE. Session 150: seeded to 0x0010, but A=0x00 at write time.
- VRAM is 16bpp BGR565. A single-byte pen color 0x10 may be wrong for 16bpp (needs 2 bytes).

## Task
Build a probe `TI-84_Plus_CE/probe-phase151-ipoint-pencolor.mjs` that:

1. **Trace A register through IPoint**: Use onBlock to log the A register value at every block from entry (0x07B451) through pixel write (0x07B682). Identify exactly where A gets its value and where it might be overwritten.

2. **Trace pen color load**: Look for any `LD A,(0xD026AE)` or equivalent instruction in the IPoint code path. Session 128 says pen color is loaded from 0xD026AE — find the exact PC where this happens and what A equals after.

3. **Test 16bpp pen color seeding**: The pen color slot may be 2 bytes wide for 16bpp mode. Try seeding 0xD026AE with 0x1F (blue in BGR565 = 0x001F). Also try seeding TWO bytes: mem[0xD026AE]=0x1F, mem[0xD026AF]=0x00. Check what A register contains at 0x07B682.

4. **Check session 129 probe**: Read `TI-84_Plus_CE/probe-phase129-graph-render.mjs` and check how it detected "pixel writes". Did it check LCD VRAM (0xD40000+) or plotSScreen (0xD52C00)? This determines whether session 129's "pen color works" claim was valid.

## Key addresses
- IPoint entry: 0x07B451
- Pixel store: `LD (BC),A` at 0x07B682 (and 0x07B68A for second byte)
- Pen color: 0xD026AE (1 or 2 bytes)
- Draw mode: 0xD02AC8
- OP1: 0xD005F8
- VRAM base: 0xD40000
- plotSScreen: 0xD52C00
- IY+0x2B (IY+43) bit 2: must be SET for IPoint to work
- IY+0x3C bit 0, IY+0x02 bit 1: may gate pixel write
