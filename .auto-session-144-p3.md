# Session 144 P3: Graph — Trace IPoint Pixel-Write Path (NZ Branch)

## Context
Working in `C:/Users/rober/Downloads/Projects/school/ti84-transpile`.

IPoint at 0x07B451 has two paths controlled by BIT 2,(IY+43):
- **Z path (bit 2 clear)**: Uses 8-bit X bounds check (`CP L`), DOES write pixels to plotSScreen (0xD09466). Session 129 got 17 non-zero bytes. Works for X range 0-255.
- **NZ path (bit 2 set)**: Uses 24-bit bounds check via CALL 0x04C979, supports full 320-pixel width. IPoint returns successfully 160 times but writes **ZERO pixels anywhere** — checked plotSScreen, LCD VRAM 8bpp/16bpp, and full 0xD00000-0xD60000 (session 143).

Session 143 concluded: "IPoint's pixel-write instruction is conditional on additional unseeded state (framebuffer base pointer, LCD control register, or draw-enable flag)."

Key addresses:
- IY = 0xD00080, so IY+43 = 0xD000AB
- plotSScreen = 0xD09466 (76,800 bytes)
- pixWideP = 0xD014FE (seeded to 320)
- pixWide_m_2 = 0xD01501 (seeded to 238)
- Pen color = 0xD026AE
- Draw mode flag = 0xD02AC8
- IPoint entry = 0x07B451
- Bounds check (NZ path) = 0x07B7B6 uses 24-bit compare via CALL 0x04C979

## Task

1. **Build `probe-phase144-ipoint-nzpath.mjs`** based on `probe-phase143-graph-vram.mjs` and `probe-phase139-graphpars-render.mjs` (read both first).

2. **Disassemble the IPoint pixel-write section**: Read ROM bytes from 0x07B451 to 0x07B850 (~1024 bytes). Decode key instruction sequences. Focus on:
   - Where does the Z path actually write a pixel byte? Find the `LD (HL),A` or `LD (DE),A` or similar write instruction.
   - Does the NZ path reach the same write instruction, or does it have a different pixel-write path?
   - Are there additional flag checks or RAM reads between the bounds check and the pixel write?

3. **Step-trace a SINGLE IPoint call on the NZ path**: Use a known-good coordinate (e.g., X=160, Y=60 which should be in-bounds for 320-wide). Log EVERY step (PC, registers) for up to 200 steps. Find:
   - Does it reach the pixel-write instruction?
   - If not, where does it bail out? What condition?
   - What RAM addresses does it read that might be uninitialized?

4. **Compare with Z path**: Also step-trace a SINGLE IPoint call on the Z path (bit 2 clear, X=50, Y=60). Log every step. Compare the two traces to find where the NZ path diverges from the pixel-write.

5. **Identify missing seeds**: Based on the trace, identify any RAM addresses that need seeding for the NZ path to write pixels. Propose exact values.
