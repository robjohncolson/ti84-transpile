# Session 140 P2 — FP gcd Type Check Disassembly at 0x07F88D / 0xD00604

## Task
Disassemble the gcd handler's type/domain check trail to understand what 0xD00604 encodes, and find what value lets gcd pass the domain check.

## Context
Session 139 P2 found that gcd(12,8) hits E_Domain (0x84) even with properly-seeded FP stack operands (12.0 and 8.0 as BCD). The error trail is:
- 0x07F881 → 0x07F88D (reads byte at 0xD00604) → 0x07F896 → 0x068D65 → 0x068D5D (E_Domain error raise)

0xD00604 is at offset +12 from OP1 base (0xD005F8). So it's OP1[12] — in the "extended OP1 region" past the standard 9-byte OP1.

The gcd handler entry is at 0x06859B. Dispatch path: 0x06859B → LD A,0x28 → CALL 0x0689DE → SUB 0x20 → index into jump table at 0x0689F9 → JP (HL).

## Deliverables

### Part A: Static Disassembly
Build `probe-phase140-gcd-disasm.mjs` that:
1. Read ROM bytes from the gcd handler trail: 0x07F870-0x07F8C0 (80 bytes around the type check)
2. Also read 0x068D50-0x068D70 (E_Domain error path)
3. Also read 0x06859B-0x0685C0 (gcd handler entry)
4. Decode each instruction manually (eZ80 instruction set) and print the disassembly
5. Specifically identify:
   - What instruction at 0x07F88D reads from 0xD00604
   - What comparison/test follows
   - What value at 0xD00604 would cause the check to PASS (not branch to error)

### Part B: Brute-Force Flag Byte Test
In the same probe, after the disassembly:
1. For each candidate value 0x00 through 0x0F (and also 0x1C, 0x20, 0x28, 0x80, 0xFF):
   - Set up: cold boot → MEM_INIT → seed FPS with 12.0 and 8.0 → seed OP1/OP2 with 12.0/8.0
   - Write the candidate value to 0xD00604
   - Call gcd handler at 0x06859B with max 2000 steps
   - Record: errNo, OP1 result, steps taken, final PC
2. Report which values pass the domain check (errNo != 0x84)
3. If any value produces errNo=0x00, report the OP1 result (expected: 4.0 for gcd(12,8))

### Part C: Cross-Reference 0xD00604
1. Search the ROM binary for all instructions that read from or write to 0xD00604
   - Look for byte patterns: `3A 04 06 D0` (LD A,(0xD00604)), `32 04 06 D0` (LD (0xD00604),A)
   - Also `2A 04 06` / `22 04 06` for LD HL,(nn) / LD (nn),HL
   - Also `ED 4B 04 06` for LD BC,(nn), `ED 5B 04 06` for LD DE,(nn)
2. Report all ROM addresses that reference 0xD00604 and which blocks they're in

## Key Constants

- OP2 = 0xD00602 (overlaps with OP1 extended — per ti84pceg.inc, OP2 starts at 0xD00602, which is OP1+10)
- Actually check: in `references/ti84pceg.inc`, search for `OP1`, `OP2`, `OP3` to find exact addresses
