# Task: Trace ParseInp's ChkFindSym Decision Point — Two Code Paths

## Context
ParseInp (0x099914) follows DIFFERENT code paths depending on whether OP1 names an existing variable:
- **Without variable (OP1 all zeros)**: 918 steps, computes OP1=5.0, errNo=0x8D (ErrUndefined)
- **With pre-created variable "A" (OP1=[00 41 00 00 00 00 00 00 00])**: 253 steps, OP1 unchanged, errNo=0x00

Session 84 disassembled the ParseInp entry at 0x099914..0x099954. The key call is `CALL 0x099B18` which calls ChkFindSym (0x08383D). If ChkFindSym FINDS the variable, the code walks the VAT entry and writes begPC/curPC from it. If NOT found, a different path is taken.

## Your Task
Create a probe `TI-84_Plus_CE/probe-phase25aa-parseinp-two-paths.mjs` that:

1. Uses the same boot/MEM_INIT/CreateReal pipeline from `probe-phase25z-full-pipeline.mjs` (copy the pattern)
2. Runs ParseInp TWICE in the same probe:
   - **Run A**: MEM_INIT → CreateReal("A") → set OP1 to variable A name [00 41 00 00 00 00 00 00 00] → ParseInp("2+3") — capture ALL visited PCs
   - **Run B**: Fresh MEM_INIT (no CreateReal) → clear OP1 to all zeros → ParseInp("2+3") — capture ALL visited PCs
3. For BOTH runs, capture:
   - Every PC visited (store in an array, up to 2000 entries)
   - The value of begPC (0xD02317) and curPC (0xD0231A) after the call to 0x099B18 returns
   - The value of OP1 (9 bytes at 0xD005F8) at every 50-step milestone
   - errNo (0xD008DF) and final OP1 at termination
   - Whether PC 0x099B18 was visited and what happened immediately after (next 5 PCs)
4. Compare the two PC traces and identify the DIVERGENCE POINT — the first PC where Run A and Run B take different paths
5. Write a report to `TI-84_Plus_CE/phase25aa-two-paths-report.md`
