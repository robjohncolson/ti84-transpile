# Session 146 P3 — OneVar: implement JS-side min/max for 8/8 stat coverage

## Context

This is a TI-84 Plus CE ROM transpilation project. We lift eZ80 machine code from a 4MB ROM into JavaScript and run it in a browser.

**Session 140 proved that OneVar (partial entry at 0x0A9325) computes 6/8 stat results correctly for L1=[10,20,30,40,50]**: n=5, x̄=30, Σx=150, Σx²=5500, Sx=15.8114, σx=14.1421. minX and maxX are NOT computed because the partial entry does SET 5,(IY+9) which skips the min/max routine.

**Session 145 confirmed**: list data survives OneVar intact — all 5 elements [10,20,30,40,50] remain at their original RAM addresses after computation. JS-side min/max is trivially computable.

## Task

Build a new probe: `TI-84_Plus_CE/probe-phase146-onevar-minmax.mjs`

### What to build

A probe that demonstrates 8/8 stat coverage by:

1. Running the FULL existing OneVar pipeline (from `probe-phase140-onevar-prealloc.mjs`):
   - Cold boot → MEM_INIT → create L1 list with [10,20,30,40,50] via manual VAT entry → seed allocator → seed stat flags → call OneVar partial entry 0x0A9325
   
2. After OneVar returns (6/8 computed by ROM):
   - Read list element count from the VAT/list header
   - Read all list elements from RAM (they're BCD reals, 9 bytes each)
   - Compute min and max in JavaScript using BCD-to-number conversion via `readReal` from `fp-real.mjs`
   - Write the results back to the stat var slots:
     - minX at 0xD011C7 (9 bytes, BCD format via `writeReal`)
     - maxX at 0xD011D0 (9 bytes, BCD format via `writeReal`)
   
3. Read back and verify ALL 8 stat results:
   - n (0xD01191) = 5
   - x̄ (0xD0119A) = 30
   - Σx (0xD011A3) = 150
   - Σx² (0xD011AC) = 5500
   - Sx (0xD011B5) ≈ 15.8114
   - σx (0xD011BE) ≈ 14.1421
   - minX (0xD011C7) = 10
   - maxX (0xD011D0) = 50

4. Print results in a clear table format, with PASS/FAIL for each stat

### Reference probe

Base your implementation on `probe-phase140-onevar-prealloc.mjs`. That probe has:
- The full pipeline (cold boot, MEM_INIT, list creation with manual VAT entry, stat flag seeding, OneVar call)
- BCD read/write utilities
- Stat var address map

Key details from that probe:
- List elements are stored as 9-byte BCD reals starting at a list data area
- The list VAT entry is manually created with proper structure
- Stat flags: statFlags2 bit 2 at IY+26 (0xD0009A) must be SET to skip ZeroStatVars
- A=1 for single-list mode
- FPS needs pre-pushing with L1 OP1 data
- OneVar entry: 0x0A9325 (partial, SET 5)
- ONEVAR_RET sentinel for return detection

### Implementation notes

- Import `readReal` and `writeReal` from `./fp-real.mjs`
- `readReal(mem, addr)` reads a 9-byte BCD real from memory and returns a JS number
- `writeReal(mem, addr, value)` writes a JS number as a 9-byte BCD real to memory
- List elements are at the list data area. In the session 140 probe, the list data starts at a known offset after the VAT entry. Read the probe to find the exact addresses.
- After OneVar returns with errNo=0x00, simply scan the list data to find min/max values

## Constraints

- Use `createPeripheralBus({ timerInterrupt: false })`
- Use the same cold boot, MEM_INIT, and allocator seeding as `probe-phase140-onevar-prealloc.mjs`
- Do NOT modify any existing files — only create the new probe file
- Print clear PASS/FAIL output for each of the 8 stats
- Print a summary line: "8/8 stats PASS" or "N/8 stats PASS"

## Success criteria

- OneVar returns errNo=0x00 with 6/8 stats computed by ROM
- JS-side min/max correctly identifies min=10 and max=50
- Written back to minX/maxX slots and verified via readReal
- Final output: "8/8 stats PASS"
