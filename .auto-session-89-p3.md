# Session 89 P3: Flash unlock purpose in error dispatch (0x03E187)

## Goal
Investigate why JError (0x061DB2) calls the flash unlock wrapper at 0x03E1B4/0x03E187 during error dispatch. Session 87 disassembled the routine but the question remains: what does the OS need flash unlocked for during error handling?

## Background
Session 87 fully disassembled 0x03E1B4 and 0x03E187:
- 0x03E1B4: save A, read interrupt state (LD A,I ×2 for eZ80 bug), DI, push AF, restore A, CALL 0x03E187, save A, POP AF, conditional EI, restore A, RET
- 0x03E187: NOP×4, PUSH AF, XOR A, DI, JR+0, DI, RSMIX, IM ADL, OUT0(0x28,A) (flash unlock), IN0 A,(0x28), BIT 2,A, IN0 A,(0x06), RES 2,A, OUT0(0x06,A) (disable flash protection), LD A,0x88, OUT0(0x24,A) (flash command), CP 0x88 (anti-tamper), JP NZ,0x000066 (reset if tampered), POP AF, RET

Session 86 incorrectly claimed 0x03E1B4 "persists errNo" — it actually unlocks flash protection. Session 87 hypothesized this is so the OS can read error message strings from flash ROM.

## What to do
Create `TI-84_Plus_CE/probe-phase25ad-flash-unlock-purpose.mjs` that does a STATIC analysis (no execution needed — this is a ROM analysis task):

1. **Find all callers of 0x03E1B4 and 0x03E187** in the ROM. Scan ROM bytes for `CD B4 E1 03` (CALL 0x03E1B4) and `CD 87 E1 03` (CALL 0x03E187). Note: eZ80 CALL is `CD` followed by 3-byte little-endian address.
2. **For each caller**, read the surrounding ROM context (±20 bytes) to understand what code does BEFORE and AFTER the flash unlock call. Does it:
   - Read strings from flash?
   - Write to flash?
   - Access flash-resident data tables?
   - Do something else entirely?
3. **Cross-reference with JError at 0x061DB2**. The known call chain is: error occurs → JError at 0x061DB2 → calls 0x03E1B4 → flash unlock → continues error dispatch. What does JError do AFTER the flash unlock returns? Read ROM bytes at 0x061DB2+offset to trace what happens next.
4. **Check for error string tables in flash**. The ROM region 0x000000-0x3FFFFF contains the OS. Error strings (like "ERR:UNDEFINED", "ERR:MEMORY", etc.) may be stored in a string table. Search for ASCII patterns like "ERR:" or known error text in the ROM.

## eZ80 CALL encoding
CALL in eZ80 ADL mode = `0xCD` + 3-byte little-endian address. So:
- CALL 0x03E1B4 = `CD B4 E1 03`
- CALL 0x03E187 = `CD 87 E1 03`
