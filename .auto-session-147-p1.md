## Problem

FpPop (entry 0x082957) is supposed to pop a 9-byte entry from the FP stack (FPS) into OP2 (0xD00603). Instead, it always PUSHES (FPS grows by +9). The call chain is:

```
FpPop (0x082957): LD HL,0xD00603  ; HL = OP2 address
  JR +8 → shared tail at 0x082965
  PUSH HL
  LD HL,0x000009              ; 9 bytes
  CALL 0x082BB5               ; THE HELPER THAT DECIDES DIRECTION
  POP HL
  LD DE,(0xD0258D)            ; FPS pointer
  CALL 0x07F978               ; LDI chain (copy BC bytes from HL to DE, or DE to HL)
  LD (0xD0258D),DE            ; store updated FPS
  RET
```

FpPush (0x082961) does the same but with HL=0xD005F8 (OP1 address).

The helper at 0x082BB5 calls 0x082266 which calls 0x04C92E. The comparison in this chain always selects the "push" direction regardless of register pre-seeding. But during actual gcd execution (1442 steps), a REAL pop (FPS→OP2, BC=9) happens at step 1425 via the LDI chain at 0x07F978 through a DIFFERENT code path.

Key RAM:
- FPS pointer: 0xD0258D (3 bytes)
- FPSbase: also near 0xD0258D area (probably 0xD0258A based on MEM_INIT)
- OP1: 0xD005F8 (11 bytes), OP2: 0xD00603 (11 bytes)

### Part C: Test if FPSbase relationship affects direction
Hypothesis: 0x082BB5's comparison checks whether the destination is ABOVE or BELOW FPS, using FPSbase as reference. If dest < FPSbase → pop (copy from stack to register). If dest >= FPSbase → push (copy from register to stack).

Test: Seed FPSbase (try 0xD0258A, 3 bytes) to different values relative to OP2 (0xD00603) and call FpPop. See if the direction changes.
