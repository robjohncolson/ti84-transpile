# Task: Disassemble 0x082bb9 error site and 0x08226b..0x0822a2 failure path

## Context
We are reverse-engineering the TI-84 Plus CE OS ROM. ParseInp (expression parser at 0x099914) fails with errNo=0x8E (ErrMemory). The exact PC trace of the failure path is:

```
0x08226b → 0x0820b5 → 0x0820c3 → 0x08226f → 0x082273 → 0x0822a2 → 0x082bb9 → 0x082bba → 0x061d3e (ErrMemory) → 0x061db2 (longjmp) → FAKE_RET
```

We need to understand EXACTLY what check fails at each step. Session 78 identified 0x082BBE region as ~15 small utility functions including FindSym wrappers. 0x082bb9 is just before 0x082BBE — it likely leads into one of those FindSym wrappers that throws ErrMemory.

## What to do
1. Read `TI-84_Plus_CE/ROM.rom` (binary, 4MB)
2. Statically disassemble these address ranges:
   - **0x08226b..0x082280** (the entry into the failure path — what does 0x08226b do?)
   - **0x0820b5..0x0820d0** (what does 0x0820b5/0x0820c3 do?)
   - **0x0822a2..0x0822b0** (what happens at 0x0822a2 right before the error call?)
   - **0x082bb5..0x082bc5** (what is 0x082bb9? A single instruction? A JP? What does 0x082bba do before calling 0x061d3e?)
3. For each range, decode eZ80 instructions: address, hex bytes, mnemonic, operands
4. Cross-reference RAM addresses against `TI-84_Plus_CE/references/ti84pceg.inc`
5. Determine: what CONDITION triggers the error? Is it a NULL pointer check? A free-space comparison? A FindSym failure?

## Decoder reference
- eZ80 (Z80 superset, 24-bit). `TI-84_Plus_CE/ez80-decoder.js` has the full decoder.
- Key opcodes: `CD xx yy zz` = CALL. `C3 xx yy zz` = JP. `C9` = RET. `20 xx` = JR NZ. `28 xx` = JR Z. `38 xx` = JR C. `30 xx` = JR NC. `DA xx yy zz` = JP C. `D2 xx yy zz` = JP NC. `CA xx yy zz` = JP Z. `C2 xx yy zz` = JP NZ.
- `ED 4B xx yy zz` = LD BC,(nn). `ED 5B xx yy zz` = LD DE,(nn). `2A xx yy zz` = LD HL,(nn).
