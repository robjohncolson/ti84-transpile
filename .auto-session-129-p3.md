# Auto-Session 129 — Priority 3: Edit Buffer Investigation

## Context

Currently the browser-shell uses a JS `<input>` element for direct-eval. The user types expressions into HTML, which are tokenized by JS and passed to ParseInp. The goal is to eventually have characters land in the OS edit buffer via real key-handler dispatch.

This is a RESEARCH/INVESTIGATION task — no browser-shell modifications needed. We need to understand the OS edit buffer system.

## Task

Build `probe-phase129-edit-buffer.mjs` that:

1. **Maps the edit buffer system from ti84pceg.inc**: Read the include file at `TI-84_Plus_CE/references/ti84pceg.inc` and extract all edit-buffer related addresses. Known from the include:
   - editTop: 0xD02437
   - editCursor: 0xD0243A
   - editTail: 0xD0243D
   - editBtm: 0xD02440
   - kbdKey: 0xD0058C (key press equate written by GetCSC dispatch)
   - kbdScanCode: 0xD00587
   - kbdGetKy: 0xD0058D
   Also find: editSym (0xD0244E from session 80), any editBuf/editLen/editMode addresses.

2. **Traces the home-screen digit key handler**: The home-screen handler at 0x058241 processes key events. When a digit key is pressed (e.g., kNum2=0x22 from ti84pceg.inc), the handler should dispatch to a routine that appends the digit token to the edit buffer. Trace:
   - Set up a post-MEM_INIT runtime with cxCurApp=0x40 (home screen) at 0xD007E0
   - Set kbdKey=0x22 (key code for digit '2') at 0xD0058C
   - Call the home-screen handler at 0x058241
   - Monitor writes to the edit buffer region (0xD02437-0xD02450) and the actual buffer data area
   - Run for 50K steps, logging all writes to the editTop/editCursor/editTail/editBtm range

3. **Disassembles the digit-key dispatch path**: From the home-screen handler's key dispatch table, identify which routine handles digit keys (key codes 0x1A-0x23 = k0-k9 per ti84pceg.inc). The dispatch likely branches based on key code ranges.

4. **Maps the edit buffer data area**: editTop/editCursor/editTail/editBtm are POINTERS (24-bit), not the data itself. Find where the actual token data buffer lives by:
   - Reading editTop after MEM_INIT to see if it's initialized
   - If zero, look for edit buffer initialization routines (search ti84pceg.inc for `InitEditBuf` or `BufClr` or similar)
   - The data buffer is likely between editTop and editBtm addresses

5. **Documents the key code → token pipeline**: What happens between GetCSC returning a scan code and a token appearing in the edit buffer? The chain is approximately: GetCSC → scan-to-keycode table (0x09F79B) → kbdKey → CoorMon dispatch → app handler → edit buffer. Document each step.
