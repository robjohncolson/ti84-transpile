# Session 141 P3: .SIS prefix — investigate IPoint bounds check block alignment

## Context
We're transpiling the TI-84 Plus CE ROM to JavaScript. Session 128 found that the IPoint bounds-check at ROM address 0x07B793 uses a `.SIS` prefix (opcode 0x40) before `LD HL,(nn)` to read a 16-bit address with MBASE composition. Session 128 reported that the transpiler treats 0x40 as `LD B,B` instead of `.SIS` prefix.

However, session 129 noted: "Session 128 DISPROVED the .SIS hypothesis — the decoder/transpiler already handle .SIS correctly, and blocks at 0x07B793+ exist with proper `read16` + MBASE composition."

Despite this, session 139/140 still report that graph rendering produces only 17 pixels "concentrated in one row due to the 8-bit X bounds check (.SIS prefix issue from session 128)." So SOMETHING is still wrong with the IPoint bounds check limiting X coordinates.

The decoder at `TI-84_Plus_CE/ez80-decoder.js` line 797 DOES check for `.SIS` prefix (`if (first === 0x40) { modePrefix = 'sis'; ... }`). The transpiler at `scripts/transpile-ti84-rom.mjs` lines 58+76 correctly handle `.SIS` for word width and MBASE. So the decoder/transpiler framework is correct.

**The hypothesis**: The block walker may have discovered the block containing 0x07B793 starting from a seed that causes the 0x40 byte to fall mid-instruction, decoded as `LD B,B` instead of `.SIS` prefix. This is a block alignment issue.
