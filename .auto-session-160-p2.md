# Session 160 P2: Trace gcd with OP1=12, OP2=8 through normalization 0x07CA48

## Context
You are working on a TI-84 Plus CE ROM transpilation project. The ROM is lifted into JavaScript blocks. The transpiler output is at `TI-84_Plus_CE/ROM.transpiled.js` (must gunzip from `.gz` first if not present).

We are investigating `gcd(12,8)`. The gcd handler bumps the exponent by +2 (e.g., 12 with exp 0x81 becomes exp 0x83 = 1200) before entering normalization at 0x07CA48. The normalization loop shifts mantissa left and decrements exponent until exp < 0x80, which destroys the mantissa for exp >= 0x80 inputs.

**KEY QUESTION**: What happens when the ORIGINAL values (12 with exp=0x81, 8 with exp=0x80) enter 0x07CA48 WITHOUT the +2 exponent bump? Does exp=0x81 also get over-shifted? Does exp=0x80 survive?

## Task
Build a probe file `TI-84_Plus_CE/probe-phase160-norm-inputs.mjs` that:

1. **Setup**: Load the transpiled JS, create executor with `timerInterrupt: false`. Cold boot + MEM_INIT as in other probes.

2. **Part A: Call 0x07CA48 with OP1=12.0 (exp=0x81, mantissa=[12 00 00 00 00 00 00])**
   - Set OP1 at 0xD005F8 = [0x00, 0x81, 0x12, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]
   - Set OP2 at 0xD00603 = [0x00, 0x81, 0x12, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00] (same)
   - Call 0x07CA48, run up to 500 steps
   - Log OP1 and OP2 after return
   - Count how many times Shl14 (0x07FB33) was hit

3. **Part B: Call 0x07CA48 with OP1=8.0 (exp=0x80, mantissa=[80 00 00 00 00 00 00])**
   - Wait, 8.0 in TI BCD is: type=0x00, exp=0x80, mantissa=[80 00 00 00 00 00 00]
   - Set OP1 = [0x00, 0x80, 0x80, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]
   - Set OP2 same
   - Call 0x07CA48, run up to 500 steps
   - Log OP1/OP2 after return

4. **Part C: Call 0x07CA48 with OP1=1200 (exp=0x83, mantissa=[12 00 00 00 00 00 00])**
   - OP1 = [0x00, 0x83, 0x12, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]
   - This is the ACTUAL gcd input after exponent bump
   - Should reproduce the known mantissa destruction

5. **Part D: Call 0x07CA48 with OP1=1.0 (exp=0x80, mantissa=[10 00 00 00 00 00 00])**
   - OP1 = [0x00, 0x80, 0x10, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]
   - Session 158 showed this gets 1 shift and zeroed — verify

For each test, log:
- Input OP1/OP2 (hex bytes)
- Steps taken
- Output OP1/OP2 (hex bytes)
- Number of Shl14 hits (block 0x07FB33)
- The exponent transition (initial → final)

## Key addresses
- OP1 = 0xD005F8 (9 bytes)
- OP2 = 0xD00603 (9 bytes)
- 0x07CA48 = normalization function entry
- 0x07FB33 = Shl14 (mantissa left-shift)
- 0x07FDF1 = exponent decrement
- 0x07C9AF = loop exit (LoopExit)
- 0x07FAC2 = BigShift (exponent reset)
- FAKE_RET sentinel = 0x7FFFFE
