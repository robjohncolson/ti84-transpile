# Auto-Session 110 P2: pTempCnt Audit + Descending-Boundary Invariant

## Context
Research refinement R1 asks: does our MEM_INIT seed properly zero pTempCnt at 0xD02596? Research refinement R3 asks: do our probes maintain the descending-boundary invariant `tempMem ≤ FPSbase ≤ FPS ≤ OPS ≤ OPBase ≤ pTemp ≤ progPtr ≤ top-of-user-memory`?

pTempCnt address: 0xD02596 (3 bytes, per ti84pceg.inc; but reports show 4 bytes at this address — use 3-byte read24 like other pointers).

## Task
Create probe `TI-84_Plus_CE/probe-phase25ay-ptcnt-invariant.mjs` that:

### Part 1: pTempCnt audit
1. Run MEM_INIT (entry 0x09DEE0) on fresh memory
2. After MEM_INIT returns, read pTempCnt at 0xD02596 and report its value
3. Confirm it's 0x000000

### Part 2: Descending-boundary invariant after MEM_INIT
After MEM_INIT returns, read all allocator pointers and verify:
```
tempMem(0xD02587) ≤ FPSbase(0xD0258A) ≤ FPS(0xD0258D) ≤ OPS(0xD02593) ≤ OPBase(0xD02590) ≤ pTemp(0xD0259A) ≤ progPtr(0xD0259D)
```
NOTE: The WikiTI ordering is `FPS ≤ OPS ≤ OPBase`. Check if this actually holds — FPS/FPSbase/tempMem should all be at 0xD1A881 (userMem), while OPS/OPBase/pTemp/progPtr should all be at 0xD3FFFF (top of user memory). So the invariant should be 0xD1A881 ≤ 0xD1A881 ≤ 0xD1A881 ≤ 0xD3FFFF ≤ 0xD3FFFF ≤ 0xD3FFFF ≤ 0xD3FFFF.

### Part 3: Invariant after CreateReal + ParseInp
Run the StoAns pipeline: MEM_INIT → CreateReal("Ans") → ParseInp("2+3"). After each stage, dump all pointers and check the invariant. Report any violations.

### Part 4: Invariant after multi-expression
Run 3 independent fresh-runtime evaluations: "2+3", "3*4", "9-1". After each ParseInp returns, dump pointers and check the invariant. This is NOT a round-trip test — just checking pointer sanity.
