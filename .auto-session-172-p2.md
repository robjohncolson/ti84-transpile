# Task: Build standalone eZ80 interpreter and compare gcd(12,8) execution against transpiled

## Context
We are transpiling the TI-84 Plus CE ROM (eZ80 CPU) into JavaScript. The transpiled gcd(12,8) produces E_Domain error instead of the correct result (4). Session 171's block-by-block comparison proved 1421/1423 transpiled blocks are correct — but it used the SAME cpu-runtime.js methods for both sides, so it can't find bugs IN those methods.

**We need an independent eZ80 interpreter that does NOT use cpu-runtime.js to serve as ground truth.**

## Your task
Build a standalone eZ80 interpreter in `TI-84_Plus_CE/probe-phase172-standalone-interp.mjs` that:
1. Implements ALL 48 instruction types found in the gcd path (listed below)
2. Reads instruction bytes directly from the ROM
3. Executes the gcd(12,8) computation step by step
4. At each block boundary, compares ALL register state (A, F, B, C, D, E, H, L, IX, IY, SP, PC) and key memory (OP1-OP6 at 0xD005F8-0xD0063A) against the transpiled execution
5. Reports the FIRST divergence with full register dumps from both sides

## The 48 instruction types from the opcode census
(from probe-phase171-opcode-census.mjs output, session 171)

call, ld-pair-imm, alu-reg, alu-imm, dec-pair, jr, ret, ld-reg-reg, ld-reg-mem, ld-mem-reg, push, pop, inc-pair, jp, ld-pair-mem, ld-mem-pair, bit-op, rld, ldi, djnz, daa, inc-reg, dec-reg, ld-special, ex, cp, set-bit, res-bit, rst, add-pair, sbc-pair, neg, rotate-a, rotate-shift, ld-reg-imm, out0, in0, ret-cc, call-cc, jp-cc, jr-cc, or-imm, and-imm, xor-imm, sub-imm, cp-imm, add-imm, adc-imm, srl

## Implementation approach
The interpreter must be INDEPENDENT of cpu-runtime.js. Write flag computation from scratch based on eZ80 spec:

### Flag definitions
- C = 0x01 (carry)
- N = 0x02 (subtract)
- PV = 0x04 (parity/overflow)
- X = 0x08 (undocumented bit 3)
- H = 0x10 (half-carry)
- Y = 0x20 (undocumented bit 5)
- Z = 0x40 (zero)
- S = 0x80 (sign)

### Key implementation notes
- eZ80 ADL mode: registers are 24-bit (HL, BC, DE, IX, IY, SP)
- For 16-bit arithmetic (SBC HL, ADC HL, ADD HL): in ADL mode, operate on full 24 bits. Carry from bit 23, half-carry from bit 19 (or 11? — check the eZ80 manual)
- RLD: A low nibble → (HL) low nibble, (HL) high nibble → A low nibble, (HL) low nibble → (HL) high nibble. S,Z from A, H=0, PV=parity(A), N=0, C preserved.
- DAA: BCD adjust after add/sub. Critical for gcd (17 occurrences). Must correctly handle N flag state.
- LDI: LD (DE),(HL); INC HL; INC DE; DEC BC. PV set if BC≠0 after dec. H=0, N=0.

### Execution model
1. Load the ROM from `TI-84_Plus_CE/ROM.rom` (4MB)
2. Create a 16MB memory array, load ROM at 0x000000-0x3FFFFF
3. Set up initial state for gcd(12,8): 
   - Load OP1 with 12.0 BCD format: bytes [0x00, 0x81, 0x12, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00] at 0xD005F8
   - Load OP2 with 8.0 BCD format: bytes [0x00, 0x80, 0x80, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00] at 0xD00603
   - SP = 0xD1A86F
   - PC = 0x068D3D (gcd entry)
   - ADL mode = 1
4. Execute by decoding instructions from ROM bytes using `TI-84_Plus_CE/ez80-decoder.js`
5. At each block boundary (after RET, JP, JR, CALL, DJNZ, conditional jumps), log full state

### Comparison side
Also run the same gcd(12,8) through the transpiled code (import from ROM.transpiled.js BLOCKS, use cpu-runtime.js CPU). At each block boundary, compare:
- All 8-bit registers: A, F, B, C, D, E, H, L
- All pair registers: BC, DE, HL, IX, IY, SP
- Memory: OP1 (9 bytes at 0xD005F8), OP2 (9 bytes at 0xD00603)
- FPS pointer at 0xD0117C (2 bytes)

### What a divergence looks like
```
DIVERGENCE at step 42, PC=0x07CA48:
  Standalone: A=0x12, F=0x40, HL=0xD005FA, ...
  Transpiled: A=0x12, F=0x44, HL=0xD005FA, ...
  Diff: F (standalone=0x40, transpiled=0x44, bit diff: PV)
```
