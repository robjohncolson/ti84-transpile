# Session 128 P2: Integrate post-ParseInp allocator re-seeding into browser-shell common-tail path

## Context
The TI-84 CE ROM transpilation project has a browser shell (`TI-84_Plus_CE/browser-shell.html`) with a direct-eval system that evaluates math expressions through the real ROM's ParseInp function. Currently, direct-eval calls ParseInp at 0x099914 directly. Session 127 proved that re-seeding OPBase/OPS/FPS to post-MEM_INIT values after ParseInp allows the full OS common-tail cleanup to complete (post-ParseInp call at 0x05822A returns to FAKE_RET in just 2 steps).

**Goal**: Modify browser-shell.html's direct-eval to use the common-tail entry point (0x0586CE) instead of direct ParseInp (0x099914), with automatic OPBase/OPS/FPS re-seeding after ParseInp returns. This exercises more of the real OS path and validates the full round-trip.

## The validated pattern (from probe-phase127-post-parseinp.mjs)

Read `TI-84_Plus_CE/probe-phase127-post-parseinp.mjs` for the exact implementation. The key steps are:

1. Set up runtime as usual (cold boot → MEM_INIT → CreateReal("Ans"))
2. Seed begPC/curPC/endPC to token buffer, place error frame, etc. (same as current direct-eval)
3. Instead of entering at ParseInp (0x099914), enter at the **common-tail** address 0x0586CE
4. The common-tail calls 0x0A1FD1 (LCD-related), 0x0A27DD, then reaches ParseInp at 0x099910→0x099914
5. After ParseInp returns with OP1=result, execution continues to post-ParseInp cleanup calls
6. **BEFORE the cleanup calls execute**, re-seed these pointers to post-MEM_INIT values:
   - OPBase (0xD0258D) = 0xD3FFFF
   - OPS (0xD02593) = 0xD3FFFF
   - FPS (0xD0259F) = value of FPSbase (0xD0259C) — typically 0xD1A88A after CreateReal
   - pTemp (0xD0258A) = 0xD3FFFF
   - progPtr (0xD025A3) = 0xD3FFFF
7. The cleanup at 0x05822A then completes in 2 steps (it's a missing block → falls to FAKE_RET)

## Implementation plan

In `browser-shell.html`, find the `runDirectEval()` function (around the direct-eval section, ~line 1700+). Currently it:
- Calls `evalEnsureRuntime()` to get cached mem/ex/cpu
- Resets OPS/FPS/FPSbase
- Sets begPC/curPC/endPC to token buffer
- Places error frame + FAKE_RET sentinel
- Runs executor from 0x099914 (ParseInp)
- Reads OP1 result

**Modify to**:
1. Add a boolean option or config: `USE_COMMON_TAIL = true` (default true, with a way to fall back to direct ParseInp if issues arise)
2. When USE_COMMON_TAIL is true:
   - Enter at 0x0586CE instead of 0x099914
   - Use a higher step budget (common-tail does ~136 extra steps before ParseInp + cleanup after)
   - After execution, the result should be in OP1 as usual
3. The re-seeding happens automatically because the cleanup calls after ParseInp will encounter missing blocks and fall to FAKE_RET. But to be safe, check if OP1 has the expected result. If not, fall back to direct ParseInp.

**IMPORTANT**: The common-tail path calls LCD-related functions (0x0A1FD1, 0x0A27DD) before ParseInp. These may write to VRAM or call other OS routines. Make sure `timerInterrupt: false` is set (already should be).

Actually, re-reading probe-phase127-post-parseinp.mjs more carefully — the re-seeding needs to happen BETWEEN ParseInp returning and the cleanup running. In the probe, this was done by:
1. Running executor from the common-tail entry until ParseInp finishes (detect when PC returns to post-ParseInp area)
2. Then re-seeding pointers
3. Then continuing execution from the post-ParseInp address

For the browser-shell, the simplest approach may be:
- Option A: Run from 0x0586CE with a step limit that's large enough for the pre-ParseInp path + ParseInp + a few cleanup steps. The cleanup at 0x05822A hits a missing block and falls to FAKE_RET, which IS the expected behavior — no re-seeding needed because the missing block short-circuits the cleanup.
- Option B: Keep using direct ParseInp (0x099914) as-is, since it works perfectly and the common-tail adds complexity without functional benefit for expression evaluation.

**Choose Option A** — run from 0x0586CE and let the missing block at 0x05822A naturally short-circuit cleanup to FAKE_RET. This is simpler and validated by the probe.
