# Session 148 P2: Graph — Disassemble IPoint's actual coordinate input mechanism

## Context
This is a TI-84 Plus CE ROM transpilation project. We lift eZ80 machine code to JavaScript.

IPoint (0x07B451) successfully writes pixels to LCD VRAM, but ALL pixels land on **row 239** regardless of the Y value passed via GraphPars. Session 147 proved that curGY (0xD022D1) and curGX (0xD022CF) have **NO effect** — IPoint ignores both.

We need to find WHERE IPoint actually gets its X and Y pixel coordinates.

Key facts from prior sessions:
- IPoint entry: 0x07B451
- VRAM address computed at 0x07B580-0x07B5A3: `A = 0xEF - C` (239 - pixelY), `MLT HL`, etc.
- Bounds check subroutine at 0x07B793
- ILine (0x07B245) saves BC/DE to 0xD022D1/D2 before calling IPoint
- pixWideP at 0xD014FE, pixWide_m_2 at 0xD01501
- Pen color at 0xD026AE
- IY+43 and IY+74 control different pixel write paths

## Task

### Build probe `probe-phase148-ipoint-coords.mjs`

Use `probe-phase147-graph-curgy.mjs` as your template (same imports, createRuntime, coldBoot, prepareCallState, seedAllocator, runMemInit, seedErrFrame patterns).

The probe should have these parts:

**Part A — Disassemble IPoint entry to VRAM computation**:
Read ROM bytes from 0x07B451 to 0x07B5B0 and print a human-readable disassembly. Focus on:
- What registers are read at entry?
- Where does pixelY (the C register used in `A = 0xEF - C`) come from?
- Where does the X coordinate feed into the VRAM offset?
- Are there any `LD C,(nn)` or `LD C,A` or similar instructions that load C from RAM before the VRAM computation?

**Part B — Register-input test**:
Call IPoint directly (0x07B451) with varying BC and DE register values to see which controls pixel position:
1. Test with BC=0x0050 (low byte C=0x50=80 decimal), DE=0x00A0: check VRAM write address
2. Test with BC=0x0014 (C=20), DE=0x00A0: check VRAM write address
3. Test with BC=0x0050, DE=0x0032 (E=50): check VRAM write address
4. Test with BC=0x0050, DE=0x00C8 (E=200): check VRAM write address

For each test:
- Create fresh runtime, cold boot, MEM_INIT
- Seed allocator, pixel dims (pixWideP=320 at 0xD014FE, pixWide_m_2=238 at 0xD01501)
- Set pen color 0x10 at 0xD026AE
- Set draw mode A=1 (normal draw)
- SET bit 2 of IY+43 (0xD000AB) for 24-bit bounds
- CLEAR bit 2 of IY+74 (0xD000CA) for plotSScreen path
- Clear bit 7 of 0xD000B5, clear bit 5 of IY+14 (0xD00094)
- Set BC and DE to test values
- Push FAKE_RET as return address
- Execute from 0x07B451 with maxSteps=200
- Read the VRAM framebuffer address from 0xD02A8A (3 bytes)
- Scan LCD VRAM (0xD40000-0xD65800) for non-zero bytes that appeared during execution
- Print: test params, VRAM addr at 0xD02A8A, any changed LCD bytes with addresses

**Part C — Trace register flow inside IPoint**:
Pick one test case (BC=0x0050, DE=0x00A0). Run IPoint with `onBlock` callback that logs:
- PC, A, B, C, D, E, H, L registers at each block entry
- Stop after 100 blocks max

Print the trace so we can see how registers transform from entry to the VRAM address computation at 0x07B580.

**Part D — ROM disassembly of 0x07B793 (bounds check)**:
Read ROM bytes from 0x07B793 to 0x07B7E0 and print annotated disassembly. This subroutine was previously thought to read curGY — now we need to see what it ACTUALLY does. Focus on any LD from RAM addresses and what it returns in registers.

Print all results clearly. The probe must be runnable as `node TI-84_Plus_CE/probe-phase148-ipoint-coords.mjs`.
