# Auto-Session 165 P3: Test if compound function 0x07C747 is a NO-OP when OP1≠OP2

## Context

You are working on a TI-84 Plus CE ROM transpilation project at `C:/Users/rober/Downloads/Projects/school/ti84-transpile`. The project lifts eZ80 machine code into JavaScript.

We've been investigating a gcd bug. The compound function at 0x07C747 does:
1. CALL 0x07F8FA — copy OP1 into OP2
2. CALL 0x07CA48 — normalize OP1 (shifts mantissa left, zeros it for exp >= 0x80)
3. CALL 0x07CA06 — InvOP1S (negate OP1 sign)
4. JR 0x07C77F — FPAdd (OP1 + OP2)

When OP1==OP2 at entry (which happens in gcd because OP1→OP2 was just copied), the result is:
- After step 1: OP1=OP2=X
- After step 2: OP1=0, OP2=X (normalize zeros OP1)
- After step 3: OP1=-0, OP2=X
- After step 4: FPAdd(-0, X) = X → stored in OP1

So it's a NO-OP: OP1 goes in as X, comes out as X. But is this the INTENDED behavior? Session 164 hypothesized: maybe the REAL purpose of 0x07C747 is not the OP1→OP2 copy but some SIDE EFFECT on OP2 (normalization of the exponent/mantissa).
