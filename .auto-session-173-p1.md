# Task: Fix z80-mode SP truncation in cpu-runtime.js

## Context
File: `TI-84_Plus_CE/cpu-runtime.js`
The TI-84 Plus CE uses an eZ80 CPU. The eZ80 has a 24-bit stack pointer (SPL) that is ALWAYS 24 bits wide, even in z80 compatibility mode. In z80 mode, PUSH/POP transfer 16-bit values but the SP itself is still incremented/decremented as a 24-bit value.

## The Bug
Lines 717-735 of `cpu-runtime.js`:

```javascript
push(value) {
    if (this.madl) {
      this.sp = (this.sp - 3) & 0xffffff;
      this.write24(this.sp, value);
    } else {
      this.sp = (this.sp - 2) & 0xffff;   // BUG: truncates SP to 16 bits
      this.write16(this.sp, value);
    }
  }

  pop() {
    if (this.madl) {
      const value = this.read24(this.sp);
      this.sp = (this.sp + 3) & 0xffffff;
      return value;
    }
    const value = this.read16(this.sp);
    this.sp = (this.sp + 2) & 0xffff;   // BUG: truncates SP to 16 bits
    return value;
  }
```

When `this.madl = 0` (z80 mode), the code masks SP with `& 0xffff`. This loses the high byte. Example: SP = 0xD1A860, after z80-mode POP → SP becomes 0xA862 instead of 0xD1A862.

## The Fix
In z80 mode on the eZ80, SP is still 24 bits wide. The data width is 16 bits (push/pop 2 bytes), but the pointer itself wraps at 24 bits. Change:
- Line 722: `this.sp = (this.sp - 2) & 0xffff;` → `this.sp = (this.sp - 2) & 0xffffff;`
- Line 734: `this.sp = (this.sp + 2) & 0xffff;` → `this.sp = (this.sp + 2) & 0xffffff;`

## Deliverables
1. Edit `TI-84_Plus_CE/cpu-runtime.js` lines 722 and 734: change `& 0xffff` to `& 0xffffff` in the z80-mode branches of `push()` and `pop()`.

## Constraints

- Do NOT change the data width (still write16/read16 in z80 mode — that's correct)
- Do NOT add comments beyond what's already there

## Success Criterion
After the fix, `push()` in z80 mode does `this.sp = (this.sp - 2) & 0xffffff` and `pop()` does `this.sp = (this.sp + 2) & 0xffffff`. The read16/write16 calls remain unchanged.
