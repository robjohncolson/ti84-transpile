# Task: CreateReal entry sequence — disassemble 0x082380-0x08239F

## Context
The CreateReal JT slot at 0x020534 has impl at 0x08238A. Session 75 disassembled starting at 0x08238A and found it reads OP1+1 (0xD005F9) as a type byte, comparing against 0x5D/0x24/0x3A/0x72. But the JT entry actually starts with:
```
xor a        ; A = 0
ld hl, 0x000009
jr 0x082386  ; jumps BACKWARDS before 0x08238A
```

So the real entry sequence starts around 0x082380-0x082386. We need to understand what 0x082386 does with A=0 and HL=9 BEFORE reaching the type-check code at 0x08239c.

Also: 0xD00619 is read after the type check — we need to identify what this address holds.

1. `TI-84_Plus_CE/probe-phase25q-createreal-entry.mjs` — static disassembly probe

## What the probe must do

Use the static disassembly pattern from `TI-84_Plus_CE/probe-phase25p-vat-disasm.mjs` as template. Import `decodeInstruction` from `./ez80-decoder.js`, read `ROM.rom`.

Disassemble these ranges (all in ADL mode):
- `0x082370` — 128 bytes (captures 0x082386 entry and the type-check code through ~0x0823F0)
- `0x08238A` — 64 bytes (the impl address, overlaps for cross-reference)
- `0x082340` — 48 bytes (context before the entry, may show other CreateXxx entries)

For each range:
1. Decode every instruction using `decodeInstruction(romBytes, pc, 'adl')`

3. Track all CALL/JP/JR targets (especially where 0x082386 jumps/falls through)
4. Track all RAM reads/writes in 0xD00000-0xD1FFFF range
5. Pay special attention to reads of 0xD00619 — annotate what this might be
6. Cross-reference any IY-relative accesses with IY=0xD00080

The report must answer:
- What does the code at 0x082386 do with A=0 (from `xor a`) and HL=9?

## Reference: ti84pceg.inc OP register addresses

- OP3 = 0xD0060A  
- OP4 = 0xD00613
