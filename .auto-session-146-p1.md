# Session 146 P1 — FP gcd: disassemble 0x07FA74 routine + investigate FpPop copy direction

## Context

This is a TI-84 Plus CE ROM transpilation project. We lift eZ80 machine code from a 4MB ROM into JavaScript and run it in a browser. The ROM file is at `TI-84_Plus_CE/ROM.rom`.

**The gcd(12,8) function keeps hitting E_Domain (error 0x84).** Session 145 traced OP2 corruption to PC=0x07FA86 — a routine at 0x07FA74 overwrites OP2 with 1.0 before the type validator at step 21. We need to understand what this routine does and why it's called during gcd.

Also: FpPop (0x082957) behaves unexpectedly in direct-call testing — FPS goes UP by 9 bytes (should go down for a pop) and neither OP1 nor OP2 is written. The helper at 0x082BB5 likely controls copy direction and needs specific register state.

## Task

Build a new probe: `TI-84_Plus_CE/probe-phase146-gcd-fp-routines.mjs`

### Part A: Disassemble 0x07FA74–0x07FAC0

Read raw ROM bytes from `ROM.rom` at offset 0x07FA74 through 0x07FAC0 (~76 bytes). Manually decode the eZ80 instructions and print them. The eZ80 instruction set is similar to Z80 but with 24-bit addressing. Key patterns:
- `21 xx yy zz` = LD HL, imm24
- `ED 53 xx yy zz` = LD (nn), DE
- `ED 5B xx yy zz` = LD DE, (nn)
- `CD xx yy zz` = CALL imm24
- `C9` = RET
- `3E xx` = LD A, imm8
- `32 xx yy zz` = LD (nn), A
- `AF` = XOR A
- `CB xx` = bit operations
- `36 xx` = LD (HL), imm8
- `23` = INC HL
- `FD CB dd xx` = IY+d bit ops

Focus on: what does this routine write to OP2 (0xD00603)? Is it loading a constant 1.0? What registers does it expect on entry?

### Part B: Trace gcd calling convention around 0x07FA74

Using the existing probe pattern from `probe-phase145-fppush-disasm.mjs`:
1. Set up gcd(12,8): MEM_INIT, seed allocator, seed FPS with 12.0 and 8.0 (9 bytes each, FPS at clean area 0xD1AA00, FPSbase=0xD1AA00, FPS=0xD1AA12).
2. Call gcd via 0x068D3D (direct handler)
3. Use `onBlock` to snapshot ALL registers (A, F, HL, DE, BC, SP, IX, IY) at PC=0x07FA74 (entry) and PC=0x07FA86 (the OP2 overwrite point)
4. Print register state at both points — this tells us what calling convention 0x07FA74 uses

### Part C: FpPop with register pre-seeding

1. Seed FPS with a single 8.0 value (9 bytes). FPSbase=0xD1AA00, FPS=0xD1AA09 (base+9, pointing past 1 entry).
2. Snapshot registers from Part B's gcd trace at the point where gcd calls FpPop
3. Pre-load those same register values before calling FpPop directly at 0x082957
4. Check: does FPS now go DOWN? Does OP2 get written?
5. Compare with the vanilla FpPop call (zero registers) from session 145
