# Task: FP gcd sign corruption — step-by-step OP1/OP2 snapshot probe

## Context
This is a TI-84 Plus CE ROM transpilation project. The ROM is lifted to JavaScript. We run the lifted code via `createExecutor()` from `cpu-runtime.js`.

The gcd(12,8) function runs via direct entry at 0x068D3D for 1,442 steps. The result should be 4.0 in OP1, but instead we get OP1/OP2 = [80 82 10...] = -1.0 (WRONG). The type byte (first byte of OP1/OP2) flips to 0x80 at step ~1440, and the type validator at 0x07F831 rejects it as E_Domain (error 0x84).

**Key known facts from session 148:**
- All blocks are present (232 unique PCs, no missing blocks)
- Real pop at 0x0828FC IS reached (step 1420)
- FPS-=9 helper at 0x082912 fires at steps 1422 and 1440
- The 1.0 constant loader at 0x07FA74 writes `[00 80 10 00 00 00 00 00 00]` to OP2 — called twice (steps 18, 737)
- Type byte 0x00 gets corrupted to 0x80 somewhere between writes
- BCD format: byte 0 = type (0x00=Real, 0x80=negative?), byte 1 = exponent (0x80=10^0), bytes 2-8 = mantissa

## Your task
Build `TI-84_Plus_CE/probe-phase149-gcd-sign-trace.mjs` that captures OP1 and OP2 snapshots at EVERY block execution during gcd(12,8) via 0x068D3D.

### Deliverables
1. A probe file at `TI-84_Plus_CE/probe-phase149-gcd-sign-trace.mjs`
2. The probe must:
   a. Use the same boilerplate pattern as `TI-84_Plus_CE/probe-phase148-gcd-realpop.mjs` (read that file for the exact import/setup pattern)
   b. Run MEM_INIT first (entry 0x09DEE0, return sentinel 0x7FFFF6, budget 100K steps)
   c. Seed the allocator (FPSbase/FPS = 0xD1AA00, OPBase/OPS/pTemp/progPtr = 0xD3FFFF, newDataPtr = 0xD1A881)
   d. Seed FPS with 12.0 at offset 0 and 8.0 at offset 9 (9-byte entries), set FPS = FPSbase + 18
   e. Write 12.0 to OP1 (0xD005F8) and 8.0 to OP2 (0xD00603)
   f. Set errSP (0xD008E0) to point to an error-catch sentinel
   g. Use `onBlock` callback to capture OP1[0..8] and OP2[0..8] at EVERY step
   h. Store results as an array of `{ step, pc, op1: [9 bytes hex], op2: [9 bytes hex] }`
   i. After execution, find ALL transitions where OP1[0] or OP2[0] (the type byte) changes value
   j. Print those transitions with step number, PC, and before/after values
   k. Also specifically check: at the type validator (0x07F831), what are OP1[0] and OP2[0]?
   l. Check if type byte 0x80 means "negative Real" (sign bit) vs "invalid type" — print what the validator does with it
