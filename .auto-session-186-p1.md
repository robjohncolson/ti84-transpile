# Auto-Session 186 — P1: Trace JError CALL chain SP values for ParseInp fix

## Context
The TI-84 Plus CE ROM transpilation project has a bug where ParseInp (the OS expression parser) hits E_Undefined (errNo=0x8D) and the error handler dispatch chain jumps to erased flash at 0x1BC300. Session 185 proved that a simple popReturn monkey-patch is insufficient because JError's `LD SP,(errSP)` at 0x061DCA changes SP to the PushErrorHandler frame, so by the time the CALL to 0x03E187 (RSMIX helper) executes, SP points to stale data.

## Task
Build a Node.js probe script `TI-84_Plus_CE/probe-phase186-jerror-sp-trace.mjs` that traces the exact SP register value at every step through the JError error dispatch chain, specifically:

1. Boot the ROM with timer interrupts disabled
2. Set up minimal OS state (memInit, etc.) similar to `probe-phase185-madl-intercept.mjs`
3. Set errNo (0xD008AF) = 0x8D (E_Undefined)
4. Call JError at 0x061DB2
5. **At every step**, log: `{ step, pc (hex), sp (hex), opcode_bytes (first 4), instruction_tag, madl_mode }`
6. Pay special attention to:
   - The `LD SP,(errSP)` at 0x061DCA — log SP before and after
   - The CALL 0x03E1B4 — log the return address pushed and new SP
   - The CALL 0x03E187 (inside 0x03E1B4 block) — log return address pushed and new SP
   - The RSMIX at 0x03E191 — log mode switch
   - Any POP/RET instructions — log what values are popped and from what addresses
7. Run for up to 200 steps or until HALT
8. Output structured JSON with the full trace

## Key addresses
- JError: 0x061DB2
- errSP: 0xD008A1 (24-bit pointer stored here by PushErrorHandler)
- errNo: 0xD008AF
- RSMIX helper: 0x03E187
- Wrapper: 0x03E1B4
- z80-mode RET that's the problem: 0x03E1B1-0x03E1B3
- PushErrorHandler error frame target: 0x099929 (hlPayload)
