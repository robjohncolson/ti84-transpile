# Task: Re-seed allocator pointers AFTER compaction in the NZ path

## Context
This is a TI-84 Plus CE ROM transpilation project. We lift eZ80 machine code to JavaScript and run it in Node.js.

The ENTER key handler's NZ path (POP AF at 0x05862F restores NZ → JR NZ 0x05866A) is the "has existing expression" path. It calls 0x07FF7B and 0x08384F which trigger the 0x0827xx allocator loop. Session 104 found that pointer zeroing happens at step ~390 via LDDR compaction at 0x0831A4 (called through 0x05E836→0x0831A4→0x083214→0x04C990). This zeroes OPBase, OPS, pTemp, progPtr — destroying the seeded state.

The hypothesis: if we intercept AFTER compaction zeros the pointers and RE-SEED them, the allocator loop should terminate and execution should reach the common tail at 0x058693, then ParseInp at 0x0586E3.

## What to build
Create a probe file: `TI-84_Plus_CE/probe-phase25at-reseed-after-compaction.mjs`

The probe should:

1. Set up the ENTER handler environment:
   - Full MEM_INIT via JT slot at 0x020164
   - Seed allocator pointers: OPBase(0xD02590)=0xD3FFFF, OPS(0xD02593)=0xD3FFFF, pTemp(0xD0259A)=0xD3FFFF, progPtr(0xD0259D)=0xD3FFFF
   - Seed FPSbase(0xD0258A) and FPS(0xD0258D) to 0xD1A881
   - Seed tokenized "2+3" at userMem 0xD1A881: bytes [0x72, 0x70, 0x71, 0x3F]
   - Seed begPC(0xD02317)=0xD1A881, curPC(0xD0231A)=0xD1A881, endPC(0xD0231D)=0xD1A885
   - Set IY=0xD00080, SP=0xD1A860
   - Push FAKE_RET sentinel onto stack
   - Set up error frame (PushErrorHandler pattern from reference probes)
   - Seed cxMain(0xD007CE) = some valid address (or leave as-is after MEM_INIT)
   - Force NZ flags for POP AF: on the stack where POP AF at 0x05862F will read, ensure the flags byte has Z=0 (bit 6 of F clear). Actually, POP AF pops AF from stack — the probe needs to push a flags value with Z=0 before starting. See reference probes for how this is handled.

2. Run from PC=0x0585E9 (ENTER handler entry) with step-by-step execution (NOT cpu.run with a budget — use a loop that calls cpu.step() one at a time)

3. After EACH step, check if OPBase (0xD02590) has been zeroed. When it transitions from non-zero to 0x000000:
   - Log the step number and PC
   - RE-SEED all allocator pointers back to their original values:
     - OPBase(0xD02590) = 0xD3FFFF
     - OPS(0xD02593) = 0xD3FFFF  
     - pTemp(0xD0259A) = 0xD3FFFF
     - progPtr(0xD0259D) = 0xD3FFFF
     - FPSbase(0xD0258A) = 0xD1A881
     - FPS(0xD0258D) = 0xD1A881
   - Continue execution

4. Use a 200K step budget total

5. After execution completes (either hits FAKE_RET or exhausts budget):
   - Report: Did execution reach 0x058693 (common tail)?
   - Report: Did execution reach 0x0586E3 (ParseInp call site)?
   - Report: Did execution hit 0x099914 (ParseInp)?
   - Report: What's OP1 (0xD005F8, 9 bytes)?
   - Report: What's errNo (0xD008AF)?
   - Report: How many steps?
   - Report: How many times were pointers re-seeded?
   - Report: Top 20 most-hit PCs

## Handling the NZ flag issue
The ENTER handler starts at 0x0585E9. The sequence is:
- 0x0585E9: various setup calls (0x058D54, 0x058BA3, etc.)
- 0x05861D: CALL 0x058212 
- 0x058621: PUSH AF (saves Z flag from 0x058212)
- 0x058626: CALL 0x099211 (DisarmScroll)
- 0x05862A: EI
- 0x05862B: CALL 0x0921CB (history manager — returns immediately since numLastEntries=0)
- 0x05862F: POP AF (restores saved flags)
- 0x058630: JR NZ 0x05866A (if NZ, take the "has expression" path)

To force NZ: bit 5 of (IY+0x44) = mem[0xD000C4] must be SET (=1) before the handler runs. 0x058212 calls 0x0800B8 which does `BIT 5,(IY+0x44)`. If bit 5 is 1, BIT instruction clears Z → NZ → JR NZ taken.

So: **set mem[0xD000C4] |= 0x20** (bit 5) before starting the probe.
