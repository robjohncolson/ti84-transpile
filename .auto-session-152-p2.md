# Task: IPoint pen color — disassemble 0x07B5B6 + test drawFGColor seeding

## Context
This is a TI-84 Plus CE ROM transpilation project. The transpiler lifts eZ80 ROM code into runnable JavaScript. We're debugging why IPoint writes drawMode (0x01) as the first pixel byte instead of pen color.

## Problem
Session 151 traced the A-register through IPoint and found:
- Step 6 (PC=0x07B793): A = pen color 0x10, loaded via `LD A,(0xD026AE)`
- Step 17 (PC=0x07B556): A still = 0x10
- Step 30 (PC=0x07B5B6): A overwritten to 0x01 (drawMode from 0xD02AC8)
- Step 36 (PC=0x07B682): `LD (BC),A` writes 0x01 to VRAM (first byte)
- Step 37 (PC=0x07B68A): Second byte written from DE register = drawFGColor (0xD026AC)

The question: why does A get overwritten with drawMode at 0x07B5B6? Is this intentional (16bpp uses drawMode+drawFGColor as two-byte pixel value) or a bug (missing block that should reload pen color)?

## What to build
Create `TI-84_Plus_CE/probe-phase152-ipoint-color-fix.mjs` with THREE parts:

### Part A: Disassemble blocks 0x07B5B6 through 0x07B682
Read the ROM binary (TI-84_Plus_CE/ROM.rom) and disassemble the blocks between 0x07B5B6 and 0x07B682 to understand the instruction sequence that loads drawMode into A.

Key question: Is there a CALL or JP between 0x07B5B6 and 0x07B682 that should go to a block which reloads pen color? Or is the `LD A,(0xD02AC8)` intentional because drawMode is used as the low byte of a 16bpp pixel?

Use the transpiled blocks to identify block boundaries: check which blocks exist in the range. Read `blocks` from ROM.transpiled.js and list all block start addresses between 0x07B500 and 0x07B700.

For each block in that range, log: start address, first few instructions (from the ROM bytes), and any CALL/JP targets.

### Part B: Test drawFGColor seeding with correct BGR565 color
Call IPoint at 0x07B451 with:
- drawMode = 1 at 0xD02AC8
- drawFGColor = 0x001F at 0xD026AC (2 bytes, little-endian: mem[0xD026AC]=0x1F, mem[0xD026AD]=0x00) — this is blue in BGR565
- drawColorCode = 0x10 at 0xD026AE (pen color byte, for reference)
- IY+2B bit 2 SET (for 16bpp VRAM path)
- Standard pixel setup: C=pixelY, DE=pixelX (e.g., Y=120, X=160)
- VRAM pre-filled with 0xAA sentinel

Check VRAM at the computed address. For Y=120, X=160: VRAM addr = 0xD40000 + (120 * 320 * 2) + (160 * 2) = 0xD40000 + 76800 + 320 = 0xD52AC0.

Expected: VRAM[0xD52AC0] = 0x01 (drawMode), VRAM[0xD52AC1] = 0x1F (drawFGColor low byte).

If this works, then the "fix" is: users should seed drawFGColor (0xD026AC) with the correct BGR565 value, and the first byte is intentionally drawMode. The pen color at 0xD026AE is only used by the Z-path (plotSScreen, 8bpp).

### Part C: Test multiple colors via drawFGColor
Test 4 colors:
1. Red BGR565: 0xF800 → mem[0xD026AC]=0x00, mem[0xD026AD]=0xF8 → VRAM should be [0x01, 0x00, ...? or maybe [drawMode, low, high]?]
   Wait — 16bpp writes TWO bytes. If `LD (BC),A` writes byte 1, then `LD (BC),A` at 0x07B68A writes byte 2. So we need to check BOTH bytes at the VRAM address.
2. Green BGR565: 0x07E0 → mem[0xD026AC]=0xE0, mem[0xD026AD]=0x07
3. Blue BGR565: 0x001F → mem[0xD026AC]=0x1F, mem[0xD026AD]=0x00
4. White BGR565: 0xFFFF → mem[0xD026AC]=0xFF, mem[0xD026AD]=0xFF

For each color, log: drawFGColor value, VRAM bytes at target address, and what color this represents.

## IPoint calling convention (from sessions 128-151)
```javascript
// Standard IPoint setup:
const IY = 0xD00080;  // IY base address (seeded during cold boot)

// Required RAM seeds:
mem[0xD02AC8] = 1;      // drawMode (1=normal draw)
mem[0xD026AE] = 0x10;   // drawColorCode / pen color (8bpp)
mem[0xD026AC] = 0x1F;   // drawFGColor low byte (16bpp)
mem[0xD026AD] = 0x00;   // drawFGColor high byte (16bpp)
mem[0xD014FE] = 0x40; mem[0xD014FF] = 0x01; mem[0xD01500] = 0x00;  // pixWideP = 320
mem[0xD01501] = 0xEE; mem[0xD01502] = 0x00; mem[0xD01503] = 0x00;  // pixWide_m_2 = 238
mem[0xD000B5] &= ~0x80; // clear bit 7

// IY flags for 16bpp VRAM path:
mem[IY + 0x2B] |= 0x04;  // bit 2 SET → 16bpp LCD VRAM path
mem[IY + 0x3C] &= ~0x01; // bit 0 clear
mem[IY + 0x02] &= ~0x02; // bit 1 clear
mem[IY + 0x4A] &= ~0x04; // bit 2 clear

// VRAM base stored at:
// mem 0xD02A8A (3 bytes, little-endian) = pointer to VRAM row start
// Computed as: 0xD40000 + Y * 640 (320 pixels * 2 bytes each)

// Call: C = pixelY, DE = pixelX
// ex.callBlock(0x07B451, { maxSteps: 100, A: drawMode, C: pixelY, DE: pixelX });
```

## Standard probe pattern
```javascript
import { createExecutor } from './cpu-runtime.js';
import { createPeripheralBus } from './peripherals.js';
const fs = await import('fs');

const ROM_PATH = new URL('./ROM.transpiled.js', import.meta.url).pathname.replace(/^\/([A-Z]:)/, '$1');
const mod = await import(`file://${ROM_PATH}`);
const { blocks, seedAddresses } = mod;

const romPath = new URL('./ROM.rom', import.meta.url).pathname.replace(/^\/([A-Z]:)/, '$1');
const romBuf = fs.readFileSync(romPath);
mem.set(new Uint8Array(romBuf.buffer, romBuf.byteOffset, romBuf.byteLength), 0);

const periph = createPeripheralBus({ timerInterrupt: false });
const FAKE_RET = 0x7FFFFF;
mem[FAKE_RET] = 0xC9;

const ex = createExecutor({ mem, peripherals: periph, blocks, seedAddresses });
// cold boot + MEM_INIT, then test
```
