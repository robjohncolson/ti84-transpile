# Session 144 P1: FP gcd — Fix FPS Entry Size (11 bytes per entry)

## Context
Working in `C:/Users/rober/Downloads/Projects/school/ti84-transpile`.

The TI-84 CE FP stack uses **11-byte entries** (not 9). Session 143 proved that seeding 9-byte FPS entries causes FpPop to read from misaligned offsets, producing garbage OP2 values. The gcd(12,8) computation gets OP2=1.0 instead of 8.0, leading to E_Domain.

Key facts:
- OP registers are 11 bytes each (session 140 confirmed)
- FpPop at 0x082957 likely subtracts 11 from FPS pointer when popping
- Session 134 found LDI chain at 0x07F978 copies 11 bytes for FP stack POP
- FPS base pointer is at RAM address 0xD0258D (3 bytes, little-endian)
- OP1 = 0xD005F8 (11 bytes), OP2 = 0xD00603 (11 bytes)
- errNo = 0xD008DF

## Task

1. **Build `probe-phase144-gcd-11byte.mjs`** based on the pattern in `probe-phase143-gcd-fixed.mjs` (read it first for the existing seeding pattern).

2. **Fix the FPS seeding**: Instead of 9-byte BCD entries, seed 11-byte entries:
   - Each FPS entry = 9 bytes BCD + 2 zero padding bytes = 11 bytes total
   - Entry layout: `[type, exponent, d1, d2, d3, d4, d5, d6, d7, 0x00, 0x00]`
   - BCD for 12.0 = `[0x00, 0x81, 0x12, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]` + `[0x00, 0x00]`
   - BCD for 8.0 = `[0x00, 0x80, 0x80, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]` + `[0x00, 0x00]`
   - Place at a clean RAM area (e.g., 0xD1AA00) well away from allocator state
   - Set FPS = clean_area + 22 (past both 11-byte entries)
   - Set FPSbase to the clean_area start

3. **Run two tests**:
   - **Test A**: Direct gcd handler entry at 0x068D3D with 11-byte FPS entries, 5000 steps
   - **Test B**: Dispatch path at 0x06859B with A=0x28, 11-byte FPS entries, 5000 steps

4. **For each test, log**:

   - FPS pointer movement (should go DOWN by 11 per pop)
   - OP2 value after first FpPop (should be 8.0, not 1.0)
   - All unique PCs visited

5. **Also verify FpPop's pop size**: Add a snapshot before and after FpPop executes to confirm FPS decrements by exactly 11.
