# Auto-Session 110 P1: RclVarSym-after-ParseInp VAT/DE Investigation

## Context
We have a validated pipeline: MEM_INIT → CreateReal("Ans") → ParseInp("2+3") → OP1=5.0. StoAns inside ParseInp correctly stores 5.0 to the pre-created Ans data slot at 0xD1A881. BUT calling RclVarSym after ParseInp returns garbage OP1=[00 70 00 8a 9a 09 fe ff 7f] instead of 5.0.

**Root cause hypothesis**: ParseInp's FP stack (FPS drops from 0xD1A88A to 0xD1A878, spanning 0xD1A878-0xD1A889) overlaps the Ans data area at 0xD1A881.

## Task
Create probe `TI-84_Plus_CE/probe-phase25ay-rclvar-vat-dump.mjs` that runs 4 scenarios:

### Scenario A: Control — dump DE from RclVarSym (working path from session 89)
MEM_INIT → CreateReal("A") with data 42.0 → RclVarSym with OP1=varA → dump DE register at entry to RclVarSym and after return. Expected: OP1=42.0, DE=data pointer.

### Scenario B: Full pipeline with VAT dump before/after ParseInp
MEM_INIT → CreateReal("Ans") → **snapshot entire VAT area (progPtr region, 0xD3FFF0-0xD3FFFF) + Ans data area (0xD1A881-0xD1A889)** → ParseInp("2+3") → **snapshot same areas again** → RclVarSym with OP1=Ans → dump OP1+DE.

Compare before/after snapshots to see exactly what ParseInp clobbers.

### Scenario C: Higher userMem base to separate FP stack from Ans data
Same as B but use userMem base = 0xD1A8C0 in MEM_INIT (64 bytes higher). This means:
- tempMem = 0xD1A8C0
- FPSbase = 0xD1A8C0
- FPS = 0xD1A8C0 (before CreateReal)
- After CreateReal, keep FPS/FPSbase at post-CreateReal values (should be 0xD1A8C9)
- Ans data slot will be at 0xD1A8C0 (first 9 bytes of userMem)
- FPS after CreateReal = 0xD1A8C9 — now 64 bytes higher than default, giving room

### Scenario D: Explicit FPS preservation after ParseInp
Same as B but after ParseInp returns, manually restore FPS/FPSbase to the post-CreateReal values (0xD1A88A) before calling RclVarSym. Test if FPS corruption is the issue vs VAT entry corruption.

## Pattern to follow
Copy the structure of `TI-84_Plus_CE/probe-phase25ax-enter-handler-sim.mjs`. Key points:
- Import createExecutor from './cpu-runtime.js', createPeripheralBus from './peripherals.js', readReal from './fp-real.mjs'
- ROM is loaded from './ROM.rom', blocks from './ROM.transpiled.js'
- Use `createPeripheralBus({ timerInterrupt: false })`
- Each run: create fresh 16MB Uint8Array, copy ROM, create executor, run to sentinel
- Multi-stage: run MEM_INIT to MEMINIT_RET sentinel, then CreateReal to CREATEREAL_RET, then ParseInp to FAKE_RET, then RclVarSym to another sentinel
- For CreateReal("Ans"): set OP1 to ANS_OP1 [0x00, 0x72, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]

- After ParseInp, DON'T reset FPS/FPSbase to MEM_INIT values (0xD1A881) — keep post-CreateReal values (0xD1A88A)
- Write tokens [0x32, 0x70, 0x33, 0x3F] to TOKEN_BUFFER_ADDR (0xD00800)
- Set begPC/curPC to TOKEN_BUFFER_ADDR, endPC to TOKEN_BUFFER_ADDR + 3
