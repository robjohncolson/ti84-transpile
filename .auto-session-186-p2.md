# Auto-Session 186 — P2: Trace 6 CALL targets in home handler to find TOKEN_STAGING populator

## Context
The TI-84 Plus CE ROM transpilation project needs to wire the foreground keyboard scan to the OS event loop. Session 185 disassembled the home handler at 0x058460-0x0584B0 and found 6 CALL sites BEFORE the `LD HL,0xD0230E; CALL 0x07F9FB` (Copy9Bytes + KeyClassifier) at 0x05849F/0x0584A3. ConvKeyToTok (0x05C52C) returns a single token byte in register E but does NOT write to TOKEN_STAGING (0xD0230E). Something among the 6 CALL targets must package the token into the 9-byte OP1-format staging area.

## Task
Build a Node.js probe script `TI-84_Plus_CE/probe-phase186-call-target-trace.mjs` that individually traces each of the 6 CALL targets to find which one reads a key code and populates TOKEN_STAGING (0xD0230E, 9 bytes).

For each of the 6 targets:
1. Boot the ROM with timer interrupts disabled
2. Set up minimal OS state (IX=0xD1A860, memInit)
3. Pre-seed a key code in register A (use 0x92 = digit 4 key, or try kbdScanCode at 0xD00587)
4. CALL the target function
5. Run for up to 500 steps
6. After execution, dump TOKEN_STAGING (9 bytes at 0xD0230E)
7. Log whether any bytes changed from the initial zero state

The 6 CALL targets are:
- 0x06EDFE
- 0x083623
- 0x083764
- 0x0800EC
- 0x0A223A
- 0x058C65

Also try entry at 0x05849F directly (the `LD HL,0xD0230E` point) with pre-seeded TOKEN_STAGING to test the copy→classify→dispatch path in isolation, with step limit of 5000.
