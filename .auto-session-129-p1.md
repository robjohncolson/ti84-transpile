# Auto-Session 129 — Priority 1: Fix 0xBB-prefix Two-Byte Token FP Dispatch Stall

## Context

When ParseInp evaluates `min(3,7)` (tokens `[0xBB, 0x0C, 0x33, 0x2B, 0x37, 0x11, 0x3F]`), it correctly parses the first argument (OP1=3.0) but then enters an infinite LCD busy-wait loop at PC=0x001221/0x001229/0x00122D. Session 128 traced the root cause:

1. ParseInp dispatches 0xBB tokens through the ROM handler chain
2. For `min`, the internal FP operation code is 0xDA
3. The FP comparison dispatch table at 0x0686D3 does not have an entry for 0xDA
4. This causes error → JError → LCD output → LCD busy-wait polling loop (infinite)

The OPS dump shows 0xDA at address 0xD3FFF8 (the operator stack).

## Task

Build `probe-phase129-fp-dispatch.mjs` that:

1. **Disassembles the FP comparison dispatch table at 0x0686D3** — read ROM bytes starting at 0x0686D3 for ~200 bytes. The table is likely a sequence of `CP n / JP Z,addr` or similar comparison-branch pairs. Map each operation code to its handler address.

2. **Identifies what operation codes ARE handled** — list all valid op codes in the table and their handler destinations.

3. **Checks the 0xDA case specifically** — where does 0xDA fall through to? Is it just missing from the table, or does it land on a catch-all error handler?

4. **Tests a runtime patch**: Before running ParseInp with `min(3,7)`, patch the LCD busy-wait at 0x001221 to break the loop. The loop structure (from session 128) is at 0x001221-0x00122D. Try:
   - Setting `mem[0xD00098] |= 0x01` (IY+0x18 bit 0 = LCD ready flag) before ParseInp
   - If that doesn't work, try patching ROM bytes at 0x001221 to NOP/RET (write to mem[] since ROM is writeable there in our 16MB flat array)

5. **Also test `max(3,7)` and `gcd(12,8)`** to see if they use different FP op codes or hit the same 0xDA issue.
