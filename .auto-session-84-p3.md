# Task: Diagnose 0x006202 loop in the committed ParseInp probe

## Context
We are reverse-engineering the TI-84 Plus CE OS ROM. The committed probe `TI-84_Plus_CE/probe-phase25x-meminit-then-parseinp.mjs` runs ParseInp (0x099914) after MEM_INIT and gets stuck at PC=0x006202 after 500K steps. A separate Sonnet variant returned in 918 steps with OP1=5.0. The difference is the error handler frame setup.

The committed probe uses a minimal error frame:
```javascript
write24(mem, cpu.sp, FAKE_RET);           // main return
const errFrameBase = (cpu.sp - 6);
write24(mem, errFrameBase + 3, ERR_CATCH_ADDR);  // just 6 bytes
write24(mem, ERR_SP_ADDR, errFrameBase);
```

This is NOT a proper PushErrorHandler frame. When ParseInp's internal error handling fires, `JError` does `ld sp,(errSP); pop af; ret`. With only 6 bytes, the `ret` lands at ERR_CATCH_ADDR — but the probe doesn't intercept that address during execution (it only checks FAKE_RET). So the error handler fires, lands at ERR_CATCH_ADDR (0x7FFFFA), and since that address has no transpiled block, execution likely continues into garbage.
