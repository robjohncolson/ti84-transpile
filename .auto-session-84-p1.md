# Task: ParseInp with full PushErrorHandler frame + token terminator variants

## Context
We are reverse-engineering the TI-84 Plus CE OS ROM. **Session 83** found that calling MEM_INIT (0x09DEE0) before ParseInp (0x099914) eliminates ErrMemory. A Sonnet subagent's alternate probe variant reported **OP1=5.0 in 918 steps** with errNo=0x8D (ErrSyntax). The committed Codex probe uses a minimal 6-byte error frame and gets stuck at 0x006202 after 500K steps.

**The hypothesis**: The Sonnet probe's success came from using a **full PushErrorHandler 18-byte ADL frame** (not just a minimal 6-byte catch frame). When ParseInp internally uses PushErrorHandler, it expects the error recovery to restore OPS/FPS from saved deltas. Without this, the longjmp lands but recovery code is broken, causing infinite loops.

**errNo=0x8D = ErrSyntax** — the parser may need a proper end-of-input token. TI-OS uses 0x3F (newline/tEnter) as line terminator for input expressions.

## PushErrorHandler frame format (from disassembly at 0x061DEF)
The frame pushed by PushErrorHandler, from SP upward:
```
SP+0:  0x061E27   (normal-return cleanup stub address)
SP+3:  0x061DD1   (error-restore stub address)
SP+6:  OPS - OPBase  (3-byte delta, signed)
SP+9:  FPS - FPSbase (3-byte delta, signed)
SP+12: previous errSP (3 bytes)
SP+15: caller's HL payload (3 bytes)
```
Then `ld (0xD008E0), SP` saves SP as new errSP.

Error path: JError at 0x061DB2 does `ld sp,(errSP); pop af; ret` → lands at 0x061DD1.
0x061DD1 restores OPS, FPS, previous errSP from the saved frame.

### Step 1: Create probe file
Create `TI-84_Plus_CE/probe-phase25y-parseinp-full-errframe.mjs` that:

Loads the runtime (copy the loading pattern from `TI-84_Plus_CE/probe-phase25x-meminit-then-parseinp.mjs` — read that file first for the exact import/loading pattern, including `createExecutor`, `readReal`, ROM loading via `pathToFileURL`, and the `coldBoot` + `prepareCallState` helper functions).

2. Cold boots and calls MEM_INIT (0x09DEE0) — exactly as the 25x probe does.

3. Sets up the token buffer at 0xD00800 with tokenized "2+3" followed by a terminator.

4. Points begPC/curPC/endPC (0xD02317/0xD0231A/0xD0231D) to the token buffer. Does NOT touch OPS (leave at MEM_INIT value 0xD3FFFF).

5. **Builds a FULL PushErrorHandler frame** on the stack:
```javascript
// After prepareCallState and pushing FAKE_RET as outer return:
// Build the 18-byte PushErrorHandler frame
const opsVal = read24(mem, 0xD02593); // OPS after MEM_INIT
const opBaseVal = read24(mem, 0xD02590);
const fpsVal = read24(mem, 0xD0258D);
const fpsBaseVal = read24(mem, 0xD0258A);

const opsDelta = (opsVal - opBaseVal) & 0xFFFFFF; // OPS - OPBase
const fpsDelta = (fpsVal - fpsBaseVal) & 0xFFFFFF; // FPS - FPSbase

// Push frame (grows downward from current SP)
// Frame: [cleanup_stub, err_stub, ops_delta, fps_delta, prev_errSP, hl_payload]
const prevErrSP = 0x000000; // no previous handler
const hlPayload = 0x000000; // unused

cpu.sp -= 3; write24(mem, cpu.sp, hlPayload);       // SP+15
cpu.sp -= 3; write24(mem, cpu.sp, prevErrSP);        // SP+12
cpu.sp -= 3; write24(mem, cpu.sp, fpsDelta);          // SP+9
cpu.sp -= 3; write24(mem, cpu.sp, opsDelta);          // SP+6
cpu.sp -= 3; write24(mem, cpu.sp, 0x061DD1);          // SP+3: error-restore stub
cpu.sp -= 3; write24(mem, cpu.sp, 0x061E27);          // SP+0: normal-return stub

// Save SP as errSP
write24(mem, 0xD008E0, cpu.sp);
```

6. Clears errNo at 0xD008DF to 0x00.

7. **Runs ParseInp 3 times** with different token terminators:
   - Variant A: tokens = [0x32, 0x70, 0x33, 0x3F] (2+3 + newline) — this is what the committed probe uses
   - Variant B: tokens = [0x32, 0x70, 0x33, 0x04] (2+3 + tEnter/→)
   - Variant C: tokens = [0x32, 0x70, 0x33, 0x00] (2+3 + null)
   
   For each variant: reset state from MEM_INIT (call MEM_INIT again), set up tokens + errFrame + curPC, run ParseInp with budget=50000 steps, dump errNo, OP1, pointer state, curPC advancement.

8. For reading OP1 as a float, use `readReal` from `./fp-real.mjs` (see the 25x probe for the import pattern and `wrapMem` helper).

9. Print a summary comparing all 3 variants.
