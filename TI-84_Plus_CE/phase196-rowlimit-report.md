# Phase 196 — rowLimit / colLimit Write Trace

Read-only investigation. Target addresses:

- `rowLimit` at `0xD02504`
- `colLimit` at `0xD02505`

Observed values during home-screen Stage 1: both `0x00`. Expected defaults (per
hypothesis) were `row=8`, `col=26`.

## 1. Grep Findings

Searched `ROM.transpiled.js` for `0x[dD]02504` / `0x[dD]02505`. Total 452
matches across the JSON + source blocks (same block appears in both the block
record and the `source` body). False-positive search (`0x02504` / `0x02505`
substring) only surfaced unrelated PCs in the `0x025000` code region — none
target the two MMIO/RAM bytes.

**Writer blocks** (`cpu.write8(0xd02504, …)` / `cpu.write8(0xd02505, …)`,
89 total write-sites — 30 shown). The value actually stored is whichever was

## 3. Final Values After Boot + Render

- `rowLimit` (`0xD02504`) = **`0x00`**
- `colLimit` (`0xD02505`) = **`0x00`**

Writes observed: 3 to rowLimit, 2 to colLimit — **all with value `0x00`**.

Write phases map cleanly onto the known writer blocks:

- `boot-z80` rowLimit / colLimit `0x00` → likely early RAM scrub in the
  real-mode boot path (pre-`08c331`), not a deliberate limit initialisation.
- `kernel-init` rowLimit / colLimit `0x00` → matches the `xor a; ld (0xd02504), a`
  at `0x03f2ab` chain or the `sub a; ld (0xd02504), a` at `0x080cc6`. Both
  callers treat this byte as a free scratch counter to be zeroed before a
  subroutine call, not a mode-limit default.
- `stage4-history` rowLimit `0x00` → matches `block_0a285c_adl` (`ld (0xd02504), a`)
  which runs early in the history-render path; `a` reaches it as `0` here.

## 4. Verdict

**(c) writer reached but writes `0x00`.**

There are ~89 write-sites against these two bytes. Every writer that fires
during the home-screen boot/render path stores `0x00`, because the
neighbourhood idiom is "zero A via `xor a` / `sub a` / arithmetic, store,
then call a helper". No writer in the covered path stores the
hypothesised defaults `0x08` / `0x1A`.

Implication: `0xD02504` / `0xD02505` are **not** static row/col limits with a
fixed default; they are mutable counters the OS edits as the home-screen
cursor advances. If Stage 1 reads these pre-render expecting `row=8, col=26`,
the assumption is wrong — either (a) the writer that sets the non-zero limit
lives on a code path Stage 1 does not reach from its entry point
(`0x0a2b72`), or (b) these addresses are not the "rowLimit / colLimit"
symbols at all. Candidate writers that *do* produce non-zero:

- `0x0b6d84`: immediate `0x04` into `0xd02504`
- `0x0a6320`: immediate `0x01` into `0xd02504`
- `0x0a2947`, `0x0a6442`, `0x0ae113`, `0x07ab8c`: immediate `0x0a` into `0xd02505`
- `0x0a5838`, `0x0a6449`: immediate `0x09` into `0xd02505`

The recurrent immediates `0x09` / `0x0a` for `0xD02505` and `0x04` for
`0xD02504` suggest these bytes are **small-domain mode counters** (menu
index, font-size, digits-remaining) rather than 26-column / 8-row display
limits. Recommend re-identifying which OS RAM symbol actually holds
`row=8 / col=26` before any further inference.
