## Question

Does CoorMon read cxCurApp (0xD007E0) **BEFORE** or **AFTER** the JP (HL) dispatch at 0x08C745?

## Answer

**cxCurApp is read BEFORE JP (HL) dispatch.** All five cxCurApp read sites are in the CoorMon main body (0x08C408-0x08C5E7), well before the dispatch sub (0x08C72F) that contains the JP (HL) at 0x08C745.

14. **0x08C73D-0x08C744** — Dispatch sub epilogue: restores IY, pops HL twice, RET to CoorMon main body.

## Implication for cxCurApp=0x00

**cxCurApp=0x00 in the context table is NOT a blocker for the ENTER key path.** Here is why:

1. **First CoorMon iteration**: cxCurApp=0x40 (home). CoorMon reads it at 0x08C408, compares with 0x58 — not equal, so no special routing. Continues to dispatch sub, reads cxMain=0x058241 (HomeHandler), dispatches via JP (HL). HomeHandler loads the context table via LDIR, which sets cxCurApp=0x00 and cxMain=0x0585E9.

2. **Second CoorMon iteration**: cxCurApp=0x00. CoorMon reads it at 0x08C408, compares with 0x58 — not equal (0x00 != 0x58), so no special routing. The key comparisons are all against specific app IDs (0x44, 0x50, 0x52, 0x58, 0x5B) — **none compare against 0x00**. So cxCurApp=0x00 falls through all the app-specific checks and proceeds to the normal dispatch path, where cxMain=0x0585E9 is the new handler.

3. **The two-pass model works**: CoorMon reads cxCurApp before dispatch, but cxCurApp=0x00 does not match any blocking condition. The dispatch happens via cxMain, which is set independently by HomeHandler.
