# Phase 102B Visual Comparison Report

## Scope

This report compares the generated composite in `home-screen-render.png` against a real TI-84 Plus CE home screen.

Evidence used:

- `probe-phase102b-vram-to-png.mjs`
- `home-screen-render.png` (320x240 PNG generated by that probe)

## What The Probe Actually Draws

The probe is not running a full "show the home screen" path. It builds a composite from a few isolated stages:

1. Boot and OS init.
2. Clear VRAM to the sentinel word `0xAAAA`.
3. Run `statusBarBackground`.
4. Run `statusDots`.
5. Manually seed `HOME_SCREEN_MODE_BUF` with `Normal Float Radian       `.
6. Run `homeRow`.
7. Run `historyArea`.
8. Force rows `220-239` to solid white with `fillEntryLineWhite()`.

That matters because the PNG is a stitched-together partial render, not a faithful end-to-end home-screen draw.

`0xAAAA` in RGB565 decodes to a dark brick-red color in the PNG, roughly `#AD5552`. Any area still showing that color was never painted by the pipeline.

## Status Bar (rows 0-16)

**Real TI-84 Plus CE**

The real home screen uses a blue/gray status strip rather than a plain white band. It normally includes battery status at the top-right and small annunciators for calculator state. The exact icon set can vary a bit by OS version and current mode state, so I am intentionally keeping this comparison at the "battery/status icons exist" level.

**Our render**

- Rows `0-16` are almost entirely flat white.
- The only non-white pixels are two tiny dot clusters:
  - rows `3-6`, cols `146-150`
  - rows `3-6`, cols `306-310`
- Those dots use only a few pixels of yellow/blue/dark accent color.
- There is no battery icon and no full status icon row.

**Delta**

The current probe captures only a minimal background pass plus a tiny "dots" stage. It does not render the actual CE status-bar chrome or iconography.

## Mode Text Area (rows 17-34)

**Real TI-84 Plus CE**

On the real calculator, the home screen does not dedicate this band to `Normal Float Radian`. That string belongs to the MODE screen, not the home screen. On the home screen, this region is part of the normal calculation/history area.

**Our render**

- The probe always seeds `HOME_SCREEN_MODE_BUF` with `Normal Float Radian       ` before drawing `homeRow`.
- In the PNG, rows `17-34` render as a dark band with white glyphs for that mode string.
- The rightmost `6` columns (`314-319`) remain sentinel red in this band, so even this row is not filled edge to edge.

**Delta**

This is the biggest semantic mismatch in the image. The render is showing MODE-screen text in a place where the real home screen should show blank history space or prior calculations.

## History Area (rows 37-74)

**Real TI-84 Plus CE**

The home screen below the status bar is mostly a large white workspace. If there is recent history, it appears here as black text lines on white. If there is no history yet, it is still a plain white display area, not a decorative boxed region.

**Our render**

This area is not a true blank history pane. It looks like partial scaffolding from the `historyArea` stage:

- rows `37-42`: solid black across cols `0-313`
- rows `43-48`: alternating black/white stripe pattern across cols `0-313`
- rows `49-54`: solid black again
- rows `55-56`: fully sentinel red
- rows `57-62`: another solid black band
- rows `63-68`: another alternating black/white stripe band
- rows `69-74`: solid black again
- cols `314-319` stay sentinel through most of this block

There is no actual calculation history text and no full white workspace.

**Delta**

The current stage seems to produce only fragments of layout chrome or text-grid structure. It does not resemble the real CE home screen's open white history area.

## Entry Line (rows 220-239)

**Real TI-84 Plus CE**

The bottom of the home screen is a white entry line with an active text cursor. When idle, it is visually a live input field, not just a blank rectangle.

**Our render**

- Rows `220-239` are forced to solid white by `fillEntryLineWhite()`.
- There is no prompt content, no cursor, and no blinking state.

**Delta**

The probe gets the gross background color right, but it is only a placeholder fill. None of the real input-line behavior is present yet.

## Unrendered Area (rows 75-219)

**Real TI-84 Plus CE**

This whole region should still be part of the main white display area between history and the input line.

**Our render**

- Most of rows `75-219` are still the `0xAAAA` sentinel color, visible as dark brick red.
- There is a leftover rendered artifact in the upper-left of this region:
  - rows `77-94`
  - roughly cols `0-133`, tapering to `0-121`
- That leftover block contains black, and in rows `83-88` also some white, but it is clearly not a real home-screen payload.

**Delta**

The majority of the middle of the screen is simply unrendered. This is the clearest sign that no stage currently paints the full home-screen body.

## Recommended Next Steps

1. Stop injecting MODE-screen text into the home-screen probe. `seedModeBuffer()` is useful for proving glyph rendering, but it guarantees the wrong content for a home-screen comparison.
2. Add a full-body white fill for the home workspace before or during the history draw. Even a temporary fill for rows `17-219` would remove the sentinel red and make later deltas easier to reason about.
3. Identify the actual ROM routine that clears or lays out the home-screen history region, then call that instead of relying on the current partial `historyArea` stage in isolation.
4. Add the entry cursor path after the white entry-line fill. This should come from the real text/cursor routine rather than another hard-coded rectangle if the goal is fidelity.
5. Treat status-bar fidelity as a later pass. The missing blue/gray bar and battery icon matter, but the more important structural bug is that most of the home-screen body is still wrong or unrendered.

## Confidence Notes

- I am confident that `Normal Float Radian` is MODE-screen content and does not belong on the real home screen.
- I am confident that the large brick-red region in the PNG is unpainted sentinel VRAM, not intended UI.
- I am moderately confident on the exact top-bar icon inventory because it can vary with OS revision and current calculator state, but the current render is clearly missing the battery/icon layer regardless of those details.
