# Phase 81 — Glyph Decode Report

Re-ran the 3 Phase 78 parent probes and decoded the rendered status-bar text by
intercepting every call to `0x0a1799` (the single-char printer) and recording
the character code in register A at each invocation.

This approach gives **exact** character codes — no image matching needed.

## Method

1. Booted ROM → OS init → SetTextFgColor(black).
2. Installed a `write8` interceptor and `onBlock` hook for `0x0a1799`.
3. Ran each Phase 78 probe with sentinel-cleared VRAM.
4. Collected (charCode, VRAM col) pairs from all `0x0a1799` invocations.
5. Sorted by column → left-to-right character sequence.

## Summary

| probe | entry | decoded text | chars | steps |
|-------|-------|-------------|------:|------:|
| `p0a2b72_05e7d2` | 0x05e7d2 | `[0xef][0x02][0xc1]eqnname,color#,[0x00] li` | 22 | 20905 |
| `p0a2b72_05e481` | 0x05e481 | `[0xef][0x02][0xc1]eqnname,color#,[0x00] li` | 22 | 20909 |
| `p0a2b72_09cb14` | 0x09cb14 | `:[0xef][0x02][0xc1]eqnname,color#,[0x00] li` | 23 | 21702 |

### Interpretation

The text `eqnname,color#,[0x00] li` appears in all three probes.
This is a **Y= equation attribute line** — the status bar is rendering the
settings for a function in the Y= editor:

- `eqnname` = equation name
- `color#` = color number
- `[0x00]` = null/separator
- `li` = line style (start of "line" or "linestyle")

The probe `09cb14` adds a leading `:` — likely a field separator or second entry.

The three non-ASCII chars at the start (`[0xef] [0x02] [0xc1]`) are TI special
tokens, possibly indicating a menu header or selection marker.
