# Phase 25AY - pTempCnt Audit + Descending-Boundary Invariant

This report file is generated by `probe-phase25ay-ptcnt-invariant.mjs` when the probe is executed. The checked-in version documents the intended audit and harness; the probe overwrites this file with observed results.

## Scope

- Part 1: run `MEM_INIT` on a fresh runtime, read `pTempCnt` at `0xD02596` with a 24-bit read, and confirm the value is zero.
- Part 2: after `MEM_INIT`, verify `tempMem <= FPSbase <= FPS <= OPS <= OPBase <= pTemp <= progPtr` and compare the observed cluster against the expected `userMem/top-of-user-memory` plateau.
- Part 3: run `MEM_INIT -> CreateReal("Ans") -> ParseInp("2+3")`, dumping pointers and checking the invariant after each stage.
- Part 4: run fresh-runtime `ParseInp` cases for `2+3`, `3*4`, and `9-1`, then dump pointers and check the invariant after each parse.

## Harness

- Fresh runtime per scenario: new `Uint8Array(0x1000000)`, ROM copy, fresh executor, fresh peripheral bus, and the usual cold-boot sequence before `MEM_INIT`.
- `pTempCnt` is reported with `read24(0xD02596)` even though some notes mention a four-byte window at that location; the probe also records the raw four-byte window for context.
- `ParseInp` setup follows the existing direct-eval probes: write tokens to `0xD00800`, set `begPC/curPC/endPC`, clear `OP1`, seed the error frame, push `FAKE_RET`, and run without reseeding FPS back to MEM_INIT values.
- The checked-in report is a stub only. Running the probe replaces it with the observed data and invariant results.
