## Summary

The TI-84 Plus CE OS uses a **4-section lookup table** at `0x09F79B` to convert
`_GetCSC` sequential scan codes (0x01-0x38) into internal OS key codes. The
section is selected by the current modifier state stored in `(IY+0x12)`:

| Section | Offset | Condition | Description |
|---------|--------|-----------|-------------|
| 0 | +0x00 | No modifier | Unshifted key codes |
| 1 | +0x38 | 2nd active | 2nd-shifted key codes |
| 2 | +0x70 | Alpha active | Alpha-shifted key codes |
| 3 | +0xA8 | Alpha+2nd | Combined modifier codes |

The lookup code at `0x03010D` adds the section offset to the scan code, then
indexes into the table: `A = table[scanCode + sectionOffset]`.

**Key finding**: The internal codes at `0x08759D (= IsFullCntx)` (the classifier from Phase 136)
use values from Section 0 of this table. Letter codes 0x41-0x5A appear in the
unmodified section for the function key row (GRAPH=D, TRACE=Z, WINDOW=H, Y\==I,
MODE=E) -- these are the alpha labels printed on those keys.

## Lookup Code Disassembly (0x030100-0x030134)

```text
0x030100  f2 fb 03 fd       JP P,0xfd03fb
0x030104  cb                db 0xCB
0x030105  12                db 0x12
0x030106  66                db 0x66
0x030107  cc f6 01 03       CALL Z,0x0301f6
0x03010b  16 01             LD D,0x01
0x03010d  c6 70             ADD A,0x70
0x03010f  fd cb 12 6e       BIT 5,(IY+0x12)
0x030113  ca 74 00 03       JP Z,0x030074
0x030117  c6 38             ADD A,0x38
0x030119  6f                LD L,A
0x03011a  26 00             LD H,0x00
0x03011c  11 9b f7 09       LD DE,0x09f79b
0x030120  19                ADD HL,DE
0x030121  7e                LD A,(HL)
0x030122  fe e2             CP 0xE2
0x030124  38                db 0x38
0x030125  0e fe             LD C,0xFE
0x030127  fc                db 0xFC
0x030128  30                db 0x30
0x030129  0a                db 0x0A
0x03012a  32 8e 05 d0       LD (0xd0058e),A
0x03012e  3e fc             LD A,0xFC
0x030130  c3 84 fe 02       JP 0x02fe84
0x030134  16 01             LD D,0x01
```

**Flow**:
1. `BIT 4,(IY+0x12)` tests alpha mode flag
2. If alpha is off (Z=1): `CALL Z,0x0301F6` (alpha key remapping)
3. `ADD A,0x70` (alpha section offset)
4. `BIT 5,(IY+0x12)` tests 2nd mode flag
5. If 2nd is off (Z=1): `JP Z,0x030074` (use alpha offset only)
6. If 2nd is on: `ADD A,0x38` (add 2nd offset = total 0xA8)
7. `LD L,A; LD H,0; LD DE,0x09F79B; ADD HL,DE; LD A,(HL)` -- table lookup
8. Result in A is compared against 0xE2 and 0xFC for special handling
9. If in range, stored directly to `0xD0058E` (key event buffer)

## Architectural Significance

1. **Two-stage key processing**: The OS converts raw keyboard matrix scans
   to `_GetCSC` sequential codes (0x01-0x38) in the ISR, then the main
   loop uses this table at `0x09F79B` to convert to internal codes.

2. **Modifier-aware lookup**: The 4-section table handles all modifier
   combinations (none/2nd/alpha/alpha+2nd) in a single indexed lookup,
   controlled by flags at `(IY+0x12)` bits 4 and 5.

3. **Internal code semantics**:
   - `0x01-0x0F`: Control codes (arrows, enter, clear, del)
   - `0x30-0x3E`: Digit/punctuation characters
   - `0x40-0x5A`: Letter codes (ASCII-based, used by Phase 136 classifier)
   - `0x80-0xFF`: Function/token codes (math ops, trig, parentheses, etc.)

4. **Function key alpha labels**: In the unmodified table, keys GRAPH through
   MODE produce ASCII letter codes (D, Z, H, I, E) -- these correspond to
   the alpha labels physically printed on those keys. This is why the
   Phase 136 classifier checks for these letters even in non-alpha mode.

5. **Key event buffer**: The converted internal code is stored at `0xD0058E`,
   which is read by the main key handler at `0x08C4A3` and dispatched
   through the key processing chain identified in Phases 113-136.
