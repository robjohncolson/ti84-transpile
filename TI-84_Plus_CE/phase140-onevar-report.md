# Phase 140 — OneVar Pre-Alloc: Fix E_DataType via Correct Initialization

Generated by `probe-phase140-onevar-prealloc.mjs`.

## Goal

Fix OneVar's errNo=0x89 error (E_DataType, not E_Memory as previously believed) by correctly initializing the runtime environment before calling OneVar. Session 139 P3 showed OneVar running 5000 steps but failing on extended runs.

## Root Cause Analysis

Three independent issues were identified and fixed:

### Issue 1: ZeroStatVars overwrites OP1 type byte (E_DataType trigger)
- OneVar at 0x0A9273 calls 0x0AA697 which checks bit 2 of `statFlags2` (IY+26 = 0xD0009A).
- If bit 2 is CLEAR, it calls 0x03E8FB (`ZeroStatVars`) which writes `0x0E` to OP1[0], overwriting the ListObj type byte.
- Later at 0x0A928F, the code re-checks OP1[0] for ListObj (0x01), finds 0x0E, and raises E_DataType (0x89).
- **Fix**: Set bit 2 of `statFlags2` before calling OneVar to skip ZeroStatVars. Pre-seed the stat var RAM slots manually instead.

### Issue 2: FP stack not primed with list variable name
- OneVar at 0x0AA697 calls `PushRealO1` (0x082961) which pushes OP1 to the FP stack.
- Later at 0x0AA6A3, `PopRealO1` (0x082902) pops from FP stack back to OP1.
- But between entry and 0x0A9273, OneVar does NOT push OP1 to the FP stack — the push at 0x0AA697 is the FIRST push. Then the pop at 0x082902 reads back the pushed value.
- However, later code at 0x0A928B does a SECOND pop, which reads the PRE-EXISTING FP stack top.
- If nothing was on the FP stack before OneVar, this second pop reads garbage.
- **Fix**: Pre-push L1's OP1 value onto the FP stack before calling OneVar.

### Issue 3: FPS/list data address collision
- `seedAllocator` sets `FPS = USERMEM_ADDR`. List data was also placed at `USERMEM_ADDR`.
- When the FP stack push wrote L1 OP1 bytes to FPS, it overwrote the first 9 bytes of list data (the element count and first BCD element).
- **Fix**: Separate FPS base from list data by placing them at different addresses (FPS at USERMEM_ADDR, list data at USERMEM_ADDR + 0x100).

### Issue 4: A register must be 1 for single-list mode
- OneVar at 0x0A933A pushes AF, then at 0x0A9343 pops AF and copies A to B.
- At 0x0A93D7, `cp 0x01` checks if B=1 (single list, no frequency list). If B=1, takes the `0x0A93F1` path.
- With A=0 (default), B=0, the code falls through to 0x0A93E9 which uses the list as both data AND frequency weights, producing wrong results (n=sum of elements instead of count).
- **Fix**: Set `cpu.a = 1` before calling OneVar.

## Key Findings

1. **errNo=0x89 is E_DataType, not E_Memory (0x8E).** The error was a type-check failure, not a memory allocation issue. The previous session's characterization was incorrect.

2. **Stat vars use dedicated RAM, not VAT.** OneVar writes stat results directly to `statVarsOffset + token*9` slots. Creating VAT entries for stat vars is unnecessary and counterproductive (causes E_DataType when FindSym finds them with the wrong type).

3. **tXMean (0x03) is the correct mean token, not tMean (0x21).** OneVar writes the mean to the XMean slot at `statVars + 1*FPLEN` = 0xD0119A. The tMean token at 0x21 maps to a different location (0xD012A8) and is NOT written by OneVar.

4. **FP stack direction confirmed.** PushRealO1 (0x082961) pushes OP1 to FP stack using `LDI (HL=OP1) -> (DE=FPS)`, advancing FPS. PopRealO1 (0x082902) pops FP stack to OP1 using `LDI (HL=FPS-9) -> (DE=OP1)`, decrementing FPS.

5. **A=1 selects single-list mode.** A=0 selects frequency-weighted mode where the same list is used for both values and frequencies.

## Initialization Recipe

To run OneVar successfully:
```
1. MEM_INIT (0x09DEE0)
2. Set FPS base away from list data: FPS_BASE = USERMEM_ADDR, list at USERMEM+0x100
3. Create L1 list data at list address (2-byte count + n*9 BCD reals)
4. Write L1 VAT entry (type=0x01, name=0x01, data_ptr=list address)
5. Pre-seed stat var slots at 0xD0117F + token*9 with [0x0E, 0x80, 0x00*7]
6. Fill stat ans region 0xD01485..0xD014E9 with 0xFF
7. Set statsValid: bit 6 of (IY+9) = 0xD00089
8. Set statFlags2 bit 2: (IY+26) = 0xD0009A |= 0x04  (skip ZeroStatVars)
9. Set OP1 = L1 name [0x01, 0x01, 0x00...]
10. Set cpu.a = 1 (single-list mode)
11. Push L1 OP1 onto FP stack (write at FPS, advance FPS by 9)
12. Seed errSP error frame
13. Call OneVar at 0x0A9325
```
