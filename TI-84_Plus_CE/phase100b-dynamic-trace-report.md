## Additional Investigation Findings (Manual ROM Analysis)

### Architecture Summary

The 0x0b2xxx-0x0b5xxx module is the **mode display management module** with three distinct function classes:

1. **CLEAR functions** (e.g. 0x0b2d6a, 0x0b2e86): zero-fill or LDIR-shift the mode buffer. These are what the static scan found.
2. **SHIFT functions** (e.g. 0x0b2f16, 0x0b2f36): LDIR-based scroll of mode buffer contents between slots (used when mode settings change to animate the display row).
3. **FORMAT functions** (e.g. 0x0b3409 via 0x0b33ff): compute display offsets, format float digits using multiply-by-9 (0x07f796 (= HLTimes9)), write packed mode data to the buffer's lower half.

### Mode String Table

The ROM string table at **0x0a044e** holds mode name strings in `len + chars + token` format (8-byte stride for angle/num modes):
- 0x0a044e: `len=6, "Radian", token=0x4E`
- 0x0a0456: `len=6, "Degree", token=0x4F`
- 0x0a045e: `len=6, "Normal", token=0x50`
- 0x0a0466: `len=3, "Sci", token=0x51`
- 0x0a0471: `len=5, "Float", token=0x0A`

No direct `LD HL, 0x0a044e` was found — string access uses computed offsets.

### Why All Writes Are Zeros

The PARTIAL functions all write **only to offsets +12..+25** (the angle-mode slot at 0xD020B2+) and write zeros because:
- The mode index byte at `0xD02048` (num mode: 0=Normal) is 0
- The angle mode byte at `0xD0204A` is 0
- With index=0, the bit-manipulation logic in 0x0b2f65/0x0b33df produces A=0, which is written as a null byte (not ASCII)
- The "Normal" and "Float" slots at +0..+11 are NOT reached by any of the 13 static-hit functions — those bytes must be written by a **separate function not found in the static scan**

### True Populate Path

The real `FullPopulate` must:
1. Read `0xD02048` (num mode), look up string in 0x0a045e region, LDIR to 0xD020A6+0
2. Write space to 0xD020A6+6
3. Read `0xD02049` (float mode), format float digits, write to 0xD020A6+7
4. Write space to 0xD020A6+12
5. Read `0xD0204A` (angle mode), look up string in 0x0a044e region, LDIR to 0xD020A6+13

The function that handles steps 1-2 (writing the "Normal" portion to offsets +0..+5) uses a **computed address** (never a direct `LD HL,0xD020A6`) and so was invisible to the static scan. It is likely inside the 0x0b2xxx-0x0b5xxx module but in an unlifted block.
