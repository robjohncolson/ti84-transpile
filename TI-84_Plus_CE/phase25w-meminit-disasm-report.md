- Entry point for the MEM_INIT block: 0x09DEE0. The immediately preceding instruction at 0x09DEDF is an unconditional `ret`, so control cannot fall through from the previous routine.
- HL for the FPSbase/FPS writes is statically determined as 0xD1A881 (userMem), loaded at 0x09DEE0. The same HL value is reused for `tempMem`, `FPSbase`, `FPS`, and `newDataPtr` before HL is reloaded with 0xD3FFFF.
- Additional initialization before those stores is not part of the same routine. The decode window shows a hard boundary at 0x09DEDF, so the pointer stores start a fresh routine rather than continuing the earlier setup code at 0x09DE80..0x09DEDF.
- Direct CALL/JP references into 0x09DE80..0x09DF10: `0x09DD62: call 0x09DEE0`.

## Secondary cluster at 0x0820FA

- 0x0820ED is not a reset routine. It is a 9-byte wrapper that seeds BC = 0x000009 and then falls into the generic helper at 0x0820F1.
- 0x0820F1 subtracts BC from OPBase immediately, then conditionally subtracts the same span from pTemp and progPtr depending on the OP1+6 type byte and helper calls at 0x07F7BD, 0x08012D, and 0x080080.
- The continuation at 0x082133 also rewrites OPS and then performs block movement with LDIR, so this is a compaction/delete helper that shrinks live allocator pointers after removing a record, not a boot-time initializer.
- Direct entry xrefs reinforce that split: 0x08237A calls the 9-byte wrapper 0x0820ED, while 0x082340 and 0x0848AC call the generic BC-sized helper at 0x0820F1.
