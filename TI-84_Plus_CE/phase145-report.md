## Critical Finding

**The dispatch table region is never populated.** The OS init crashes at step 691 (missing block at 0xFFFFFF) long before the dispatch table populator at 0x056900/0x0AD459 is reached. All 512 bytes in the 0xD02200-0xD02400 region remain at their cold-boot value of 0xFF.

The only writes to this region come from `0x00287D` (the cold-boot LDIR that fills RAM with 0xFF). No OS-level initialization code ever touches these addresses during the 691 steps that execute.

This means:
1. The dispatch table structure **cannot be analyzed from runtime memory** with the current boot path
2. The step-691 boot crash (Phase 139's finding) blocks all dispatch table initialization
3. To analyze the dispatch table, either the boot crash must be fixed first, or the analysis must be done **statically from ROM bytes** (as Phase 138 already did)

## 0x0AD459 Function Analysis

The function at `0x0AD459` (`block_0ad459_adl`) is the dispatch table entry processor.
It is called in a loop with `LD B, 0x12; CALL 0x0AD459` (18 iterations).

### Execution Flow

```text
0x0AD459: ld c, 0x00          ; clear C
          ld de, 0x000000      ; clear DE
          push ix              ; save IX
          ld h, a              ; save A in H
          ld a, i              ; read interrupt vector register
          jp pe, 0x0AD46A      ; branch on parity (interrupt state)

0x0AD468: ld a, i              ; (fallthrough: re-read I)

0x0AD46A: push af              ; save flags
          ld a, h              ; restore original A
          di                   ; disable interrupts
          ld ix, 0xD02670      ; IX = node workspace at 0xD02670
          res 6, b             ; clear bit 6 of B (flags field)
          ld (ix+0), b         ; store B (flags) into workspace
          ld (ix+1), c         ; store C into workspace
          ld (ix+2), 0x00      ; clear workspace byte 2
          ld hl, (0xD0231A)    ; HL = current dispatch pointer
          push hl              ; save current pointer
          push de              ; save DE (0)
          bit 6, (ix+0)        ; test bit 6 of flags
          jr z, 0x0AD48E       ; if clear, call 0x09BAC9 (= NxtFetch)
```

### Key Observations

1. **IX = 0xD02670** is a 3-byte workspace used during processing (flags, C, zero)
2. **Reads 0xD0231A** (current pointer) and saves it on the stack
3. **At 0x0AD4DF**: writes BC to 0xD0231A — this advances the current pointer
4. **At 0x0AD4B4**: also writes BC to 0xD0231A (alternate path)
5. **At 0x0AD69B** (exit): writes HL to 0xD0231A — final pointer update
6. The function processes one entry per call, advancing the current pointer (0xD0231A)
7. B register carries flags between calls (bit 6 tested, bit 0 tested at various points)
8. Calls to 0x09BAC9 (= NxtFetch), 0x09BBAD suggest token parsing / string comparison

1. Calls helper functions to compute sizes
2. Writes base to 0xD02317 and 0xD0231A (initializes base = current)

4. Recomputes and stores final current to 0xD0231A

## Node Structure Hypothesis

The stride `LD DE, 0x000011` (17 bytes) at 0x05693B strongly suggests each dispatch
table entry is **17 bytes**. With 18 entries, that would be 306 bytes total.

The structure appears to be a **contiguous array** (not a linked list), where:
- `0xD02317` = base pointer (start of array)
- `0xD0231A` = current position / cursor
- `0xD0231D` = end pointer (last valid byte)

Each 17-byte entry likely contains:
- Flags/type byte(s)
- A token or command identifier
- Handler address (3 bytes for eZ80 ADL mode)
- Additional context/parameters

## Static Analysis of 0x0AD459 (Dispatch Entry Processor)

Reconstructed from `ROM.transpiled.js` blocks:

### Full call chain

```
0x0AD459 entry:
  C = 0, DE = 0, push IX, H = A, read I register
  -> 0x0AD46A (or 0x0AD468 fallthrough)

0x0AD46A:
  push AF, A = H (restore), DI
  IX = 0xD02670 (3-byte workspace)
  (IX+0) = B & ~0x40    (flags, bit 6 cleared)
  (IX+1) = C
  (IX+2) = 0x00
  HL = (0xD0231A)        (current pointer)
  push HL, push DE
  bit 6, (IX+0) -> branch

0x0AD48E: call 0x09BAC9 (= NxtFetch)  (token reader / string scanner)

0x0AD492: cp 0x3E         (compare A with '>')
  if A != 0x3E -> 0x0AD4BD (call 0x09BBAD, another scanner)
  if A == 0x3E -> 0x0AD496 (call 0x0AD6D4, bit 3 test)

0x0AD4DF: ld (0xD0231A), BC  <- ADVANCES THE CURSOR
  cp 0xBB -> branch on token type

0x0AD69B (exit path):
  write A to (0xD02673)     (workspace byte 3)
  pop AF, restore registers
  ld (0xD0231A), HL         <- FINAL CURSOR UPDATE
  pop IX, restore interrupt state, ret
```

### Interpretation

The function is a **token parser / command dispatcher**. Each call:
1. Reads the current position from `0xD0231A`
2. Scans forward through token bytes (calls to 0x09BAC9 (= NxtFetch)/0x09BBAD)
3. Classifies the token (comparisons against 0x3E `>`, 0xBB, 0x2A `*`, etc.)
4. Writes the new position back to `0xD0231A`

The workspace at `0xD02670` (3 bytes via IX) holds per-entry state:
- Byte 0: flags (bits 0,1,2,3,5,6,7 all tested at various points)
- Byte 1: C register (secondary flags or counter)
- Byte 2: always cleared to 0
