## Root Cause and Fix Recommendation

### The Bug

The memcpy utility at 0x00285F is an eZ80 ADL-mode routine. On real hardware:
- `LD B, 0x01` in ADL mode loads the full 8-bit B register (bits 15:8 of the 24-bit BC).
- The upper byte (bits 23:16, called BCU) is a separate physical register on the eZ80.
- On real hardware, `LD B, imm8` does NOT touch BCU -- it only modifies the middle byte.
- The transpiler correctly implements this: `cpu.b = 0x01` only sets bits [15:8].

The problem is that the **caller at 0x048B5B sets BC = 0xD13FD8** (the destination address)
before calling the memcpy. The memcpy expects to use BC as a counter, and its `LD B, 0x01`
only resets bits [15:8]. The upper byte 0xD1 from the caller remains, making BC = 0xD101xx
instead of the intended 0x0001xx.

### Where to Fix

**Option A: Fix the memcpy utility (0x00285F)**
- Before using BC as a counter, clear the upper byte: force `cpu.bc &= 0x00FFFF` after the IY setup.
- Pro: Fixes this and any other caller that leaves garbage in BCU.
- Con: May not match real hardware behavior if the real OS relies on BCU being preserved.

**Option B: Fix the caller (0x048B5B)**
- Have the caller clear BCU before or after pushing parameters.
- Con: The transpiled code matches the ROM bytes exactly -- the caller IS setting BC=0xD13FD8.

**Option C: Fix the memcpy parameter parsing**
- The memcpy reads length from stack via IY indexing. If the stack frame setup is correct,
  the length (0x000448) should be read from the stack, not from the BC register.
- The SBC HL,BC at 0x00286F is comparing HL against BC -- if BC should be 0x0001xx
  (from LD B,0x01 with C leftover), but the length 0x000448 was pushed to stack,
  then the real question is: how does HL get set to the source address?

**Recommendation**: The most likely fix is **Option A** -- the memcpy utility should mask BC
to 16 bits when setting up the LDIR counter, since this is a Z80-era memcpy pattern that
was not updated for the eZ80's 24-bit registers. The `LD B, 0x01` instruction was intended
to set BC to 0x01xx (256-byte page), but in ADL mode the upper byte leaks through.

Concretely, in the transpiled `block_00285f_adl`, after the `cpu.b = 0x01` line, insert:
```javascript
cpu.bc &= 0x00ffff;  // Clear BCU — Z80-era memcpy assumes 16-bit BC
```

Or equivalently, patch the LDIR count in `block_00287d_adl` before `cpu.ldir()`:
```javascript
cpu.bc &= 0x00ffff;  // Ensure LDIR uses 16-bit count
```

### Secondary concern: HL = 0xCFFFF8 as source length

Even with BCU fixed, HL=0xCFFFF8 enters the memcpy as the "source length". This value comes from
0x05206E which computes: SBC HL,HL (=0xFFFFFF with carry set) + LD L,B (=0x78) + ADD HL,DE
(0xD00080) = 0xCFFFF8. On real hardware with 16-bit Z80-mode HL, this might wrap differently.
The HL value should be investigated further if the BCU fix alone doesn't resolve the boot crash.
