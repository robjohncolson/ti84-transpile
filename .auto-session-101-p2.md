# Auto-Session 101 — P2: Check 0xFFF909 + seed numLastEntries=1 for history-recall path

## Context

This is a TI-84 Plus CE ROM transpilation project. The ROM is at `TI-84_Plus_CE/ROM.rom` (4 MB, eZ80), the transpiler is `scripts/transpile-ti84-rom.mjs`, and the runtime is `TI-84_Plus_CE/cpu-runtime.js`.

**The problem**: When numLastEntries (at RAM 0xD01D0B) is forced to 1, the history manager at 0x0921CB runs 99 steps then hits a **missing transpiled block at 0xFFF909**. This is the path that would recall the last entry and jump to 0x058693 (the common tail where ParseInp is called at 0x0586E3). We need to:
1. Check if 0xFFF909 is valid ROM code or a corrupted/garbage address
2. If valid, add it as a transpiler seed and retranspile
3. Re-run the probe with numLastEntries=1 to see if the history-recall path reaches ParseInp

## What we know

- 0xFFF909 is in the upper ROM region (ROM is 0x000000-0x3FFFFF, so 0xFFF909 is way outside ROM!)
- This strongly suggests 0xFFF909 is a corrupted address from an uninitialized history buffer at 0xD0150B
- The history manager at 0x0921CB checks numLastEntries, and when nonzero, tries to recall the last entry from the circular history buffer at 0xD0150B
- On fresh boot, the history buffer is uninitialized (all zeros or garbage), so recalling from it produces a garbage address

## Your task

### Part A: Verify 0xFFF909 is invalid

Create a probe at `TI-84_Plus_CE/probe-phase25ap-fff909-check.mjs` that:
1. Reads ROM bytes at 0xFFF909 (if within ROM range) OR confirms it's outside ROM (>0x3FFFFF)
2. If outside ROM, document that 0xFFF909 is NOT a valid code address
3. Check what's in the history buffer at RAM 0xD0150B after MEM_INIT — is it zeroed? What does 0x0921CB read from it when numLastEntries=1?

### Part B: Trace the history manager with numLastEntries=1

In the same probe (or a second one), trace 0x0921CB step-by-step with numLastEntries=1:
1. Set up MEM_INIT
2. Force 0xD01D0B = 1
3. Run 0x0921CB with a detailed PC trace (every step for 200 steps)
4. Record which RAM addresses the history manager reads from the history buffer
5. Show what address it tries to jump/call to (the 0xFFF909 or wherever the corrupted address comes from)

### Part C: Analysis

In the report, answer:
- Is 0xFFF909 a valid ROM address? (Almost certainly NO since ROM ends at 0x3FFFFF)
- What RAM address does the corrupted return/jump target come from?
- Can we seed the history buffer with valid data to make this path work?
- What would that valid data look like?
