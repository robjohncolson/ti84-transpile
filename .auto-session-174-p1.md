# Auto-Session 174 P1: Fix RST push in transpiler emitter (surgical approach)

## Background
RST instructions on eZ80 are like CALL: they push the return address (fallthrough PC) onto the stack, then jump to a fixed vector (0x00, 0x08, ..., 0x38). Our transpiler currently emits RST as a plain jump (no push), which means execution after RST returns via RET goes to the wrong address.

## The Bug
In `scripts/transpile-ti84-rom.mjs`:

**Line 281** (block walker classification): RST is grouped with jp/jr as `kind: 'jump'`. **DO NOT CHANGE THIS.** Changing it to `kind: 'call'` would cause the block walker to seed fallthrough addresses for every RST instruction, splitting existing blocks and causing an 84K-byte coverage regression (tested in session 173, reverted).

**Line 354** (code emitter): RST is grouped with jp/jr and emits only `return target;` — missing the push of the return address.

Current code at line 354:
```js
if (tag === 'jp' || tag === 'jr' || tag === 'rst') {
    return [`return ${hex(instruction.target, 6)};`];
}
```

## The Fix
Separate RST from jp/jr in the emitter and emit a push before the return, matching how `call` is emitted (lines 372-376):

```js
if (tag === 'jp' || tag === 'jr') {
    return [`return ${hex(instruction.target, 6)};`];
}
if (tag === 'rst') {
    return [
      `cpu.push(${hex(instruction.fallthrough, 6)});`,
      `return ${hex(instruction.target, 6)};`,
    ];
}
```

The decoder at `TI-84_Plus_CE/ez80-decoder.js` line 417 already sets `fallthrough: startPc + length` on RST instructions, so `instruction.fallthrough` is available.
