# Auto-Session 184 P3: Bypass Error Handler — Direct errSP Setup for ParseInp

## Context

This is a TI-84 Plus CE ROM transpilation project. ParseInp (0x099914) fires E_Undefined (errNo=0x8D) when processing "2+3" tokens. The error dispatch chain goes:
1. JError at 0x061DB2 writes errNo
2. Calls 0x03E1B4 (error handler wrapper)
3. Which calls 0x03E187 (RSMIX to z80-mode for port cleanup)
4. z80-mode POP AF + RET pops from {MBASE=0xD0, SPS} page
5. Gets 0x000000 (zeros) → boot vector → ISR → HALT

The PushErrorHandler mechanism (0x061D90) sets up an error catch frame with:
- errSP stored at a fixed RAM location (likely 0xD007EA based on JError's `LD SP,(0xD007EA)` at 0x061DCA)
- hlPayload at another location

If we set errSP BEFORE calling ParseInp, JError's `LD SP,(errSP)` would restore SP to our catch point. But currently the z80-mode RET in step 4 prevents execution from reaching 0x061DCA.

## Task

Build a Node.js probe at `TI-84_Plus_CE/probe-phase184-errsp-bypass.mjs` that:

1. **Find errSP location**: Disassemble around 0x061DCA to find the exact address JError reads SP from. Look for `LD SP,(nnnn)` instruction. Also find where PushErrorHandler (0x061D90) stores the error frame.

2. **Find the full PushErrorHandler frame structure**: Disassemble 0x061D90 to understand what it stores and where. It likely pushes a return address + error handler address onto a "handler stack" at some fixed RAM location.

3. **Bypass approach**: Set up the error frame manually before calling ParseInp:
   - Write errSP = current SP (so JError restores to our known SP)
   - Write the error handler address (hlPayload) so after `LD SP,(errSP)`, execution goes to our catch point
   - Then call ParseInp
   - If ParseInp errors, JError should now reach 0x061DCA and restore SP

4. **But wait**: The problem is that JError never reaches 0x061DCA because the z80-mode RET at 0x03E1B3 diverts to boot vector. So the REAL bypass is different: **skip the 0x03E1B4 call entirely**. Can we patch the error dispatch to NOT call 0x03E1B4? Or can we set up the 0xD0xxxx stack page to contain a return address that goes back to JError at the point after the CALL 0x03E1B4?

5. **Concrete experiment**: 
   - Find the exact PC where JError calls 0x03E1B4 (should be around 0x061DB2-0x061DCA)
   - Calculate what return address the CALL pushes
   - Pre-write that return address at the 0xD0xxxx SPS stack location where the z80-mode RET will pop from
   - This way: z80-mode RET → pops our planted return address → continues at JError after the CALL → reaches `LD SP,(errSP)` → restores to error catch frame

6. **Execute the experiment**: Boot ROM, set up tokens "2+3" in buffer, plant the return address, call ParseInp, trace what happens. Does it reach the error catch point?
