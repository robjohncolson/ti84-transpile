# Auto-Session 111 P1: Fix RclVarSym-after-ParseInp via OPS Reset

## Context

This is a TI-84 Plus CE ROM transpilation project. We run lifted ROM code (eZ80 machine code transpiled to JavaScript) via `cpu-runtime.js` + `ROM.transpiled.js`.

**Problem**: After ParseInp evaluates "2+3" and returns OP1=5.0, calling RclVarSym to recall the Ans variable fails — OP1 contains garbage `[00 70 00 8a 9a 09 fe ff 7f]`.

**Root cause** (proven in session 110): ParseInp's operator stack (OPS) grows UPWARD from 0xD3FFF6 to 0xD40002 during evaluation, directly overwriting the VAT entry for Ans at 0xD3FFF8. The garbage in OP1 is literally the corrupted VAT bytes. The Ans DATA at 0xD1A881 is intact (contains 5.0 BCD), but the VAT METADATA at 0xD3FFF8 is destroyed.

## Task

Build a probe `TI-84_Plus_CE/probe-phase25az-rclvar-ops-fix.mjs` that tests TWO fix approaches:

### Approach A: Post-ParseInp cleanup + re-create variable

After ParseInp returns:
1. Save the OP1 result (5.0)
2. Reset OPS back to OPBase (write 0xD3FFF6 as a 3-byte LE value at address 0xD02593)
3. Reset FPS back to FPSbase (write 0xD1A88A as a 3-byte LE value at address 0xD0258D)
4. Re-run CreateReal("Ans") to rebuild the VAT entry that OPS destroyed
5. Manually write the saved 5.0 BCD into the Ans data slot
6. Call RclVarSym with OP1 set to the Ans variable name
7. Check if OP1 now contains 5.0

### Approach B: Higher progPtr to give OPS headroom

Before the whole pipeline:
1. Set progPtr (0xD0259D) to 0xD40020 instead of 0xD3FFFF — this gives 32 extra bytes above OPBase for OPS to grow into before hitting any VAT entries
2. Set OPBase (0xD02590) and OPS (0xD02593) to 0xD40006 (matching the new boundary)
3. Set pTemp (0xD0259A) to 0xD40020 (same as progPtr)
4. Run: MEM_INIT → CreateReal("Ans") → ParseInp("2+3") → check VAT integrity → RclVarSym
5. Check if the VAT entry at the new location survives ParseInp and RclVarSym returns 5.0

### Reference code pattern

Copy the pipeline pattern from `TI-84_Plus_CE/probe-phase25ay-rclvar-vat-dump.mjs` (Scenario B). The key setup steps are:
- `createExecutorFromFile()` to load ROM.transpiled.js
- Cold boot via `callBlock(0x000000)` + kernel init via `callBlock(0x0586DA)` with `cpu.ix = 0xD1A860`
- Post-init via `callBlock(0x09DEE0)` for MEM_INIT
- After MEM_INIT, adjust FPS/FPSbase to 0xD1A88A (past Ans data slot)
- CreateReal("Ans"): write OP1 = [0x15, 0x72, 0x00, ...] (type 0x15 = Real, name bytes 0x72 = "Ans"), then `callBlock(0x08227C)`
- ParseInp("2+3"): write tokens [0x32, 0x70, 0x33, 0x3F] to userMem 0xD1A881, seed begPC/curPC/endPC, clear OP1, set error frame, then `callBlock(0x099914)`
- RclVarSym: set OP1 to Ans name pattern, then `callBlock(0x09AC77)`

### Key addresses (all 3-byte little-endian in RAM)
- OPBase: 0xD02590
- OPS: 0xD02593  
- pTempCnt: 0xD02596 (4 bytes)
- pTemp: 0xD0259A
- progPtr: 0xD0259D
- FPSbase: 0xD02587
- FPS: 0xD0258D
- begPC: 0xD02317
- curPC: 0xD0231A
- endPC: 0xD0231D
- OP1: 0xD005F8 (9 bytes)
- userMem: 0xD1A881
- errSP: 0xD02589 (3 bytes, but stored/read as 2-byte LE at this address in some contexts)

### Error frame setup (required for ParseInp)
```js
const ERR_CATCH_ADDR = 0x7EEDF0;
const FAKE_RET = 0x7FFFFE;
const errSP_val = cpu.sp - 6;
// Push error frame: AF, HL (catch address), DE
mem[cpu.sp - 1] = 0x00; mem[cpu.sp - 2] = 0x40; // AF = 0x0040
mem[cpu.sp - 3] = (ERR_CATCH_ADDR >> 16) & 0xFF;
mem[cpu.sp - 4] = (ERR_CATCH_ADDR >> 8) & 0xFF;
mem[cpu.sp - 5] = ERR_CATCH_ADDR & 0xFF; // HL = ERR_CATCH_ADDR
mem[cpu.sp - 6] = (FAKE_RET >> 16) & 0xFF;
mem[cpu.sp - 7] = (FAKE_RET >> 8) & 0xFF;
mem[cpu.sp - 8] = FAKE_RET & 0xFF; // DE = FAKE_RET
// Write errSP
mem[0xD02589] = errSP_val & 0xFF;
mem[0xD0258A] = (errSP_val >> 8) & 0xFF;
mem[0xD0258B] = (errSP_val >> 16) & 0xFF;
```

### Success criteria
- Approach A: RclVarSym returns OP1 = [00 80 50 00 00 00 00 00 00] = 5.0
- Approach B: VAT entry survives ParseInp intact AND RclVarSym returns 5.0
- Print clear PASS/FAIL for each approach with the actual OP1 bytes
