# Task: Phase 25P — VAT CreateReal IX-offset disassembly

## Context
You are working on a TI-84 Plus CE ROM transpilation project. CreateReal (0x08238A) and ChkFindSym (0x08383D) both crash — CreateReal jumps to 0x58c35b after ~50 steps, ChkFindSym jumps to 0x7eedf3 after 9 steps. The 0x58c35b address is a known stack-corruption/missing-block pattern.

Session 73 captured the CreateReal PC trace: 0x082bba → 0x061d3e → 0x061db2 → 0x03e1b4 ... 0x082bc2 → 0x03e1ca → 0x061dba → 0x58c35b.

We need to understand WHY it crashes — likely IX-offset reads hitting garbage, or uninitialized VAT head pointers.

## Goal
Write a **static disassembly probe** `TI-84_Plus_CE/probe-phase25p-vat-disasm.mjs` that:
1. Reads ROM.rom directly (no execution needed)
2. Disassembles the CreateReal entry path from 0x08238A through the first ~128 bytes
3. Also disassembles the path at 0x061d3e (where the trace goes) for ~64 bytes
4. Also disassembles 0x03e1b4 and 0x03e1ca for ~32 bytes each
5. For each instruction, flags any IX-offset or IY-offset reads/writes, and any absolute RAM loads from 0xD00000-0xD1FFFF range
6. Produces a report identifying which RAM addresses are read before being written (i.e., expected to be pre-seeded by the caller)

## Template
Use `TI-84_Plus_CE/probe-phase25o-parseinp-helpers.mjs` as your template — it's a pure static disassembly probe that uses `ez80-decoder.js`. Copy its approach for `decodeInstruction`, `formatInstruction`, and the RAM-reference extraction logic.

Key imports:
```js
import { decodeInstruction } from './ez80-decoder.js';
```

The decoder expects: `decodeInstruction(romBytes, pc, 'adl')` — returns an object with `.tag`, `.length`, `.pc`, and various fields depending on the tag. Mode is the string `'adl'`, not a boolean.

## Known addresses
- IX is set to 0xD1A860 in our postInitState
- IY is set to 0xD00080
- progPtr = 0xD01C00, pTemp = 0xD01C03, OPBase = unknown
- OP1 = 0xD005F8

## Deliverables
1. `TI-84_Plus_CE/probe-phase25p-vat-disasm.mjs` — static disassembly probe
2. `TI-84_Plus_CE/phase25p-vat-disasm-report.md` — written by the probe

The report should have:
- Full disassembly listing for each target range
- A table of all RAM accesses found (address, read/write, which instruction, via IX/IY/absolute)
- A conclusion section identifying which addresses need to be seeded for CreateReal to work

## Constraints
- Do NOT modify any existing files
- Do NOT touch cpu-runtime.js, peripherals.js, or the transpiler
- This is a READ-ONLY probe — it reads ROM.rom and decodes, no execution
- Runnable with `node TI-84_Plus_CE/probe-phase25p-vat-disasm.mjs`

## Success criterion
A clear report showing which RAM addresses CreateReal reads early in its execution path. This tells us what to seed for a future dynamic probe.
