# Session 128 P3: Investigate and lift ROM's 0xBB-prefix two-byte token dispatch handler

## Context
The TI-84 CE ROM transpilation has a browser shell that tokenizes math expressions and feeds them to ParseInp. Session 127 extended the tokenizer with two-byte tokens (0xBB prefix): min(, max(, mean(, median(, randInt(, remainder(, lcm(, gcd(. The tokenizer correctly emits [0xBB, secondByte] sequences. However, when ParseInp processes these tokens, it stalls at PC=0x001221 — the ROM's 0xBB-prefix dispatch handler isn't fully lifted (missing transpiled blocks).

**Goal**: Investigate the 0xBB dispatch path from ParseInp, find which blocks are missing, add seeds, re-transpile, and verify that at least one two-byte function (e.g., min(3,7)) works through ParseInp.

## What we know
- ParseInp at 0x099914 processes tokens from the token buffer
- Single-byte tokens (0x70=+, 0x71=-, etc.) work correctly
- Two-byte tokens use 0xBB as the first byte, then a second byte identifies the function
- When ParseInp encounters 0xBB, it dispatches to a handler — likely a lookup table
- This handler is at or near PC=0x001221 where the probe stalled
- The stall means either a missing block at 0x001221 or a block there that loops infinitely

## Your tasks

### Task 1: Investigate the 0xBB dispatch path
Build a probe `TI-84_Plus_CE/probe-phase128-bb-dispatch.mjs` that:
1. Sets up a standard runtime: cold boot → MEM_INIT → CreateReal("Ans")
2. Places a simple two-byte token expression in the token buffer: `[0xBB, 0x0C, 0x2B, 0x33, 0x2B, 0x37, 0x11, 0x3F]` — this is `min(3,7)` with comma=0x2B, closing paren=0x11, end=0x3F
   (Check the token values: 3=0x33, 7=0x37, comma=0x2B, right-paren=0x11, end=0x3F — verify against browser-shell.html's EVAL_TOKEN_MAP)
3. Seeds begPC/curPC/endPC and runs ParseInp
4. Traces execution with a high step budget (50K), collecting every unique PC visited
5. Identifies where execution stalls (which address gets the most hits)
6. Dumps the ROM bytes at the stall address and surrounding 32 bytes

Reference `TI-84_Plus_CE/probe-phase127-post-parseinp.mjs` for the standard setup pattern (MEM_INIT, CreateReal, pointer seeding, error frame, etc.).

### Task 2: Disassemble the stall point
In the same probe:
1. Read ROM bytes at 0x001220-0x001260 and disassemble them using ez80 rules (the ROM runs in ADL mode)
2. Check if 0x001221 is inside a valid instruction or a block boundary
3. Check PRELIFTED_BLOCKS to see if any block covers 0x001221
4. If the block exists, dump the transpiled JS for that block

### Task 3: Add seeds for uncovered blocks
If the stall is due to missing blocks:
1. Add the needed seed addresses to `TI-84_Plus_CE/seeds-phase128.txt`
2. Re-run transpiler: `node scripts/transpile-ti84-rom.mjs`
3. Re-run the probe to check if the stall is resolved
4. If new stalls appear at different addresses, add those seeds too (iterate up to 3 times)

### Task 4: Verify
1. Run `node TI-84_Plus_CE/probe-phase99d-home-verify.mjs` — must pass 26/26
2. If the 0xBB dispatch now works, report the result of min(3,7) — should be 3.0

- 0x30-0x39: digits 0-9
- 0x2B: comma
- 0x11: right parenthesis )
- 0x10: left parenthesis (
- 0x3F: end-of-expression
- 0xBB: two-byte prefix
- 0xBB 0x0C: min(
- 0xBB 0x0D: max(
- 0xBB 0x07: gcd(
