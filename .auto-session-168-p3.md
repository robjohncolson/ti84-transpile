# Auto-Session 168 P3: IY Flags Check + Normalization Path Analysis

## Context

This is a TI-84 Plus CE ROM transpilation project. We're investigating the gcd bug where gcd(12,8) produces E_Domain instead of 4.

The normalization function at 0x07CA48 zeros OP1's mantissa for all integer inputs (exp >= 0x80). Session 158 mentioned that the secondary normalization at 0x07C9BF "checks bit 3 of IY+0x48" which controls normalization behavior. If this bit is set/cleared differently on real hardware vs our transpilation, the normalization might take a completely different path.

Also, the normalization loop exit goes through 0x07C9AF → BigShift at 0x07FAC2 → zeros OP1. But there might be an alternative exit path (through 0x07CA87+ exponent combination logic) that preserves the mantissa — session 159 showed this path exists but "was NEVER REACHED" in our trace.

## Task

Build a probe `TI-84_Plus_CE/probe-phase168-iy-flags-norm.mjs` that:

### Part A: IY and System Flags Dump
1. Set up gcd(12,8) using the same setup as `probe-phase167-stack-balance.mjs`
2. Before running gcd, dump:
   - IY register value
   - The byte at (IY+0x48) — which flags are set?
   - The byte at (IY+0x0E) — session 161 mentioned RES bit at (IY+14=0x0E) in FPAdd
   - The byte at (IY+0x59) — session 161 mentioned SET bit at (IY+89=0x59) in FPAdd
   - All bytes (IY+0x00) through (IY+0x60) — full system flags dump
3. Compare against known TI-84 CE flag defaults. IY is typically 0xD00080 on TI-84 CE. The flags area 0xD00080-0xD000E0 controls calculator modes, error states, graph settings, etc.

### Part B: Normalization Path Trace
1. Run gcd until the first normalize entry at 0x07CA48 (~step 62)
2. At normalize entry, log:
   - The value of A after `SUB 0x80` (this determines shift count)
   - The branch decision at each conditional (JP C, CP, JR C, etc.)
   - Whether the loop goes through 0x07C9AF (BigShift/zero path) or 0x07CA87 (exponent combination path)
   - The value of bit 3 at (IY+0x48) at the secondary normalization check point 0x07C9BF
3. After the normalize returns, log the full OP1 state

### Part C: Force Alternative Path
1. Set up gcd(12,8) again
2. Before the first normalize call, FLIP bit 3 of (IY+0x48) to the opposite value
3. Re-run and trace: does the normalization take a DIFFERENT path?
4. Log the OP1 result — is the mantissa preserved?

### Part D: Test with IY flags from real TI-84 init
1. The OS init sequence at 0x00007A sets up IY and system flags. Run a boot-to-homescreen sequence (same as probe-phase99d-home-verify.mjs) and capture the (IY+0x00)-(IY+0x60) bytes AFTER boot completes.
2. Then run gcd with these real IY flags and check if normalization behaves differently.
