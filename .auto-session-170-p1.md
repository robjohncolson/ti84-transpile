# Task: Full F Register Deep Trace at Every Block Boundary During gcd(12,8)

## Context
We're investigating a systemic gcd bug in the TI-84 Plus CE ROM transpiler. Every gcd(a,b) call produces E_Domain instead of the correct result. All individual components (gcd_helper, RndGuard, ConstLoader, normalization, FPDiv, RLD, IY flags, stack balance, POP AF, register A, ROM bytes) have been verified correct in isolation. The bug must be in how components interact — possibly a wrong flag causing a conditional jump to take the wrong path.

## What to build
Create `TI-84_Plus_CE/probe-phase170-flag-deep-trace.mjs` — a probe that traces the FULL F register (all 8 bits decoded) at every block boundary during gcd(12,8).

## Reference pattern
Copy the boilerplate setup from `TI-84_Plus_CE/probe-phase169-gcd-helper-disasm.mjs`:
- Same imports, same constants, same `createRuntime()`, same `coldBoot()`, same `prepareCallState()`
- Same BCD loading helpers

## Probe logic

1. Cold boot the system (same as phase 169 probes)
2. Set up gcd(12,8): load OP1=12 (type=0x00, exp=0x81, mantissa 12 00 00 00 00 00 00), OP2=8 (type=0x00, exp=0x80, mantissa 80 00 00 00 00 00 00), set fpCategory=0x28
3. Use `executor.onBlockEntry` callback (or intercept mechanism) to log state at EVERY block boundary. The executor object has a way to hook block entries — check `cpu-runtime.js` for the exact API. If there's no built-in hook, use a step-by-step approach: run with maxSteps=1 in a loop, logging after each step.

At each block boundary, log:
- Step number
- PC (hex)
- Full F register value (hex)
- Decoded flags: S (bit 7), Z (bit 6), H (bit 4), P/V (bit 2), N (bit 1), C (bit 0)
- Also log bit 5 (Y/undocumented) and bit 3 (X/undocumented) since eZ80 may use them differently

4. After execution completes, print:
- Total steps
- Error code
- Final OP1, OP2 values
- A summary of all CONDITIONAL JUMPS/CALLs encountered — for each, print the PC, the condition (Z/NZ/C/NC/etc), the F register value at that point, and whether the jump was taken (i.e., what PC follows)

5. Analysis section: look for any block where the F register seems wrong for the instruction that just executed. Specifically look for:
- Z flag set when it shouldn't be (or vice versa) near conditional jumps
- Carry flag anomalies near CP instructions
- Any flag value that doesn't match the expected result of the prior instruction

## Key addresses
- GCD entry: 0x068D3D
- GCD inner body: 0x068D82-0x068DA1
- GCD post-body: 0x068DA3-0x068DE3
- GCD loop top: 0x068D5D-0x068D81
- Compound function (INT): 0x07C747
- InvSub: 0x07C74F
- Key conditional jumps in gcd: JR Z at 0x068D65, JR C at 0x068D67, JR NC at 0x068D73, JR NZ at 0x068DA1

A single file `TI-84_Plus_CE/probe-phase170-flag-deep-trace.mjs` that runs with `node TI-84_Plus_CE/probe-phase170-flag-deep-trace.mjs` and prints the complete flag trace.
