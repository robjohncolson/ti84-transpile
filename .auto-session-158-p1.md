# Session 158 P1 — Transpiled Block Fidelity Check for Shl14 (0x07FB33/0x07FB50)

## Context
We're debugging why gcd(12,8) fails with E_Domain on our TI-84 CE ROM transpiler. The root cause has been traced to the mantissa left-shift routine at 0x07FB33 (Shl14), called from the normalization loop at 0x07CA48. The shift ALWAYS zeros the mantissa when called during gcd.

The ROM instruction at the heart of this is **RLD** (opcode ED 6F) — a BCD rotate-left-through-accumulator instruction.

## Your Task
Build a probe `TI-84_Plus_CE/probe-phase158-shl14-fidelity.mjs` that:

### Part A: Standalone RLD verification
1. Create a fresh runtime (mem, executor, peripherals with timerInterrupt:false)
2. Set OP1 at 0xD005F8 to [00, 83, 12, 00, 00, 00, 00, 00, 00] (= 1200.0 in TI BCD)
3. Manually call `cpu.rld()` with cpu.hl=0xD00600, cpu.a=0 — verify the result matches expected: mem[0xD00600] should be 0x00 (unchanged since last byte was 00), A should be 0x00
4. Then set mem to [00, 83, 12, 34, 56, 78, 90, 12, 34] and repeat the full 7-RLD chain (HL starting at 0xD00600, DEC HL between each). Print before/after mantissa and final A. Expected after chain: mantissa bytes should be [23, 45, 67, 89, 01, 23, 40] with A=0x01 (leading digit shifted out).

### Part B: Transpiled block execution of Shl14
1. Load ROM.transpiled.js blocks
2. Check that blocks exist for keys `07fb33:adl` and `07fb50:adl`
3. Set OP1 = [00, 83, 12, 34, 56, 78, 90, 12, 34] (rich mantissa to detect any nibble errors)
4. Execute block `07fb33:adl` → should set HL=0xD00600, XOR A, JR to block `07fb50:adl`
5. Execute block `07fb50:adl` → should perform 7 RLD + 6 DEC HL
6. Print the resulting mantissa bytes and A register
7. Compare against Part A's manual result — they MUST match
8. Also test with OP1 = [00, 83, 12, 00, 00, 00, 00, 00, 00] (the actual gcd input, mantissa = 1200)

### Part C: Transpiled block fidelity — ROM byte comparison
1. Read the ROM bytes at 0x07FB33 for 29 bytes (until JR target at 0x07FB50)
2. Read ROM bytes at 0x07FB50 for 21 bytes (until RET at 0x07FB64)
3. Use ez80-decoder.js to decode each instruction from the ROM bytes
4. Print: address | ROM hex | decoded mnemonic
5. Also decode the block at 0x07CA48 for 64 bytes (the normalization loop): address | ROM hex | decoded mnemonic. Pay attention to CALL/JR/JP targets and loop structure.

## Key Addresses
- OP1: 0xD005F8 (9 bytes: type, exponent, 7 mantissa bytes)
- OP1 mantissa start: 0xD005FA
- OP1 mantissa end: 0xD00600
- OP1 exponent: 0xD005F9
- Shl14: 0x07FB33 (entry with HL=0xD00600, XOR A, JR to 0x07FB50)
- Shl14 chain: 0x07FB50 (7× RLD, 6× DEC HL, RET)
- Normalization loop: 0x07CA48

## Implementation Pattern
Use the same probe boilerplate as `probe-phase157-gcd-07ca48-trace.mjs`:
- Import createExecutor from cpu-runtime.js, createPeripheralBus from peripherals.js
- Load ROM.rom and ROM.transpiled.js
- Create executor with `createPeripheralBus({ timerInterrupt: false })`
- For block execution: `ex.executeBlock(BLOCKS, pc)` returns next PC
- Access CPU state via `ex.cpu` (ex.cpu.a, ex.cpu.hl, ex.cpu.f, etc.)
- Memory access: `mem[addr]` for reads, `mem[addr] = val` for writes

## Success Criteria
- Part A manually computed result matches Part B transpiled block result → RLD implementation is CORRECT
- Part A doesn't match Part B → transpiler/decoder bug identified (report WHICH instruction diverges)
- Part C decoded ROM matches expected instruction sequence
