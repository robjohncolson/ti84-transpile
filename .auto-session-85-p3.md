# Task: ParseInp with pre-created variable via CreateReal — full pipeline test

## Context
The TI-84 Plus CE ROM transpiler lifts eZ80 machine code to JavaScript. ParseInp (0x099914) computes 2+3=5.0 but returns errNo=0x8D (ErrUndefined). Hypothesis: after computing 5.0, the parser tries to store the result to a variable but no variable exists → ErrUndefined.

Session 78 showed CreateReal works (creates a real variable in the VAT). Session 83 showed MEM_INIT works (17 steps). The committed probe `TI-84_Plus_CE/probe-phase25x-meminit-then-parseinp.mjs` is the reference for MEM_INIT + ParseInp setup.

## What to do

Build a probe `TI-84_Plus_CE/probe-phase25z-full-pipeline.mjs` that runs:

### Step 1: Cold boot + MEM_INIT (same as 25X probe)
- Boot (0x000000), kernel init (0x08C331), post-init (0x0802B2) — same as all recent probes
- Call MEM_INIT at 0x09DEE0 (completes in ~17 steps)
- Verify post-MEM_INIT: OPS=0xD3FFFF, FPS=0xD1A881, OPBase=0xD3FFFF

### Step 2: CreateReal("A")
After MEM_INIT, create a real variable named "A":
- Set OP1 to [0x00, 0x41, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00] (type=RealObj=0x00, name="A")
- Push FAKE_RET (0x7FFFFE) onto stack as return address
- Set up errSP frame: push ERR_CATCH_ADDR (0x7FFFFA), then write SP to errSP (0xD008E0)
- Clear errNo (0xD008DF) to 0
- Jump to CreateReal at 0x08238A (sets A=0, HL=9, then JP 0x0822D9)
  Actually, the entry is: LD A,0; LD HL,9; JR to JP 0x0822D9
  So set A=0x00 (RealObj), HL=9, and jump to 0x0822D9 directly
  OR just call the JT entry: 0x020368 (JP 0x08238A)
  Actually simplest: set registers and call 0x08238A
- Run up to 50,000 steps
- Check errNo — should be 0x00 for success
- Record DE (should point to the variable's data area in RAM)
- Record OPBase movement (should decrease by 9 bytes)

### Step 3: ParseInp("2+3")
After CreateReal succeeds:
- Write tokens [0x32, 0x70, 0x33, 0x3F] to TOKEN_BUFFER_ADDR (0xD00800)
- Set begPC (0xD02317) = 0xD00800
- Set curPC (0xD0231A) = 0xD00800
- Set endPC (0xD0231D) = 0xD00803
- Set OP1 to [0x00, 0x41, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00] (variable "A" for result storage)
- Push FAKE_RET onto stack
- Set up errSP frame again
- Clear errNo
- Jump to ParseInp at 0x099914
- Run up to 1,500,000 steps

### Step 4: Verify
- Read OP1 — should be 5.0: [0x00, 0x80, 0x50, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]
- Read errNo — hoping for 0x00 (success) instead of 0x8D
- Read the variable data at the DE address from step 2 — is 5.0 stored there?
- Print detailed results
