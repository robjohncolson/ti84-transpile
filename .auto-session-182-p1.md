# Auto-Session 182 P1 — ParseInp E_Undefined Root Cause: OS State Diff

## Context
Session 181 found that BufInsert (0x05E2A0) works correctly in a standalone Node probe — tokens [0x32, 0x70, 0x33] ("2+3") are placed at 0xD00A00. But ParseInp (0x099914) hits E_Undefined (errNo=0x8D) and jumps to 0x1BC300 (erased flash). The browser-shell's direct-eval path works in the browser but the same code path fails in Node.

Root cause hypothesis: the standalone probe is missing OS initialization state that the browser-shell accumulates during boot. Specifically, the app error vector table at cxErrorEP (0xD007D6) may be zero, causing the error handler to jump to erased flash.

## Task
Build a diagnostic probe `TI-84_Plus_CE/probe-phase182-parseinp-state-diff.mjs` that:

1. **Boots the ROM** using the standard boot sequence (same as probe-phase99d-home-verify.mjs — read that file for the pattern)
2. **Dumps key RAM regions** after boot but before calling ParseInp:
   - Error vector table: 0xD007D0-0xD007E0 (includes cxErrorEP at 0xD007D6)
   - Edit buffer state: 0xD00500-0xD00600 (editCursor, editTail, etc.)
   - Parser state: 0xD02300-0xD02600
   - Operator/Function stacks: 0xD007C0-0xD00800
   - curRow/curCol: 0xD00595-0xD00596
3. **Checks if cxErrorEP (0xD007D6, 3 bytes little-endian)** is zero or points to erased flash (>= 0x100000 and the target is 0xFF fill)
4. **If cxErrorEP is zero**: try setting it to a known error-catch address. The OS error handler jump table starts at 0x0598xx area. A safe choice is to set cxErrorEP = 0x059A00 (or wherever ERR_CATCH lives — grep the codebase for "cxErrorEP" or "errorEP" to find what browser-shell uses).
5. **Re-runs ParseInp** after fixing cxErrorEP and reports whether it still hits E_Undefined or progresses further.
6. **Prints a structured report** of all dumped values and the before/after ParseInp behavior.
