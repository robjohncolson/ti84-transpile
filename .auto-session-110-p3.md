# Auto-Session 110 P3: Static Disassembly of 0x099211 — ParseInpGraph Wrapper Hypothesis

## Context
Research refinement R4 hypothesizes that 0x099211 is a "ParseInpGraph-style wrapper" — analogous to the TI-83+ `_ParseInpGraph` which sets graph/split-screen override flags before calling ParseInp. Evidence:
- 0x099211 is 0x703 bytes before ParseInp core (0x099914)
- Called from ENTER handler at 0x058626
- Session 98 named it "DisarmScroll" but that was a guess

If 0x099211 sets ParsFlag/ParsFlag2/fmtFlags/splitOverride bits before calling ParseInp, those flags are the exact ambient state our direct-ParseInp probe bypasses — meaning they document what ParseInp expects.

## Task
Create probe `TI-84_Plus_CE/probe-phase25ay-disasm-099211.mjs` that:

1. Loads ROM.rom directly (no transpiled blocks needed)
2. Statically disassembles 300 bytes starting at 0x099211 using the eZ80 decoder
3. For each instruction, also check:
   - Is it a CALL to a known address? (PushErrorHandler=0x061DEF, ParseInp=0x099914, ChkFindSym=0x08383D, any IY-relative flag manipulation)
   - Is it a RES/SET/BIT instruction on (IY+offset)? If so, decode the IY offset to identify which OS flag byte it touches (IY=0xD00080, so IY+0x44=0xD000C4=ParsFlag area)
   - Is it a JP or JR to ParseInp (0x099914) or near it?
   - Is it accessing any of these flag addresses via absolute addressing: ParsFlag(0xD000C4), ParsFlag2(0xD000C5), fmtFlags(0xD000CA), numMode(0xD000CB), graphFlags2, onFlags, hookflags4
4. Generate a clean disassembly listing with annotations

## Known IY offsets (IY=0xD00080)
- IY+0x00 = 0xD00080 (ioDelFlag)
- IY+0x09 = 0xD00089 (statFlags — bit 6 = statsValid)
- IY+0x12 = 0xD00092 
- IY+0x24 = 0xD000A4
- IY+0x44 = 0xD000C4 (ParsFlag region)
- IY+0x45 = 0xD000C5 (ParsFlag2)
- IY+0x49 = 0xD000C9
- IY+0x4A = 0xD000CA (fmtFlags)
- IY+0x4B = 0xD000CB

## What to look for specifically
1. IY flag manipulation (RES/SET/BIT) — these are the parser state flags
2. CALL 0x061DEF (PushErrorHandler) — error frame setup
3. CALL/JP to ParseInp (0x099914) or near addresses
4. OP1 reads/writes (LD from/to 0xD005F8 area)
5. The endpoint: where does this function end? (RET, JP to ParseInp, or fall-through)
6. Any reference to 0x099910 (the trampoline we already know about)
